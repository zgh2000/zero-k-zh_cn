# Cold Take #12 - 强力变形（Mighty Morphing）
日期：2024-05-19

从其他策略游戏转来的玩家往往会发现经典惯例被打破，常见系统缺失。改善单位的三个核心系统——变形（morphing）、老兵等级（veterancy）和研究（research）——在 ZK 中基本缺失。今天是 Sprung 系列客座文章的第一篇，将讨论这些问题。

首先，我们来谈谈在 ZK 中仍然以残余形式存在的单位改善系统：变形。变形通常被称为升级，但我不会使用那个词以避免混淆，因为它也可能指研究。广义上，变形让你能改善或修改单个单位，而不是像研究那样升级所有现有和未来的同类型单位。这可以小到花费三秒时间部署一台人族攻城坦克；也可以大到将两个神族圣堂武士合并为执政官。当然，变形可以成为阵营身份和美学的重要组成部分，但我们来讨论游戏性。什么时候应该用变形，而不是让玩家直接建造？

一个简单的答案是，有时候你已经决定不让玩家建造更多这种单位。在 ZK 中你只能获得一个指挥官（Commander），而且地热点数量有限。相应地，这是仅有的两个变形是直接升级的情况。敏锐的读者可能注意到金属点（metal spots）也是有限资源，确实墨西哥钻机（mexes）曾经在过载（overdrive）引入前短暂地变形为深层钻机（Moho mines）。这里变形让你在有限的条件下决定获得更多收益。

另一个游戏中的好例子是《命令与征服：将军》中的中国炎黄坦克（Overlord tank）。它本身因为太贵而有软限制，但你可以在其上添加额外武器以获得更强火力。《魔兽争霸3》英雄的技能点分配也可以视为一种变形，ZK 指挥官的模块分配看起来如此相似也就不足为奇了。

单位限制的反面是灵活性。RTS 游戏的核心之一是制造正确的单位来应对快速变化的战场。变形的灵活性让玩家在面对限制时能继续对单位组合做出选择。《魔兽争霸》英雄、《将军》炎黄坦克和 ZK 指挥官都为此提供了多种变形选择。同样，在《战争黎明》早期，强迫玩家在第一支战术小队中盲目选择火焰喷射器或爆弹枪，不会是好的设计。让部分陆战队变形以装备武器，让玩家稍后再做选择，同时仍然有一支小队在战场上占领点位、侦察和执行其他不涉及特定单位组合的任务。然后，当你侦察到兽人时？兄弟，拿上火焰喷射器。重型火焰喷射器。

但回到 ZK，由于我们实际上没有以单位限制的形式给玩家施加压力，几乎不需要这种灵活性。不断增长的经济意味着即使最昂贵的单位随着时间推移也能变得可负担，到那时你自然拥有所有灵活性。多一点灵活性会有什么坏处？

灵活性的问题在于它创造了一种薛定谔的单位，同时占据设计空间中的多个位置，在观察敌人时才解析为其中之一。ZK 看到的危险是，单一重复的单位可能通过自身的灵活变形变成一种组合。当你能进行即时补给正确类型的单位，而不需要做"后勤"或"预判"这类乏味的蠢事时，一切都会变得简单得多。没有从工厂出发的行进时间。减少过度建造某个变体的风险。当然这些可以通过其他数值调整，但这都会伴随着灵活性的降低。

那么"不灵活"的变形呢？这里没有真正的组合可言，你只是想增加更多属性？比如说，为什么工厂没有+10建造力（Buildpower）的升级？这比看护者（Caretaker）更原子化，后者还能执行维修等其他任务！或者，如果我想升级费伦（Felon）的电池充电速率，我必须要么获得罪犯（Convict）并支付我不需要的建造力，要么获得暴徒（Thug）并附带低效的枪。与此同时，+15护盾再生变形将是原子化的同时仍然是一个选择！部分答案在于单位是世界中的实体实体，因此它们之间有更多的交互。特别是，罪犯的建造力和暴徒的枪会引导玩家使用它们，然后当敌人必须回应时，就会引发游戏通常的复杂性。另一部分是涌现性。费伦最初并不是设计为独立单位并添加"电池"；相反，它被设计为加入护盾球（shieldball）的一种方式，用来消耗电荷（charge）。电池是玩家的发现，即使是一个极其明显的发现。

变形的一个有趣特性是它能提供建造力，并且可以有不同的科技需求，这可以用来调整不同选项的吸引力。如果你要求单位被"正常"建造，那么它就要与其他单位竞争基础设施（建造力、后勤）。变形是避免这种情况的一种方式，这使得该单位更有吸引力，或者使常规基础设施吸引力降低。具体来说：假设你想鼓励隐形移动单位（cloaking），因为隐形很酷。如果你要求人们从隐形工厂（cloaky factory）建造它们，那么有时候他们宁愿建造另一个工厂作为二级工厂，或者如果他们已经有隐形工厂，他们宁愿建造另一种隐形机器人。变形让单位成为自己的临时小工厂，其优雅之处在于，就像能量需求一样，它只触及基础设施而不触及单位本身。隐形移动者和护盾的变形，最初只是为了让它们重新部署，就属于这一类别。我们试图在非辅助单位之外避免这种变形，因为基础设施有其存在的理由，虽然容易管理但不应被淘汰。

在开头我提到攻城模式（Siege Mode）作为变形的例子。在 ZK 中，少数单位——例如螃蟹（Crab）、剑客（Fencer）和使者（Emissary）——有软攻城模式，静止不动会给予某种好处。它们部署得如此之快，不这样做主要是操作速度（APM）的问题，所以与其让玩家每次想移动时都要与 UI 战斗去频繁点击部署，不如让单位自己部署。为了实现这一点，单位设计空间中刻意留出了一个大空档，没有任何单位拥有部署时间长到不想自动化，但又短到常见到你会怀念自动化的程度。在这个空档的另一边是毁灭者（Desolator），它可以部署成装甲形态，但时间足够长以至于这是一个罕见的决定，可以手动处理。因此，单位智能，或者说避免造成单位愚蠢的情况，是阻止足够廉价的变形存在的原因。

变形的最后一种用法是将单位随机变成完全不同的单位，显然没有任何充分理由。为什么潜伏者（Lurker）从刺蛇（Hydralisk）变形而来？答案似乎与任何明确的单位设计无关，更多是因为空槽位不足只能塞进去。刺蛇是最不尴尬的地方。ZK 当然不会做这么愚蠢的事情，比如让骑士（Knight）变形为螃蟹（Crab）而没有极其充分的理由，对吧？

……对吧？

但那是另一周的话题。

# Cold Take #13 - 支持简洁，反对核弹（Pro-Simplicity, Anti-Nuke）
日期：2024-06-02

Zero-K 有核弹（nuke）。这应该不奇怪，否则游戏类型警察会吊销我们的科幻执照。即使是少数奇幻游戏也有核弹，但无论在哪里发现，它们的工作方式都基本相同：

攀升到科技树的顶端。
建造核弹发射井（nuclear missile silo）。
建造核弹。
发射到敌方基地。
敌方现在少了一个基地。

《横扫千军》（Total Annihilation），一个以不简单著称的游戏，用反核弹导弹（anti-nuke missiles）拦截来袭核弹来增添趣味。这意味着准备充分的敌人可能不会失去基地。同样的系统可以在《横扫千军》的每一个后代中找到，其中许多在此基础上使其更加复杂。正如你现在可能预料的，Zero-K 走了另一条路，将反核弹简化到最简单的形式。我们仍然有可识别的《横扫千军》风格的核弹和反核弹，但它们并不比产生出色的战略核弹游戏所需更复杂。

战略上，反核弹有一个目的：成为敌人可以攻击并造成巨大伤害的弱点。至少，这是我们在《完全毁灭》（Complete Annihilation）中决定的，当时我们正在寻找鼓励精确打击敌方基地的方法。反核弹是终极弱点，确实，基地往往在反核弹被摧毁后不久爆炸。最接近的竞争者是剧烈爆炸的奇点反应堆（Singularity Reactor），但它通常可以建在侧面以最小化友军伤害，而且考虑到回收（reclaim），摧毁它的收益相对较小。反核弹也特别容易受到寡妇（Widows）和战术 EMP 导弹的攻击，因为暂时瘫痪的反核弹和死掉的一样糟糕。没有核弹和反核弹，军队必须艰难地穿越基地来摧毁它们，这会拖延已经获胜的比赛，给落后者很少的翻盘机会。

除了战略，反核弹对"酷"（Rule of Cool）也很棒。讽刺的是，给核弹一个硬克制（hardcounter）让它们更强大、更常见。这很棒，因为玩家似乎喜欢核弹，但不喜欢它们突然结束比赛。Zero-K 可能拥有最大的核弹，因为自《横扫千军》以来它只增不减，但在不同规模的游戏之间很难比较。我们的核弹当然足够大，可以摧毁整个基地或前线，并极大地改变战斗进程。

无法阻挡且高度毁灭性的核弹本质上就像《帝国时代》的奇观（Wonder）；它给游戏设定了严格的时间限制，然后宣布赢家。这没问题，烟花表演是个不错的附加，但如果太多游戏这样结束就变得不令人满意了。奇观和其他无法阻挡的超级武器互动性不足，所以输掉感觉不公平，而有明确克制的核弹给予防守方更多主动性。许多有无法阻挡核弹的游戏，如《星际争霸》和《命令与征服》，通过让核弹相对不那么令人印象深刻来避免这个问题。

Zero-K 甚至提供了可阻挡和不可阻挡核弹的直接比较，因为它的三个"真正"超级武器无法被阻挡。其中最便宜的比核弹发射井贵4倍，所以它们相当稀有，只能在游戏后期出现。尽管如此，真正的超级武器在社区中更有争议，也更难平衡，尽管核弹出现频率高得多。能够阻挡核弹的公平性似乎很重要。

如前所述，Zero-K 的反核弹尽可能简单。这没什么了不起，因为反核弹本质上只是射击另一枚导弹的导弹。其他游戏中反核弹的复杂性往往来自：

多种类型的核弹和反核弹，阵营之间可能略有不同。
可移动反核弹，常见于主力舰或脆弱的辅助车辆。
可囤积的反核弹导弹，必须在反核弹开火前生产。

Zero-K 没有这些花哨玩意。我们曾经都有，但它们因为违背设计目标或只是增加复杂性而被稳步移除。可移动反核弹在适当防御时几乎无法被消除，所以必须移除。囤积反核弹导弹可以阻止加速（boosting），这是一个有效目的（参见阿尔忒弥斯（Artemis）），但它有严重的缺点。没有什么比认为你有反核弹，却发现你忘了囤够导弹更糟糕的了。被核弹炸已经不是最好的体验了，所以我们至少能让它感觉尽可能公平。

Zero-K 只有一种反核弹，以专用建筑形式存在。它极其原子化，且相对昂贵，将平衡倾向于进攻而非防御。核弹发射井花费8000，每枚导弹3240，而反核弹花费3000。两个反核弹在大型地图上只提供最基本的覆盖，所以可靠覆盖可能比它要阻挡的核弹还贵。这是有意的，因为随着游戏进展进攻应该增长威力，核弹需要常见以给基地弱点。至少反核弹会阻挡飞过其覆盖范围的导弹，因为否则用圆形覆盖矩形地图背面会花费太多。

反核弹的简洁性延伸到它们的物理特性，属于游戏化光谱的一端。反核弹导弹穿透一切；友军、敌人、地形，因为利害关系太大。系统必须足够可预测才能感觉公平，公平性的重要性随着失败成本的增加而增加。所以如果地形阻挡反核弹，游戏就必须绝对清楚何时会发生这种情况。但这样的 UI 听起来很难又没意义，所以我们给了反核弹一条很少会碰撞到地形的飞行路径，然后就完事了。但反核弹更不寻常的方面是它们能射击什么。

反核弹覆盖一个简单的圆形区域，保证没有核弹会落在或穿过这个区域（同时发射的核弹少于三枚时）。但反核弹的"攻击范围"——即核弹需要多近反核弹才会开火——超出了这个圆圈，这样它可以在核弹落地前射击。从单位物理的角度思考这很奇怪，因为反核弹理论上可以保护其侧面的目标。确实，更物理化的反核弹可以射击所有可达的核弹，但这会给它一个奇怪的有效覆盖区域，取决于来袭导弹的方向和俯仰角，甚至附近地形的形状。传达这样的覆盖范围将是 UI 的噩梦，并不可避免地被误解，让拦截感觉像掷骰子。

如果你真的要较真，那么，技术上，Zero-K 中阻挡核弹远非简单。因为每个单位都能做到，我甚至见过以下单位用于拦截核弹：

Dirtbag
Jack
Raven
Athena
Funnelweb
Odin（就在本文发布后不久）

核弹与实体碰撞，所以任何有实体的东西都能阻挡它们。然而，它们不与友军碰撞，因为因一架误入的轰炸机而炸掉自己的基地令人愤怒，而用跳跃的 Dirtbag 炸掉敌方基地则很搞笑。大多数单位通过在发射井发射导弹的瞬间跳过或飞过来拦截核弹。Funnelweb 是一个显著例外，它的护盾有足够电荷阻挡核弹。至少，如果你的"阻挡"指的是"在略微不同的位置爆炸"。护盾只是停止弹道，它们仍然爆炸，所以范围伤害不受影响。这意味着 Funnelweb 只能阻挡一枚核弹。尽管如此，这是一种绝望的技术，可以减轻一些伤害，特别是当 Funnelweb 位置很高或被弹射到空中时。

但反核弹说够了，核弹本身呢？这里可说的少得多，因为它们遵循基本的核弹公式，改进主要通过反核弹实现。查看 BAR 中的核心核弹，它有相同的伤害和范围，所以共同祖先至少可追溯到2007年。但确实有一些变化。

核弹发射井的打开动画在2010年代中期被修改以削弱 Athena 的核弹拦截。以前，核弹发射井只在被命令开火时才打开，这让警觉的敌人可以飞一架隐形 Athena 到上方拦截。这会使 Athena 解除隐形，所以时机很重要。玩家通过用等待命令手动预备发射井来反制。所以发射井被改为在囤积导弹后立即打开，使其能立即开火，这只是消除单位 AI 变蠢的一种方式。

闪烁的"核弹已发射"指示器是另一个相对较新的添加。这更多是可访问性方面的东西，因为我们一直围绕全局可听的发射声音来平衡核弹。但并非所有声音都能听到，这成了不公平的来源。替代方案是移除声音的全局性，但这会移除我最喜欢的核弹部分；心理博弈。核弹需要一段时间落地，所以军队如果猜对了着陆点就能勉强躲开。这创造了一个大规模军队调动的猜测游戏，甚至可以被发射核弹的队伍利用或引导。

但为什么人们会向军队发射核弹？因为反核弹覆盖在敌人领地边缘更容易被突破，所以有时军队是唯一可用的目标。所有机制相互支持。

迄今为止最有趣，也可能是最古老的核弹变化发生在《完全毁灭》时期。你有没有想过为什么核弹看起来像火箭，但发射时直接满速没有加速？这要归功于一位叫 det 的开发者，Detriment（毁灭者）的名字来源，他在 Azure Rampart 上的一场混战中向邻居发射了一枚核弹。他的邻居有一个反核弹，不幸的是，它在 det 的核弹攻击范围内。这导致反核弹立即开火，所以它迅速飞过来，在 det 的基地正上方炸掉了核弹。核弹因此获得了更多发射速度，尽管如今"新"的反核弹打开动画可能已经使增加的速度变得多余。

更多单位和机制因发生在 Azure Rampart 上 det 身上的事情而改变，但这些将等待未来文章。

# Cold Take #14 - 人人免费工厂（Free Factories For All）
日期：2024-06-30

每场比赛都必须以某种方式开始，策略游戏在玩家知道对手动向之前就开始了。我称这个阶段为"布局"（setup），在这个阶段，玩家倾向于建设基地并部署一些部队，所有这些都没有外界干扰。明确地说，布局是接触敌人之前发生的一切，或者换句话说，在你能够根据对手做出决策之前。根据这个定义，给玩家一个免费的侦察兵可以让布局阶段相对简短。

更注重经济的游戏往往有相对长的建设期，但通过上述侦察兵来缓解。《横扫千军》风格的游戏处于经济光谱的更经济端，但缺乏这个缓解因素。它们让玩家从一个单独的慢速指挥官开始，在能够负担工厂和侦察兵之前，通常需要先建设一些经济。这导致了很长的布局，这被《完全毁灭》开发者所不喜欢，所以我们着手缩短它。

在看解决方案之前，我们应该先深入了解我们试图解决什么，理想情况下挑出有问题的部分。我们还需要知道如何衡量成功，这引出了以下问题：在任何给定的游戏中，布局到底有多长？由于"外界干扰"的本质，答案有些模糊。考虑现代 Zero-K 中的1v1，理论上你可以放置一个飞机工厂（Airplane Plant）并立即派出一架雨燕（Swift）飞越地图，整个过程约20秒。双方现在都大致了解对方在做什么，所以他们离开了布局阶段。

但是等等，如果20秒后没有雨燕飞过你的基地呢？在这种情况下，你的对手没有放置飞机并扔出雨燕，这在技术上也是信息。那么，你离开布局了吗？没有，因为我想从这个定义中获得更多。关键是飞机工厂接雨燕侦察是糟糕的开局，所以不观察它几乎学不到什么。总的来说，负面观察应该以确定程度来对待。30秒内什么都没看到很正常，所以基本没有揭示任何信息。三分钟内什么都没看到就更令人担忧，可能需要一些盲目的反偷袭措施。同样，如果人们总是把炮塔放在特定位置，那么侦察那个炮塔就毫无意义。

简而言之，布局阶段的长度大约相当于将对方的行为从"他们几乎什么都可能做"缩小所需的时间。至于人们在这个阶段做什么，归结为以下内容：

决定他们想在布局结束时拥有什么。
尽可能最优地执行该决策。

执行部分在 Zero-K 设计中引发了一些警示信号，因为在没有对手的情况下，执行失败是由与 UI 战斗引起的。顺便说一句，这就是为什么命令可以在游戏前排队，以及为什么工厂会自动播放展开动画。展开的时间给了玩家机会点击工厂并告诉它建造东西，而不用担心浪费时间。

这种对布局的分析揭示了我们想要保留的一个方面：决策。理想情况下，这些决策体现了策略、创造力和定制化，这些对 Zero-K 很重要。"自然"也很重要，短暂的布局阶段比某种预游戏部队部署菜单更有机。布局与游戏其余部分之间的模糊边界也相当巧妙。所以目标是缩短布局而不移除决策。

我们的第一个解决方案是加速（Boost）机制。指挥官获得个人资源供应，表现为一个红色光环，可以更快速地消耗。这让玩家能快速加速出一个工厂、一点经济和几个单位。它加快了速度，但太有创意了。玩家会运输盟友指挥官横穿地图来加速在敌方基地建造炮塔。其他策略包括联合从单个工厂加速出可观的军队，这对将大部分加速用在其他事情上的队伍非常有效。

加速rush向我们强调了我们已经知道的关于团队游戏的事情：工厂是资源的浪费。八人团队需要所有八个工厂的几率有多大，当只有两三个自然前线时？非常低，但整场比赛协助盟友工厂很无聊，所以很少有人做。这会在锦标赛中发生，或被偶尔的战队组队"滥用"，但普通游戏玩家谴责它为"廉价"，而且很快就过时了。

这里的一般原则是，人们会把游戏中的乐趣优化掉。作为回应，游戏需要有有趣的目标解决方案，鼓励人们以有趣的方式玩耍。竞技RTS的目标是胜利，所以好的策略也应该是有趣的策略。《横扫千军》的标准方法让每个玩家花费宝贵的初始资源来建造工厂，只是为了好玩，这严重违反了这一原则，加速让它变得更糟。所以我们重新开始。

加速rush巩固了一条新的设计规则，即工厂所有权是乐趣的必要条件，所以下一个实验是直接给每个人一个免费工厂。这被证明是足够的，因此"Facplop"优惠券已经存在了大约16年，没有变化。它主要解决了布局问题，虽然我们也增加了指挥官的固有收入。超过一个金属钻机和两个太阳能板"搬进"了指挥官，让你从游戏开始就能更容易使用工厂。

Facplop 并没有完全消灭辅助rush，因为它们仍然会不时出现，但现在以玩家回收免费工厂的形式。我们不能简单地让工厂更便宜，减少回收的金属量，因为那会违背工厂作为阵营的本质。我们也不愿意让免费工厂不可回收，或给它降低回收价值，因为那是任意的。

值得庆幸的是，新的辅助rush往往勉强处于可行的边缘。削弱它们主要是平衡调整的问题，有时我们甚至可以等待反制手段被普及。新rush差得多是因为回收消耗建造力，只产出金属。这意味着必须建造能源建筑来及时消耗多余的金属，削弱了rush的大部分威力。只有圣骑士（Paladin）rush需要显著改变，这就是为什么步行者中心（Strider Hub）最终有了电网需求。

回顾我们缩短布局的原因，我有时怀疑我们是否做得太短了。我们可能忽略了给新手更多呼吸空间的实用价值，或者强制在匹配游戏之间形成某种休息形式。我甚至从《帝国时代》游戏前十分钟的执行挑战中找到乐趣，尽管我不想在竞技中反复练习。RTS 的整体趋势似乎与 Zero-K 一致，特别是《星际争霸2》的扩展包中建设阶段被缩短的方式。也许一切都回到了尊重玩家的时间。如果我们假设 Zero-K 将其布局缩短了一分钟（这似乎是低估了），那么这样做已经为我个人节省了整整四天。在整个社区中节省的时间一定是巨大的。

# Cold Take #15 - 经验教训：老兵等级（Experiences With Veterancy）
日期：2024-07-28

这周我们回到 Sprung 关于 Zero-K 缺失的单位改善机制系列。继"强力变形"之后，现在轮到老兵等级了。这个仍然存在，嗯，某种意义上。

早在 Zero-K 最古老的祖先《横扫千军》中，单位可以通过击杀获得老兵等级。《横扫千军》的界面对此相当隐晦：除了在鼠标悬停在至少5次击杀的单位上时添加"Veteran"这个词外，没有提供更多信息。我们这些用大贝莎（Big Berthas）轰击敌方基地数天的人经常注意到准确度的提高。通过大量努力，也有可能发现生命值的轻微增加。但前线作战单位就没那么幸运了，老兵等级很难获得，效果也几乎察觉不到。

然而 Zero-K 不受《横扫千军》引擎的限制，它运行在 Spring（现 Recoil）上，后者从早期就扩展了老兵等级。所以到《完全毁灭》出现时，老兵等级做了三件事：

增加单位的生命值。
减少单位武器的装填时间。
减少单位武器的不准确性。

Spring 还转向了更公平的经验分配系统：计算造成的伤害而不是击杀，并考虑攻击者和防御者的成本。这意味着像剃刀（Glaive）这样廉价的玻璃炮通过射击指挥官可以积累大量经验，而毁灭者（Detriment）摧毁整个军队却学不到多少。从某种意义上说，老兵等级取决于一个单位在投入资源后的预期表现。更高等级的老兵单位在受伤时也给予更多老兵经验，奖励通过公式平滑应用，而不是在固定阈值。老兵等级更易获得且更有影响，但这揭示了问题。

考虑增加生命值，在某些生命值阈值下，单位开始需要额外一击才能被击杀。这本身没问题：《命令与征服》游戏中，老兵等级同时增加生命和伤害，通常让反步兵单位如谭雅（Tanya）每击对基础步兵造成90%或110%的生命值伤害，特别是根据射手和/或目标的老兵等级在1击和2击之间切换。阻止 ZK 中这种优雅的因素之一是单位间的石头剪刀关系没有如此清晰界定或紧密遵循。如果没有基础步兵和突击队员，也没有离散的军衔标志，你就不能有简单的规则如"有两条军衔的基础步兵能在突击队员一击下存活"。角色克制关系如袭击者（raider，大致对应基础步兵）和暴徒（riot，谭雅）存在，但内部差异巨大。玩家必须学习和记忆巨大的关系表格，了解 ZK 的十几种袭击者在任何给定军衔配对下对抗任何其他袭击者的表现，才能保持竞争力，更不用说所有其他单位类型了。

装填时间造成了更多麻烦，因为时间实际上是离散的。通俗地说，这意味着装填时间必须以0.033秒为步长。同样，袭击者往往受此影响最大，因为它们快速射击的武器没有太多调整空间。剃刀的装填时间是0.1秒，即三个0.033步长。从那里，你只能减少到两个步长，代表+50%的射速。这太陡峭了！另一个影响是，一些具有慢速弹道的单位被设计为在其装填完成前刚好命中目标。这是一种粗糙的过量伤害保护，因为如果目标死了你就不需要射击两次。装填时间的轻微改进意味着单位会将后续射击浪费在即将死亡的单位上。

不准确性通常是三个问题中最小的，因为大多数单位都非常精准。但有些单位故意不准确，强制的精度提升会当场大幅重新设计单位，让你得到对原始角色可能无用的东西。例如，一个精准的震波（Tremor）无法伤害其通常范围内的大部分目标，使其对隐形或大范围地形变形无用。（顺便说一句，原始的《完全毁灭》震波只是一个车轮上的小型迪斯科狂欢派对。）

除了具体问题，所有老兵等级奖励都进一步强化了这个已经容易滚雪球的类型中的滚雪球效应。失去单位使你的军队变弱，但有了老兵等级也使敌军变强。这需要某种方式避免势均力敌的战斗在几次早期击杀后就滚雪球变成胜利，要么通过削弱优势，要么通过促进翻盘。《命令与征服》似乎主要通过让单位之间有很强的克制关系来实现，即使精英单位也无法对抗其克制者；同时有老兵等级上限，单位不能无限成长。《魔兽争霸》和 DotA 英雄在较高等级被击杀时给予更多经验，需要更长时间重生，且每一级都更难获得。当然我们可以想出不错的系统，但额外的复杂性值得吗？此外，ZK 已经在击杀敌方单位时给你一点额外奖励；但其他游戏通过老兵等级做到这点，ZK 给你残骸（wreckage）来回收。这可能感觉不如获得精英单位那么酷，但这是一个如此深刻而优雅的机制，我们相当满意。

避免老兵等级奖励的另一个原因是，简单的属性加成影响的是公式而不是物理。《命令与征服》在这方面也有不错的方法：达到最高"精英"等级的老兵单位有时除了原始数值属性外还会获得行为改进。例如，重型坦克获得溅射伤害，或《红色警戒3》步兵获得对狗眩晕的免疫。但在那时你变成了别的东西，在某些方面是不同的单位，这应该反映在其视觉效果上。军衔图标，在其他RTS中的使用方式，对于纯粹的属性来说太显眼，而对于行为差异来说又太不显眼。这里的主要障碍是 ZK 的美术本质上是多年来从广泛志愿者那里收集的一批松散资源，根本没有人力引入这种额外内容。历史上我们确实有一些模型拥有少数逐渐增强外观的变体，但这只是沧海一粟，几乎不一致。这也是我们没有研究（research）的最大原因之一——一个更强的单位需要在视觉上也显得更强，无论通过研究、老兵等级还是任何其他机制。所见即所得。

很快，老兵等级的属性增益被移除，当时技术水平相当低，所以实现武器类型改变之类的东西不太可能。但老兵等级很酷，我们必须拥有它！正如变形文章中暗示的，解决方案是让拥有足够老兵等级的单位变形为任何稍微更强且看起来大致合适的东西，然后有时甚至不是那样。跳蚤（Flea）变形为剃刀，因为它们当时都在"隐形"工厂，即Arm T1 KBot。剃刀变形为掠夺者（Reaver），然后骑士（Knight）。骑士变形为螃蟹（Crab），因为骑士在"蜘蛛"工厂，Arm T2 KBot。这个链条以毁灭者（Detriment）结束。许多单位可以通过老兵等级以这种方式升级，许多升级路径在阵营合并和工厂重组后仍然保留。

人们喜欢这个！跳蚤进化成毁灭者的幻想看起来真的很酷。不幸的是，它仍然是幻想。当然，人们可以在对抗AI的游戏中做到，而且在PvP中有一个已知案例，一个前十玩家在戏弄新手时炫耀了一把，但链条变形在竞技游戏中从未真正发生过。只有少数可行的变形是人们追求的目标。但除了被有效地添加到随机单位而没有事先设计之外，还有变形的所有问题，而老兵等级让它变得更糟，因为它是被动的。当你的骑士被散兵（skirmishers）克制时，螃蟹是完美的答案——但你不能计划你的骑士获得足够的老兵等级。你只能希望其中一个在战斗中随机存活足够长时间，而强大的力量摆动不应该来自如此不可靠的来源。

最终，这些基于老兵等级的变形选项也被移除了。作为趣闻，这揭示了一个小的工厂设计问题，导致了红背蜘蛛（Redback）的创造。蜘蛛当时没有"伤害输出暴徒"（DPS riot）的角色，而跳蚤如此便宜，你可以可靠地让它达到足够高的等级，通过射击任何东西约两秒钟直接变形为掠夺者。在失去掠夺者后，阵容中的空缺变得明显，所以红背蜘蛛被快速拼凑出来填补它。

现代 Zero-K 单位仍然获得老兵等级，由其上方的军衔图标表示。机制上，它绝对没有任何作用（除了激励模组制作者）。但获得图标让玩家本能地感觉良好和获得奖励，所以我们保留它。然而，玩家经常开始怀疑老兵等级实际上没有任何好处，他们会问问题。我应该追求获得老兵等级吗？我应该保护我的老兵吗？我们的回答是：是的！获得军衔的效果是你的敌人不再拥有你为了获得那些军衔而必须杀死的单位，这已经直接帮助了你的目标。老兵等级也是明确衡量一个单位表现如何的好方法，尽管对炮兵和超级武器等经常射入战争迷雾的单位类别有些偏见，因为为了防止信息泄露，只计算可见伤害。

我们避免奖励老兵等级的最终线索可以通过观察高水平的《魔兽争霸3》或 DotA 游戏来发现。毕竟，升级英雄如果不是老兵等级的极端案例是什么？在这两个游戏中，你经常想杀死自己的单位只是为了防止敌人获得经验。在 DotA 中，元游戏被优化到每个英雄获得"正确"数量的经验（和金币，同样来自击杀敌人，所以也可以被视为老兵等级的软形式），辅助有时会放弃它以支持核心。

在 Zero-K 中我们预见问题并立即咬住未来可能出现的任何子弹。强大的UI可以让你从老兵等级优化中挤出很多，比如选择性自毁以防止敌方老兵等级，以及停止开火以免让成长性较差的单位从较好的单位那里"抢最后一击"。高技能上限？当然。大部分被自动化过时了？也许。我们想发现这一点，可能引导玩家陷入优化这种交战细节的泥沼吗？宁愿不要。

# Cold Take #16 - 瞄准与开火（Aim and Fire）
日期：2024-08-11

终于要讨论武器了，从它们如何开火开始。武器设计空间巨大，这要归功于弹道物理，我们充分利用了它。但这种自由是双刃剑，因为完整设计空间的大部分包含令人沮丧或难以理解的机制。在这个空间中有有趣的游戏，但 Zero-K 首先是一个策略游戏，这意味着它需要足够稳定和可预测来奖励规划。所以，为了对抗物理的完全自由，我们在武器设计周围安装了一些护栏；一套松散的规则，如果遵循，可以限制武器破坏游戏的程度。

武器设计的许多方面都有规则，但围绕瞄准和开火的规则是最重要的。这些规则考虑了我们设计目标的以下三个方面：

智能性（Intelligence）——瞄准和开火只是能力，单位应该擅长使用它们的能力。
可预测性（Predictability）——玩家应该能够学习单位如何瞄准以及什么导致它们开火。
逼真性（Verisimilitude）——武器应该感觉可以在游戏世界可观察虚构中以合理的方式使用，而不仅仅是一堆任意机制的组合。

你可能已经猜到，弹道物理导致了这些目标之间的大多数冲突。这组特定问题主要限于《横扫千军》的后代，因为弹道在其他类型游戏中作用较小。标准RTS弹道是开火和造成伤害之间的一个动画和短延迟，因为它们保证命中目标，且只命中目标（除非发生不寻常情况，如死亡或传送）。这限制了单位可能愚蠢地误用武器的程度，并允许武器开火相当可预测。

《横扫千军》可能使用弹道物理来增强逼真性，但我不同意这意味着其他游戏天生缺乏它。每个RTS都简化了现实的许多方面，简化的弹道很受欢迎。模拟与现实的贴近程度只是与合理一致且可信的游戏世界体验松散相关。当然，有许多玩家追求严格的真实感，但 Zero-K 模拟弹道不是为了真实。我们使用物理来创建复杂但根本简单的机制，奖励直觉和创造力。但我离题了，物理通常似乎对上述目标造成问题，而不是解决它们。

物理弹道难以使用。你向敌人射击，但然后敌人移动，弹道击中空地。所以你向敌人将要去的地方射击，但他们向另一个方向躲避，或者你由于固有的武器散布而未命中。单位在提前量方面做得不错，但很多仍然未命中。这使得决定何时射击至关重要，但这个决定必须快速，这可能使其变得不可预测。

更糟的是，Zero-K 所遵循的逼真性感是，理想情况下，武器只提供两个世界内能力：

改变武器瞄准的位置。
开火武器。

换句话说，我们认为单位拥有与第一人称射击游戏玩家相同的基本能力。实际上实现没有那么远，但至少我们让单位盲目地向地面开火，因为那里可能有隐形炸弹。但即使这种有限的能力意味着，理论上，玩家可以手动控制每一枪，这引发了与UI战斗的幽灵。所以单位必须足够智能，让玩家不需要诉诸手动控制。

我们可以编写一些炫酷的第一人称射击级别AI，但那样战斗就变成了看着单位在屏幕上飞驰。这让我们回到可预测性，也许智能性和可预测性本质上是矛盾的。不用担心，通过施加旨在使武器更简单的限制可以打破僵局，这样单位可以在不破坏可预测性的"智能预算"的情况下合理智能。这些限制往往相当游戏化，超出纯物理范畴，这导致与逼真性的权衡。

考虑无处不在的范围环（range ring）及其对制导导弹的含义。剑客（Fencer）只能向其周围圆圈内的敌人发射导弹，但在这个圆圈边缘跳舞的敌人仍然会被击中，这有点奇怪。如果我们让目标们争论会怎样？

逼真性：这太荒谬了！导弹显然可以命中超出其任意"范围"的单位。剑客应该在更多情况下射击，向所有它物理上能命中的东西。
智能性：嗯……听起来会被利用，我们不想向勉强处于物理范围内的目标开火。边界目标可以（也应该）只是后退，让导弹在命中前燃料耗尽。
可预测性：我们只能让导弹射击建筑，而不是单位。那些保证会静止不动。
逼真性：那太任意了！
智能性：我们用目标的最大速度怎么样，考虑减速和EMP、爬坡、朝向、角速度……
性能：呃……
可预测性：而怎么有人能知道哪些因素组合会导致或不会导致剑客发射导弹？

此时我们可能用一些"导弹锁定范围"的含糊说法来安抚逼真性，但背景故事无法匹敌玩家在游戏中能观察到的。此外，我们有许多弹道武器也能稍微超出其范围环射击，这不能用任何"导弹锁定"来解释。所以事实是逼真性输掉了这场战斗，我们能做的就是让这些情况尽可能简单和一致。

Zero-K 的逼真性要求单位能做它们看起来有能力的一切，可预测性要求对玩家最容易传达的任何东西，智能性只关心结果能力很难搞砸。考虑这些约束设计武器看起来有点像这样：

想象武器的理想形式，充满逼真性。剃刀（Glaive）手里拿着枪。它可以向任何方向移动手臂并在任何时候扣动扳机。它是一个真正的机器人。
然后考虑武器的正确使用。剃刀应该花时间向天空疯狂射击吗？可能不会，除非敌人在头顶飞行。
然后施加你想要的尽可能多的行为或机制限制，只要你只剔除理想武器可能使用的愚蠢或无意义的方式。这些应该帮助单位避免变蠢。向天空射击几乎总是无意义的，所以剃刀只在头顶有敌人飞行时才这样做。
最后，添加一小套标准限制，如范围环。一致性是关键，因为目标是可预测性，而任意限制最伤害逼真性。

在实践中，过程不像处理四个步骤那么简单。大多数限制同时影响所有三个方面，有时你需要修改武器的理想版本以使下游实现可行。也可能从实现开始，然后反向推导出什么理想武器会导致它。

这个过程最重要的推论之一是武器行为不能依赖目标的存在。这被称为"虚空光辉剑"（No Void Ray）规则，因为它是开发回《星际争霸2》有一个单位开火越久伤害越高的时期。该单位在第一个扩展包中被重新设计，可能因为在《星际争霸》中射击自己的东西是允许的，但激励玩家这样做看起来有点蠢。无论如何，虚空光辉剑甚至无法进入 Zero-K，因为没有什么能阻止它在空闲时向天空射击。保持其伤害加成是明智的，所以我们不能良心上阻止它。这会让机制变得无意义，所以我们一开始就会避免添加它。

同样的规则适用于锁定机制，或惩罚单位切换目标的机制。向一排敌人射击的线性武器不应关心单位"瞄准"哪个单位，因为它都涉及向同一方向瞄准。任何机械区分都会将"目标选择"的界面抽象拖入游戏世界领域。在导致单位瞄准和开火相同方向的命令之间，不应该有"最佳命令"。命令旨在帮助玩家操纵游戏世界，而不是它们自己成为需要优化的东西。

有办法绕过"虚空光辉剑"规则。例如，武器可以对目标施加递增状态效果，导致目标从该类型武器受到更多伤害。这看起来有点像虚空光辉剑，甚至解决了向自己东西射击来充能的问题。然而，这个特定机制会被"物理对公式"（Physics vs. Formulas）所阻止，但通常有方法在武器设计指南内实现设计。

也许最好将武器设计规则视为一套保证，而不是一套限制。在规则内工作保持 Zero-K 一致，并划定了设计空间中相对安全的部分。这些限制可以辅助创造力，不仅通过通常的论点，而且因为安全保证释放了否则会花费在确保某个想法不会完全破坏游戏的脑力。例如，规则内在地禁止激励玩家向自己基地射击的设计。所以规则可以弯曲，但需要更多小心，因为游戏的其余部分建立在它们的假设之上。

防空武器是对武器可以向任何方向开火这一想法最公然的违反者。如此公然实际上很好，因为一个大而明确定义的例外感觉远不那么任意。每种武器都属于以下类别之一：

只能射击飞行器（aircraft）的武器。拥有这些武器的单位在两词角色描述中有"防空"（Anti-Air）。
所有其他武器。这些武器如果有合理命中机会就会射击飞行器。近战武器可以且确实能击中飞机，当挥舞者被抛得足够高时。

我们可以避免创建特殊的防空类别。一种方法是让飞行器显著变慢或变脆弱，允许它们的最佳克制足够弱以也能对地射击。不幸的是这会让空军更像地面单位，或者只是总体变弱，违反了奎特法则（Quant's Rule）。我们也考虑给防空武器向上朝向的射击弧线，允许它们向天空射击而不击中地面。这对可预测性有不良影响，因为倒置圆锥会给防空可变的射程，取决于地形和巡航高度，同时成为对悬崖的绝佳火炮。倒置圆锥也只是把问题推后，因为它提出了为什么这个了不起的武器不能以一定角度建造的问题。

没有一个防空的替代方案能像定义一个简单类别那样好，让空军有自由变得强大且不同。尽管如此，大多数有空军的游戏都有防空，所以也许 Zero-K 更有趣的部分是缺乏严格的对地限制（Anti-Ground）。这个选择是一种设计极简主义，我们从弹道物理开始，然后在上面添加尽可能少的机制。我们可以有广泛的对地限制，要么作为目标类别，要么只是让飞机飞得如此之快如此之高以至于只有防空能击中它们，但游戏现状运行良好。

说实话，地面单位向飞机射击是一个特性，而不是缺陷。目标不是真实感，目标是相当一致的世界感觉，我们喜欢游戏不同部分之间的额外互动。从某种意义上说，我们选择了自己的逼真性风味，因为当将其推到逻辑结论时，我们喜欢最终到达的地方。但现在感觉这篇文章即将自我折叠。只需记住，对于未来的文章或讨论，Zero-K 单位通常被允许瞄准和射击它们想要的任何东西，只要它们同意避免使用这种自由去做任何太无意义或愚蠢的事情。

# Cold Take #17 - 战术与战略隐形（Tactical and Strategic Cloaking）
日期：2024-08-25

隐形（Cloaking）很酷。至少，某些类型的隐形很酷。取决于你问谁。让我重新开始。

隐形是两极分化的。有些人喜欢，有些人讨厌，而作为隐形打击的接收方可能感觉相当糟糕。更复杂的是，隐形的具体机制在不同游戏间差异巨大。有时隐形单位几乎免疫伤害，可以肆无忌惮地战斗，而在另一个游戏中隐形可能被最轻微的接触驱散。但真正有趣的是同一套隐形机制如何在同一个游戏中以不同方式使用。

Zero-K 有很多隐形，这相当于很多出错的方式。除了个人隐形，隐形场（cloaking fields）在《完全毁灭》早期作为区分 Arm 和 Core 的方式被添加，从那以后我们一直在削弱它们。随着时间的推移，我们开始从两个不同用例的角度来思考隐形：

战术隐形（Tactical Cloak）：敌人知道你的单位在哪里，但很难攻击它们。
战略隐形（Strategic Cloak）：那种偷偷摸摸的隐形，敌人不知道你在这个区域有单位。

《星际争霸》中的黑暗圣堂武士（Dark Templars）是典型的战术隐形者。当基地满是黑暗圣堂武士时，从所有死亡的工人和科幻噪音中可以明显看出，但没有特殊探测器单位就无法对抗它们。这个例子依赖于《星际争霸》有很少的伤害隐形单位的方式（它们甚至在战斗时保持隐形），但战术隐形的概念适用于所有游戏。Zero-K 和许多其他游戏中的战术隐形取决于隐形单位确切位置的不确定性。当然，单位应该向"空地"射击，如果它们知道有隐形单位在那里，但不需要太多不确定性就足以让这样的射击变得高度可疑。

甚至可以说《星际争霸》缺乏"真正的隐形"（如果关于"真正"任何东西的争论不是往往无意义的话），因为单位向敌人展示自己时会有微弱的闪光（除了埋地单位），这可能有助于隐形感觉公平。但 Zero-K 无法使用这样的机制，因为玩家和单位知识之间的这种脱节违背了其他设计。

与UI战斗的情况在玩家知道某些事情但无法将这些知识传达给其单位时蓬勃发展。隐形在 Zero-K 中相对容易被驱散，正是出于这个原因，目标是单位只有在其位置存在真正不确定性时才保持隐形。首先，建筑不能隐形，因为没人想教单位如何向隐形炮塔射击。单位开火时也解除隐形，因为弹道的突然出现是明显的暴露，当它们受到伤害时，因为空中的弹道消失也很可疑。有非可疑的伤害方式，如走过火焰，但它们也会解除隐形以保持一致性并为隐形提供更多克制手段。

隐形也会因过于接近敌人而被穿透，"接近"由隐形单位的解除隐形范围定义。这个范围是个人隐形的重要平衡因素，有些个人解除隐形范围相当大。其他单位有基于大小的一致解除隐形范围，当它们进入隐形场时，隐形场也对个人隐形者应用这个范围（如果这是改进的话）。隐形场投射器不能减少其解除隐形范围，移动投射器有大的解除隐形范围来控制战术隐形。

基于接近的解除隐形让 Zero-K 完全避免了被动远程探测能力。主动探测能力存在，如凝固汽油导弹和麻雀（Sparrow）的死亡暴露能力，但隐形单位通常通过基地设计和轻型单位巡逻的组合来克制。我们喜欢这个系统中的细微差别，它比让单位被动地抵消区域内所有隐形有更多的玩法。

缺乏广泛的探测器让战略隐形在游戏后期更加可行。这肯定是每个听到隐形的人的梦想，他们想成为詹姆斯·邦德；潜入敌方基地并炸掉它。这很棒，但 Zero-K 也想成为一个有趣的多人游戏，而基地突然爆炸并不好。所以战略隐形主要针对在基地内和无人区隐藏军队调动进行调优。尽管如此，渗透总是可能的，我们的目标只是让让一支隐形军队进入你的基地感觉像是一个可避免的错误，而且，反核弹仍然为偷偷摸摸的渗透者提供炸掉整个基地的方式。

与大多数定义一样，有边界情况模糊了战术和战略隐形之间的区别。一支隐形军队可能开始时是战略隐形，在攻击目标时变为战术隐形，然后在幸存者脱离接触后慢慢转回战略隐形。战略隐形的特别流行用法甚至在变得足够标准可预期时可能采取战术方面。DotA 的烟雾（Smoke of Deceit）就是一个很好的例子，因为它有巨大的解除隐形半径，显然是作为战略隐形工具设计的，但有技巧的对手会预期烟雾并在感觉它们很可能是时做出反应。

除了边界情况，这些类别有什么意义？意义在于玩家反馈往往围绕不良事件，而涉及隐形的事件往往可以干净地分为这两类之一。对反馈进行分类很好，但分类真正发挥作用的地方在于它如何告知可能需要的平衡调整类型。战略隐形的问题暗示削弱速度或探测范围，就像巫师（Conjurer）获得隐形场后一样。战术隐形的问题暗示削弱重新隐形时间和隐形场能覆盖的区域。

如前所述，隐形场自首次亮相以来一直被大幅削弱。部分原因是它们被摸透了，但可能也有最初偏向使其强大的偏见。单位以前在受伤时只解除很短时间的隐形，隐形场也在稳步变小。建筑曾经可以隐形，直到我们意识到大多数建筑隐形很无聊或令人沮丧。隐形你的反核弹和奇点反应堆显然很好，但这完全去除了这些建筑作为多汁弱点的意义。

隐形场的战术隐形比所有其他形式的隐形造成了更多问题。个人隐形没问题，因为任何问题都与单个单位相关，平衡单位相对简单。甚至幻影（Phantom）无需解除隐形就能开火的能力也已被平衡。隐形场则是另一回事，因为它们赋予所有东西隐形，而我们的原则阻止我们做出任意例外。散兵和炮兵是个问题，因为隐形它们可能使报复太难。为了解决这个问题，我们削弱了非个人隐形的重新隐形时间为六秒。这仍然让军队进行隐形突袭，但让单位在战斗时保持解除隐形状态。

炸弹是最令人兴奋的单位放在隐形场下，也是最难平衡的。我们倾向于削弱场投射器本身，减少场半径和移动速度，同时增加解除隐形范围。目标不是移除隐形炸弹，因为那会移除想出一个很酷组合然后意识到它有效的乐趣。相反，目标是给防守方一些事情可做。这是让隐形感觉公平的一部分；攻击方和防守方之间的互动不应该在隐形单位被发现时就结束。就炸弹而言，找到并摧毁隐形场是一种安慰奖。这也是为什么地雷（Skuttle）比寡妇（Widow）更难平衡的原因之一：它们单独工作。

除了建筑不可隐形外，隐形在几种情况下被禁用。一个是投射护盾的单位不能隐形。这是单位智能问题；如果你能看到护盾，你就应该能定位投射它的单位。技术上护盾也可以隐形，但那样弹道会在空中碰撞，或者护盾必须在单位解除隐形时突然出现，看起来很乱。另一个例外是水下隐形；完全浸没的单位不能隐形。这主要是因为缺乏好方法通过常规手段探测水下单位，以及水下单位已经需要声呐（sonar）来探测。

隐形与声呐的关系提出了一个好问题；隐形是否只是视野（line of sight）的延伸？视觉、雷达、隐形和声呐可以统一在"如何不被看见"（How Not To Be Seen）的旗帜下。事实上，我关于战术和战略隐形的大部分内容可以应用于视野和雷达。有时缺乏视野使战斗困难，而有时缺乏视野隐藏了有东西可战斗的事实。所以有平行之处，也有许多差异，探索这些将等待未来文章。

# Cold Take #18 - 地形至关重要
日期：2024-09-07

Zero-K 的地图对单位移动设置了出人意料的硬性限制。相反，许多地图主要使用地形来影响位置的价值，以及它们的可达性和防御性。还有足够的空间容纳巨大的悬崖和广阔的海洋，但真正让我们地形与众不同的是每个山丘和山谷的微妙影响。这甚至还未考虑爆炸和地形改造在游戏中如何改变地形。地形改造提升了地形高度，但它需要一个由重要且细微地形构成的强大基础才能有意义。因此，现在是时候看看 Zero-K 如何运用地形了。

地图可以分解为三个主要部分。

形状：地图上每个点的地形高度。
纹理：绘制在地形上的颜色、反射率和图形粗糙度。
资源：金属点、地热喷口以及可回收的岩石和树木。

形状定义了单位如何与地图互动，而资源则提供了整体战略目标。精心设计的地形还为玩家提供工具性目标，例如占据一个重要的山顶。纹理纯粹是视觉上的，但对于使地图看起来美观且易于理解至关重要。

Zero-K 地形的一个更引人注目的特征是其坚固性。很少有东西能穿过坚实的岩壁，但在大多数策略游戏中，单位可以轻松地射穿悬崖。但除了物理弹道之外，还有地形的三维形状确实重要的感觉。普通策略游戏中的地形不过是二元掩码，将地图整齐地划分为可行走和不可行走区域。高度感通常仅限于高原层级，以及它们之间的坡道，这些坡道可能会阻挡视线或对向更高层级射击施加惩罚。Zero-K 地形不是分级的，它平滑变化，惩罚源于地形的自然特征，例如山丘不透明且难以攀登。

武器被地形阻挡非常重要，因此许多武器围绕其是否以弧线射击或需要直接瞄准线而设计。激光沿直线传播，因此会被小山丘阻挡，而大山可能会阻挡除最高弹道等离子炮和导弹之外的所有武器。这实际上为我们提供了一个动态且具有方向性的掩护系统，它依赖于理解地图形状，而不是更游戏化的奖励类型。这种掩护形式为短射程军队提供了接近目标的方法，例如，走在沟渠中或紧贴悬崖的背风面。翻越山丘和坡道也极具影响力，因为攀登山丘的单位只有在到达山顶后才能向另一侧射击。然而，山谷下方的所有单位在目标出现在地平线上时就可以立即射击。这在整个地图上制造了不断变化的不对称性，形成了一个复杂的位置优势系统。

山顶的单位相当暴露，但好处是它们可以看到周围的一切。地形阻挡视线和雷达，这使得高处成为侦察兵和雷达塔的理想位置。山脊会产生雷达阴影，允许隐蔽的军队移动，直到它们被翻越。即使能够越过地形射击的单位也需要某种东西来标记另一侧的目标。一个看似空旷的山丘可能实际上隐藏着一个隐形侦察兵，为下方的炮兵提供标记。

山丘也是部署大量单位的好地方，因为一种常见的武器类型——等离子炮——具有弹道轨迹。这使其在向下射击时获得射程奖励，向上射击时受到惩罚。激光则不然，其球形三维范围使其在向上或向下射击时射程减少，而导弹射程不受高度影响。这对于决定在何处使用哪些单位很重要。当向上攻击坡道时，你可能想使用导弹，而向下攻击时最好使用等离子炮，其他条件相同。

单位在攀爬地形时会受到速度惩罚，这使得山丘更具吸引力。这可能是一把双刃剑，因为在向下的突击后撤退上山也更加困难。除了战术用途外，崎岖的地形也会减慢单位在地图上移动的速度。地形还可以通过设置单位无法攀爬的陡峭坡度来形成硬性边界。这取决于单位的坡度容忍度：载具（Vehicles）的坡度容忍度低，机器人（Bots，步行机器人）的坡度容忍度中等，而蜘蛛（Spiders）则能攀爬一切。

坡度容忍度在一个原本平滑的系统中是一个硬阈值，指望玩家处理这个是危险的。我们只使用两个阈值，一个用于机器人，一个用于载具，以保持简单。据我所知，这些值自《横扫千军》（Total Annihilation）以来没有改变。保持地图间的兼容性至关重要，任何更改都会破坏许多地图。地图制作者需要小心避开坡度容忍度阈值，以免我们最终在模糊或不明确的坡道上遇到令人沮丧的寻路问题。这些阈值可能会给高度逼真的地形带来麻烦，因为可行走性可能很难解析。理想情况下，地图纹理帮助玩家理解地形，避免误导他们。许多地图顺利避开了这些陷阱，它们倾向于使用坡度岩石纹理来描绘坡度阈值。

我最喜欢我们地形的一点是它使整个地图都可游玩。每一个小凸起都可能与单位的行为相关，没有纯粹的美学布景远离游戏。例如，崎岖的悬崖比光滑的悬崖有更多的表面积，所以蜘蛛需要更长的时间攀爬。如果地形看起来不同，玩法就不同。这种“纯粹美学”的缺失可能被视为限制了艺术自由，但我认为它将艺术与机制结合在一起，使两者都受益。悬崖的精确崎岖度很少重要，但当它重要时，它强调了地图制作者的风格选择。它使地图感觉像一个独特的景观，而不仅仅是景观的图片。Zero-K 中的山脉不仅仅是单位必须绕过的视觉点缀；当炮弹开始飞行时，它们的形状确实重要。

所有这一切的结果是，两张地图可能有相似的资源布局，完全可供机器人行走，但玩法却完全不同。许多1v1地图符合这种描述，差异来自山丘如何减慢移动、阻挡武器以及视线。山丘和山谷形成的天然战线和据点赋予每张地图独特的风格。我们甚至考虑过基于单一资源布局的地图竞赛，并限制机器人必须能走到所有地方。载具很少能在地图上行驶到所有地方，因为它们的坡度容忍度很低，但这本身是赋予工厂不同角色的重要部分。

研究坚固而细致地形的含义是一个漫长的过程，仅阻挡武器射击就引入了许多破损的交互。不是那种有趣的“破损”，比如把 Outlaw（法外狂徒）放在坑里，而是无趣的漏洞利用，比如使炮塔在某些角度实际上无敌。需要大量关注细节来解决所有边缘情况，我多年没有听说任何问题，所以也许我们已经解决了所有问题。地形改造极大地加速了这个过程，因为拥有自由操控地图能力的玩家远不如原始地图制作者那么仁慈。我们从地形操作中获得了更多收益，但这个话题需要单独一篇文章。

# Cold Take #19 - 游戏马拉松（Game Jams）
日期：2024-10-04

本周的 Cold Take 提前发布、不同寻常且简短，因为 Ludum Dare 在这个周末举行。Ludum Dare 是一个游戏马拉松：一个个人或团队在短时间内制作游戏的活动。它们通常持续几天到一整周，具体取决于马拉松，并且团队会得到一个简短的提示或“主题”来融入游戏中。如今有许多游戏马拉松，我只是因为时间安排而参加 Ludum Dare，并且我喜欢在最后评价其他人的作品。

每个对制作游戏感兴趣的人都应该尝试游戏马拉松。实际活动只需要一个周末，尽管事先投入一些时间寻找合适的游戏引擎大有帮助。入门门槛相当低，我甚至认为游戏马拉松是学习某些编程和设计方面的绝佳方式。马拉松是项目管理的缩影，项目开始时做出的选择往往会在最后给你带来麻烦。当选择和结果之间只有三天时间时，这个过程尤其具有启发性。到最后，你会得到一个游戏。

我特别向任何从事长期游戏（如 Zero-K）的人推荐游戏马拉松，因为快速完成整个过程可以阐明你在长期游戏中的进展。它也让你有机会尝试新想法并发挥创造力，这可能有助于防止功能蔓延。此外，制作游戏是实践的问题，正如围棋大师对游戏设计师所说：“快速制作你的前五十个游戏”。

我通过 Spring Cabal 参与了游戏马拉松，这是一个在 2015 年至 2018 年间多次使用 Spring 引擎参加 Ludum Dare 的 Spring 开发者团体。大多数游戏的发布页面在 LD 网站改版中丢失了，但构建版本仍可在该组织的 GitHub 存储库中找到。Dominic（又名 Shadowfury333）也很好心地在他的 YouTube 频道上记录了大部分内容。

唯一缺少的视频是 LD 42, My Cube，我记得这是一个关于保卫冰山的塔防/单位防御风格游戏。我只参加了其中几个周末，所以只参与了 Hunted, Area 17 和 To The Last Drop。主要组织者是 gajop，随着 Spring 开发者社区的逐渐解体，加上跨时区组织团队的困难，我们更难召集团队了。

我对 To The Last Drop 非常满意。主题是“必须做出牺牲”，所以我们想出了利用由城镇居民生命驱动的魔法来保卫城镇的想法。结果它变成了一个游戏，有开头、中间和结尾，它会杀死闲置玩家，但除此之外允许你尝试不同的方法。To The Last Drop 的成功可能就是为什么我后来与本地认识的人又制作了十款游戏，解决了时区问题，使活动更具社交性。对于这些游戏，我们使用了 2D Lua 游戏引擎 LÖVE，因为我有很多 Lua 经验，而且 2D 美术更容易制作。

你可以在我的游戏页面查看这些游戏。我们倾向于制作介于解谜和沙盒之间的游戏，可以称为问题解决或“策略”游戏。但我们也很小心地给游戏设定目标，并使游戏的基本动作本身就很有趣。你可以在其中许多游戏中看到 Zero-K 的影子，它们关于掌握具有涌现复杂性的简单系统。我认为它们都相当扎实，至少“赛后”版本（Post Jam）是如此（请确保点击正确的下载链接），所以我的推荐取决于你在寻找什么。以下是一些：

# Cold Take #20 - 团队合作是默认设置
日期：2024-10-19

Zero-K 将 RTS 团队游戏的极限推向了新的高度，支持每边最多 16 名玩家——这听起来不可能。你的普通 RTS 上限是 4v4，少数达到 6v6。在 Zero-K 中，玩家经常进行三倍于此规模的游戏，虽然这样的大型游戏并不适合所有人，但我建议仅为了体验而尝试一下。在有如此多玩家、采取如此多行动的游戏中，确实能感受到规模，你只能真正放大并理解相对较小的一部分。但令人惊讶的是这样的游戏竟然能运行，所以在本文中，我们将深入探讨使它们成为可能的设计。

故事中最奇怪的部分是，在大多数情况下，对大型团队游戏的支持是偶然出现的。Zero-K 设计哲学的其余部分恰好自然地允许团队拥有大量玩家。我们偶尔添加针对大型游戏的功能，但大部分工作是由以前 Cold Take 中涵盖的原则完成的。我们本可以刻意移除大型团队，但这种尝试会扭曲其余设计，而且它们并不伤害其他游戏模式。小型游戏，低至 1v1 和 2v2，运行良好，在我看来，支持更多游戏模式是好事。

大部分工作由“与对手战斗，而非与界面战斗”完成，加上玩家通过用户界面与团队协调的认识。本质上，团队内的合作应该没有人为限制或繁琐步骤，就像游戏中的任何其他行动一样。Zero-K 包含简单的合作 UI 示例，例如告诉玩家盟友核弹预计落点的核弹指示器，但“不与界面战斗”比这更深入。

Zero-K 是一个将单位愚蠢性设计掉、并使用无法企及的理想设计武器的游戏，我们对合作也采用同样的方法。像单位一样，玩家有影响游戏的能力，如果游戏机制迫使玩家走上无聊且费力使用 UI 的道路，我们就认为这是失败的。所以虽然大多数游戏会欣然添加核弹指示器，但很少有游戏会像我们这样深入地消除根本上反合作的机制。

我们从《横扫千军》（Total Annihilation）继承了关键的玩家能力，即自由共享单位和资源的能力。这是 RTS 游戏相当标准的功能，但其影响深远，Zero-K 倾向于将想法推向极端并承受由此产生的后果。共享的问题在于共享 UI 通常是 RTS 中最糟糕的部分之一。它们往往像大型电子表格，充满了琐碎的按钮或滑块，这不是我们想要的。因此，应尽量减少与此 UI 的接触，这使共享成为一种几乎“理论上的”能力，游戏围绕它设计，但玩家应该很少手动调用。这推广为所有权无关性原则：

游戏世界机制中的任何内容都不允许依赖团队内的所有权细节，因为那样玩家会受到激励去触碰单位共享 UI 以获取优势。
很少有游戏使用这个原则，因为它被特定玩家的升级自动违反了。即使在 TA 血统中，Supreme Commander（最高指挥官）允许同一玩家的单位互相射击，但不允许穿过盟友单位。甚至 Zero-K 也违反了这个原则，因为未识别的雷达点显示团队颜色，这可能让敌人猜测未看见单位的身份。我一直希望能关闭这个功能，但它需要引擎工作，并且为了稍微回退这种极端主义，点颜色可能确实改善了游戏。

所有权无关性原则听起来限制性很强，但也很解放，而且无论如何我们都不打算添加全局升级。该原则的一个关键含义是资源收入可以在团队内任意分配，对游戏世界内的平衡没有任何理论影响。实践中，这让我们能够调整收入共享以“模拟”互联网上陌生人之间的合作，这是允许如此大型团队游戏运作的部分原因。例如，金属提取器的收入在团队所有成员之间平均分配，这在早期的《完全歼灭》（Complete Annihilation）中被称为共产主义模式。共产主义与游戏其余部分一起发展，并很快与超载（Overdrive）集成，因为超载公式从共享资源中受益。

共产主义的理论依据是确保领土是共同努力，远比仅仅建造一个金属提取器更困难，因此团队共享收益。但当时的实践依据更重要，因为我们都有在《平衡歼灭》（Balanced Annihilation）中人们争夺金属点的经验，这可能升级为团队伤害。在 BA 中的次要影响是地图池有限；玩家倾向于玩一小部分地图，其中每个人都理解金属点所有权的元游戏（meta），部分是为了减少潜在的资源争论。

值得一提的是，我偷偷在共产主义中插入了一个假设：玩家想要合作。如果人们想保留自己的收入怎么办？这触及了设计的核心：我们假设团队想要合作，团队合作是默认设置。如果玩家想要争吵并摧毁彼此的建筑，那么他们可能在 UI 中找不到对这些行动的支持。这里也有一点悖论，任何玩过不同技能水平团队游戏的人都可以告诉你：谁拥有什么实际上非常重要。但我们没有试图评估每个玩家的价值，而是选择制作一款游戏，其中技能的一部分在于成为一个好的队友，而不是在建立个人帝国时碾压团队中的较弱成员。

共产主义解决了金属点争论并扩大了地图池，但它遭受了搭便车问题。一些玩家会完全跳过建造金属提取器，希望他们的队友来补足。完全的共产主义也不利于激励超载，即使它对团队有显著的好处。所以我们添加了回扣。例如，金属提取器在几分钟内向建造者退还其成本的 80%，以略微增加收入份额的形式。能源结构提供类似的 60% 退款，来自其增加的金属超载收入。多年来，这些数字经过了调整，因为更高的回报百分比可能导致能源刷子被视为自私。我们甚至最近为团队建造的前几个反核弹（Antinukes）引入了回报，作为一种社会方式来削弱低合作情况下的核弹。

共享金属提取器也帮助解决了由一些旋转对称地图引起的“Tabula 问题”。Tabula 的南北方向有高地，分别更容易从西北和东南方向占领和防御。在共产主义之前，Tabula 通常由从强大高地入侵弱低地的团队赢得，然后他们的对手在地图的另一侧做同样的事情。低地只是缺乏进行防御所需的金属收入，除非玩家手动共享单位或金属提取器。在共产主义之后，Tabula 的玩法仍然有点类似，低地地形仍然不佳，但额外的收入使防御更公平、更有趣。

仅靠收入共享不足以使大型游戏有趣。为此，我们需要 Zero-K 的其余部分，主要是其单位和阵营设计。我们大量的类阵营工厂让玩家能在团队内开辟独特的角色和单位集，工厂放置（Factory Plop）让每个人都能建造工厂而不“浪费”资源。最近添加的工厂板（Factory Plates）曾被担心会破坏这种在团队中拥有角色的感觉，但这种担忧并未成为现实。如果有的话，它们似乎通过让玩家承担一个他们不确定的角色来改善了团队游戏，并且可以低成本切换出来。

最接近破坏游戏的协作功能是共享控制模式（Shared Control），玩家拥有并控制同一组单位。战役有共享控制，但也可以在团队游戏中通过邀请系统启用。共享单位控制的危险在于它可能很强大，但当别人不小心使用“你的”单位时也很烦人。这就是可怕的权衡，以与界面战斗为代价获得力量，但这种模式在公共游戏中很少使用。锦标赛和战役游戏不同，因为人们可以充分沟通以避免缺点，在这些情况下它是一个很好的功能。

与许多自动化功能一样，值得问一下合作是否已被自动化消除。我认为我们在这一点上做得很好；队友发送消息、侧翼攻击并以其他方式互相帮助。人们仍然偶尔共享一个单位来加强真正需要的前线。我们主要移除了多余的 UI 交互，并设定了合作的基准水平，这样玩家可以放心扩张而不必担心“偷取”金属点。也许最大的好处是对地图池的，因为任何地图都可以与任何数量的玩家一起玩，提供真正广泛的游戏体验。

这标志着第一个 Cold Take 系列的结束。二十是个不错的整数，是时候短暂休息了。期待下一个系列在明年一月开始。

# Cold Take #21 - 看见你能看见的
日期：2025-01-25

欢迎回到 2025 年的 Cold Take。今年的目标是每三周发布一篇，但不与更新说明争抢空间。

几乎每个实时策略游戏都有战争迷雾（Fog of War）。没有它，你可以监视敌人基地并开始反击他们的任何行动。这种动态可以奏效——有完整的类型建立在完全信息之上——但它似乎不适合那些需要时间产生回报的渐进式实时决策游戏。但在深入探讨视野的作用之前，值得看看它是如何运作的。

战争迷雾可以非常简单。你只需要在每个单位和建筑周围画一个圆圈，然后宣布圆圈内的一切都是可见的。许多游戏到此为止就停止了，但我怀疑大多数会采取额外步骤考虑梯田和树木。梯田视野系统有明确定义的地形层级，较低层级的单位无法看到较高层级，游戏通常用不透明的静态特征（如树木和建筑物）阻挡视线。这种简单性效果很好，直到模拟主义者介入。

《横扫千军》血统的游戏比你的《星际争霸》（StarCraft）或《命令与征服》（Command and Conquer）更模拟主义。也就是说，它们更关心匹配现实的机制，或者至少看起来匹配现实。Zero-K 经常淡化这一点，但它仍然有点过于模拟主义，不适合使用梯田和树木视野系统。一方面，我们太关心地形，以至于无法通过一系列带有坡道的高原来近似山丘和山脉的表现力。这使得模拟主义路线很有吸引力，但在没有其他设计目标明确指导的情况下，很难知道在哪里停止。

在考虑这些目标之前，考虑完全逼真的视野是什么样子很有启发性。这样的系统将涉及在每个单位的“眼睛”和单位可能想看的一切之间画直线。如果直线有清晰无阻碍的路径，那么单位就能看到目标，我们甚至可以画线到单位的边缘，当它们从角落探出时看到它们。这是一种称为光线追踪（Ray Tracing）的通用技术，它反映了光在现实世界中的行为。但这样的系统会对 Zero-K 有用吗，其余设计对此有何看法？

为了直奔主题，我们将诉诸“与对手战斗，而非与界面战斗”。简而言之，界面是你的朋友，它不应该妨碍你，它应该告诉你关于你部队需要知道的一切。这包括关于你的部队能够或不能看到什么的信息。履行这一承诺的接口必须清晰简单，这对逼真的视野来说是个问题。问题的关键在于你需要知道你不知道什么。

任何旧的视野可视化都可以告诉你你知道你知道什么。如果你能看到一个巨型机器人从悬崖上探出头，那么 UI 只需要绘制这个巨型机器人。但如何排除一个偷偷摸摸的中型机器人的存在？UI 必须告诉你如果它在那里，你是否能看到它。标准解决方案——诡异的黑色迷雾——是未知的绝佳表现，但如果你试图用它做太多事情，它就会失效。迷雾本质上是模糊的，所以它如何告诉你你能发现一个巨型机器人，但不能发现中型机器人？不透明度的变化很难精确识别，更清晰的线索如轮廓或颜色编码会相当杂乱。所以 Zero-K 直接避免了单位可见性依赖于其大小。

这是许多游戏使用的视线技巧：单位实际上从不互相看到，它们只看到地形的一部分。视线着色器（Shader）模糊了这个事实，但视野基于网格工作，一个方格的可见性由追踪到每个方格中心的射线决定。射线从每个单位的“眼睛”开始，因此高大的单位更善于越过地形看到东西，而射线末端悬停在地形（或海洋表面）上方固定距离处。单位如果位于可见地形上就可见，这使得敌方单位的潜在可见性非常容易告诉玩家。

Zero-K 的视野与简单的梯田和树木系统相比仍然有点笨拙。平滑的地形使得难以精确预测你自己的单位到达给定位置时能看到什么，但这比不知道它到达后能看到什么要好得多。为了进一步改善这一点，我们调整了悬停射线追踪目标的高度，使其相当宽容，尽管理论上它们代表“平均”单位的高度。

所以你明白了，Zero-K 中的视野检测基于一个“平均”高度的单位是否可见，因为这样做允许简单的二元战争迷雾告诉玩家他们需要知道的一切。只不过，这只对地面单位是正确的。是时候谈谈例外了。

弹道、爆炸和未降落的飞机忽略通常的视野规则，而是使用称为“空中视线”（Air Line of Sight）的圆形视野系统。这是一个并行的视野系统，但我们确保所有东西的空中视线范围与其标准视线范围匹配。这个系统是对现实主义的致敬，因为它让你能看到飞机而看不到其下方的地形。在视觉方面，当弹道在峡谷上射击时不会突然从视野中消失是很好的。为什么不为特定的巨型机器人添加空中视线？一致性和清晰的阈值；飞机显然是飞机，但巨型和非巨型机器人之间没有明确的界限。

只不过，这也不完全正确，因为巨型机器人通常可以被空中视线看到。事实上，在最后几秒钟内开过火的任何东西都对空中视线可见。这是合理的，因为枪口火焰很难隐藏，但它也解决了一些边缘情况。没有这种机制，你可以将炮塔放置在高悬崖后面，使它们难以被瞄准，但仍能射击接近的单位。作为额外的好处，在悬崖边缘发现你基地的巨型机器人可能会开始向你射击，所以这几乎就像它一开始对空中视线可见一样。

有时我想知道有多少人知道空中视线。它是那些在后台平滑一切的机制，使游戏看起来不错并合理。不知道它似乎没问题，并且使其更可见是有代价的。我从经验中知道，改变迷雾的不透明度效果不佳，因为空中视线过去被绘制成较浅的迷雾形式。引擎默认的空中视线是视野范围的 150%，但我们将其覆盖以匹配视野范围，这样我们就可以避免绘制它（见上文）。我为这些图像选择了一张对比度良好的地图，但一般来说，很难在所有地图上使半透明战争迷雾足够可见。

Zero-K 也有雷达和声纳，但对于理解视线的人来说并不新鲜。雷达只是视野的一种形式，射线追踪到悬停在地形上方的相同点，而声纳是一个简单的圆形系统。看来引擎开发者不想实现完整的声波传播和检测系统。这就是视野。单位为什么想互相看到必须等待，但至少现在有了如何做到的记录。

# Cold Take #22 - 雷达与视野的极限
日期：2025-02-16

策略游戏有时提供检测敌人的额外方式，超越标准的视线，雷达是 Zero-K 中的一个经典例子。雷达揭示了玩家互动的很多方面，但我们还需要阐述视野本身，因为这两个系统相互补充。两个系统使用相同的底层机制，上次已经介绍过，尽管细节对于理解机制的使用并非严格必要。

雷达很棒，因为它在设计中实现了很多功能，但在机制上只是视野的变体。在 Zero-K 中，雷达与视野在一方面不同：它只揭示单位的大致位置，其他什么都没有。单位类型甚至未被揭示，尽管对于之前侦察过的“雷达点”（Radar Dots），信息会被记住（以避免与界面战斗）。就机制的使用而言，两者截然不同：所有东西都有视野范围，而只有少数专业单位和建筑有雷达。雷达还能“看”得更远，大多数雷达范围大约是平均视野范围的五倍。

雷达的主要目的是允许在战斗之外进行直接互动。我所说的直接互动是指具有紧密反馈循环的互动，你和对手可以实时看到并相互回应。仅视野将玩家限制在相对较短的互动范围内，这些互动可能会演变成战斗。各种形式的专业侦察可以允许更多的战斗外互动，但这可能使信息过于廉价。雷达提供了一个中间地带，你可以在不被信息淹没或降低战争迷雾价值的情况下，对大规模军队移动做出反应。当然，如果你知道敌人在哪里，那么你应该能够射击他们（再次，以避免与界面战斗），但为了扩展其含义，我们需要绕道讨论视野。

视野的细节归结为以下问题：单位应该能看到多远？我们的答案在大多数情况下是“只比他们能射击的距离远一点”。这是对《横扫千军》启发的许多其他游戏中情况的刻意逆转，但想法是与可见的单位战斗比与雷达点战斗更令人满意。更充分的理由是，对敌人的视野在战术上很强大，侦察单位大多将这种视野置于繁琐操作之后，而非任何重要的战术或战略决策。这源于侦察兵廉价、能看到比平均战斗单位更远，并且速度快到死亡成为与界面战斗的问题。将侦察兵混入军队通常只是最优选择，因为它让所有单位都能看到目标，但将这项工作交给侦察兵大多只是给了他们一种额外的愚蠢方式。所以为了跳过整个繁琐过程，我们直接让单位看到它们能射击的距离。

上述的一个显著例外是炮兵。为炮兵侦察目标是令人满意的，因为侦察兵必须被派往危险的地方。为了支持这种动态，炮兵不能看到它们能射击的距离。事实上，我们有一个通用规则，单位能比它们的射击距离远10%，最大到660范围，这大约是重型激光炮塔（Heavy Laser Turret）的范围。然而，这个规则有许多例外。有些是技术必需的，例如视野网格的粒度要求特别短射程的单位看得更远一些。其他例外是刻意的设计选择，例如雷达塔（Radar Tower）、Owl（猫头鹰）和 Toad（蟾蜍）独特的长视野范围。其他的，嗯，无武装建筑的视野范围是273，原因已随时间流逝而失传，而无武装建造者的视野范围似乎基本上是随机的。

到目前为止，炮兵似乎有两个冲突的约束：我们想鼓励侦察，而单位应该能够射击雷达点。解决方案在几段前已经揭示：让雷达只揭示单位的大致位置。这是通过“雷达晃动”（Radar Wobble）实现的——雷达点相对于单位真实位置的漂移随机偏移。晃动阻碍了对点的随意射击，因为它们很可能未命中，所以侦察兵必须冒险外出以获得视野。雷达晃动带来了自己的问题，我们尝试通过各种 UI 功能来缓解。主要的是，对于已侦察的建筑，雷达晃动被移除，因为建筑无法移动。还有一个状态切换，告诉特别精确的炮兵避免射击晃动的点，尽管默认未启用，原因需要一整篇额外的文章来解释。

饱和炮兵怎么办？一些炮兵单位太不准确，以至于不关心雷达晃动。在这种情况下，射击雷达点是一个功能。敏锐的读者可能回忆起在隐形设计中出现的一个原则，即花所有时间命令单位射击你知道存在但看不见的敌人是令人沮丧的。同样的问题出现在区域封锁炮兵中，并且主要通过射击雷达点来解决，因为这样做模拟了玩家对视野外可能目标的猜测。没有雷达或雷达被干扰的情况可以反制这种能力，但这是可以接受的，因为目标是让单位在大多数情况下自行处理，足以减轻总是手动处理饱和炮兵瞄准的重复性。

到目前为止，很明显雷达是 Zero-K 的核心，我们通过使雷达非常便宜来进一步鼓励这一点。不仅指挥官（Commanders）有免费的车载雷达，而且基础雷达塔（Radar Tower）也几乎免费。这是一个根据单位的常见程度而非绝对效用来平衡的绝佳例子。雷达塔还身兼炮兵侦察兵的职责，具有异常大的视野范围，这激励玩家将脆弱的建筑放置在离谱的位置，这成为很好的目标练习。高级雷达塔（Advanced Radar Tower）甚至有更多视野，并且雷达范围是250%，但成本相当高。这使得高级雷达不那么常见，但其效用在于观察敌方基地，这是我们不太感兴趣鼓励的。高级雷达过去还减少雷达晃动，但这降低了为炮兵侦察的重要性，所以该能力被移除了。

雷达最不协调的部分是雷达点有团队颜色。这向敌人泄露了信息，而团队游戏依赖于团队内所有权没有机械重要性的理念。这意味着，如果你知道橙色对手有重型坦克，那么一个缓慢的橙色点更可能是一辆坦克。Supreme Commander（最高指挥官）用灰色的未识别雷达点解决这个问题，Zero-K 也讨论过添加这样的功能，但我逐渐倾向于认为团队颜色的雷达点可能改善了游戏。至少在休闲团队游戏中。

团队颜色的雷达点高效地告诉你团队每个前线有多少对手存在，并且看到你在与谁对抗增加了社区感。它有助于区分敌方空中玩家的飞机，尽管在这种情况下，点的垂直位置通常就足够了。反对有色雷达点的原则立场也会主张完全移除敌方团队颜色，将敌方团队变成匿名群体，这在某种程度上社交的游戏中肯定感觉不好。无论如何，我想尝试真正的未识别雷达点，但我担心效果可能太广泛，在短期测试中无法衡量。无论如何，这种测试必须推迟到引擎支持这种功能。

# Cold Take #23 - 反对太空
日期：2025-03-08

Zero-K 技术上有“太空”战斗。对于任何单位可能最终飞得很高，并且飞得很高有任何机制意义的游戏来说，这都是正确的。在我们的例子中，极端高度使单位对来自下方的武器射击免疫。即使是具有圆柱范围的导弹也有燃料限制，给予它们最大射程。因此，太空旅行的进步往往被修补掉，这值得注意，因为我们通常倾向于保留创造性的特殊战术在游戏中的。但随着时间的推移，“Zero-K 无法进入太空”的规则出现，因为太空旅行太缺乏互动性。将太空与地面一起查看也会导致界面问题。

《完全歼灭》（Complete Annihilation）见证了我们唯一一次完全有意的太空尝试。这是最初的 Owl（猫头鹰），一个间谍卫星（可能），由 CarRepairer 添加。
这个隐身球可以作为你的炮兵侦察兵或监视对手基地。它消耗大量能量停留在空中，但对几乎所有单位都是安全的。易受高级战斗机和远程导弹塔攻击。它飞得比最高飞机（Nimbus，雨云）高5倍，移动速度比最慢地面单位（Jugglenaut，杂耍者）还慢，并且视野范围只有微小的300 elmo。它在核心炮艇工厂（Core Gunship Plant）花费300金属，并且如描述所述，只能被最重型的地面防空和空中优势战斗机击落。换句话说，这个单位只与 Artemis（阿耳忒弥斯）和基于导弹的 Raptor（猛禽）等效物互动。

Owl 没有持续下去，并成为了“Owl 某物”一词的起源。它被削弱到无用，因为主要的平衡设计师不忍心移除它，但能看到如果它变得普遍会多么麻烦。如果敌人有飞机工厂（Plane Factory），那么它就死了，但如果没有飞机，它需要非常昂贵的防空才能驱逐。硬反制太难太贵，而没有任何软反制。所以它被软性移除，没有在过渡到 Zero-K 时存活下来。

Owl 的反响为后续的太空探索设定了模式，缺乏直接支持激发了创造力。人们将不得不重新利用单位来实现他们的太空梦想，主要罪魁祸首是 Newton（牛顿）、Lobster（龙虾）和 Placeholder（占位符）的重力诡计。每个这样的单位都已被修补以限制太空旅行，尽管 Newton 作为三者中最古老的受到了最多的关注。

Newton 是一个使用引力枪吸引或排斥目标的炮塔。只要有一些理由，单位可以瞄准并射击任何东西，因此 Newton 可以瞄准盟友单位。人们很快意识到 Newton 可以将单位射过地图，用地形改造的坡道来帮助瞄准和调整角度。这些牛顿大炮（Newton Cannons）是“太空”旅行的第一种形式，但它们仍然存在至今，这似乎与上述禁令矛盾。是什么让 Newton 的当前用途“合法”，还有什么必须被修补掉？

太空在主导游戏的大部分部分且几乎没有反制手段时才是个问题。牛顿大炮有时赢得游戏，但它们很少是廉价、不可阻挡、来自上方的死亡来源。大多数牛顿大炮发射跳跃喷气单位（Jumpjet Units），但这些单位在飘落时可以被看到，并在空中被击落或拦截。这是一个不错的备用策略，但可以反制，所需投资不会比其成本高太多。同样重要的是，单位几乎不去太空，即它们可以在舒适的缩放级别看到。最糟糕的太空旅行情况是单位飞得如此之高，以至于地图只是远处的一个小方块。

第一个主导的牛顿大炮策略是由于 Skuttle（潜行者），这是一个麻烦的单位，可以隐形、跳跃并造成巨大伤害爆炸。第一个问题（去年提到过）是它过去能够从太空跳跃，然后隐形飘落到敌人基地。唯一理论上的反制是在所有重要东西上方悬停炮艇，并在附近驻守压倒性火力以在短暂的近身解除隐形窗口内杀死 Skuttle。所以我们发明了跳跃腿（Jumplegs）来阻止 Skuttle 在空中跳跃。但这并没有阻止人们从高空降下隐形死亡。

Skuttle 的下一个问题被证明是炮艇的更广泛问题。炮艇可以被 Newton 推动，但不会坠毁，这使其成为安全太空旅行的理想选择。一艘载有 Skuttle 的运输机可以被高高射入天空，超出防空范围，然后飞过敌人基地。然后 Skuttle 可以被投掷到下方的重要目标上，在下落过程中隐形。这引发了几个变化。第一个是对引力枪的普遍削弱，只有其力的25%垂直分量应用于炮艇。第二个是针对运输机的定向削弱，即它们向下加速到期望高度更快（在任何其他情况下都是增益），并且在投放单位时给予单位一个小随机推力，以消除轨道投放的精确性。

炮艇没有 Skuttle 的帮助也可能造成问题。在高空炮艇被阻止后，人们开始尝试水平“太空”。牛顿大炮可以被设计为将炮艇侧向射出地图，超出地面防空范围。这让运输机和 Krows（鸦）可以潜入敌人基地后方，而无需穿透前线防御。这种战术比轨道 Skuttle 有更多的反制手段，但到此时，界外战术已被确立为通常保护创造力的例外。因此我们通过将飞机推回地图来阻止炮艇弹弓，缓冲区为400 elmo，这大约是最笨重飞机的转弯半径。飞机也可以被射出地图，缓冲区甚至不得不减半以适应最近添加的 Odin（奥丁）。

另一个边缘但最终被移除的战术是跳蚤发射器（Flea Launcher）。跳蚤（Fleas）廉价且非常轻，这使得使用基础牛顿大炮发射它们成本低廉。这里的问题是视野范围默认是圆柱形的，即二维的，这使得跳蚤发射器主导了其他后期侦察选项。我们对其他更细致的侦察形式的传播和效果感到满意，所以跳蚤发射器被削弱了。削弱包括动态减少发射单位的视野范围，以近似三维视野范围的形式（实际形状是一个高度为视野范围两倍的锥体）。玩家仍然可以自由地通过以快速、低轨迹射击跳蚤来侦察，但这需要更昂贵的牛顿大炮。

Lobster（龙虾）和 Placeholder（占位符）呢？这两个单位过去都能够创建自我维持的漂浮空间站，任意高度在空中。Lobster 通过连续射击足够大的 Lobster 群来实现这一点，因为它过去能够在空中射击时开火。Placeholder 可以瞄准盟友，并倾向于将其目标略微抬离地面。足够多的 Placeholder 可以形成一个自我维持的群体，逐渐向上漂移。这里的主要问题是消失在天空中使得失败几乎不可能，Zero-K 以拥有比国际象棋更好的平局情况而自豪。天空基地也是给予弹道武器高度优势的好地方，尽管这是一个很酷的能力，只要它变得合理。

太空龙虾升降机（Space Lobstervator）和占位站（Placestation）在同一天被移除，就在 Lobster 的避免失败能力被展示之后。两种情况下的修复都是使 Placeholder 和 Lobster 的能力随高度衰减。Lobster 不能将任何东西提升超过目标地形高度400 elmo，并且自那以后失去了在空中射击的能力。一个 Placeholder 射击在地面以上1200 elmo 处爆炸没有效果。这恰好是旧 Owl 的高度，尽管降到1200意味着实际上物体会在1000 elmo 左右漂浮。所以单位仍然可以被维持在相当高的空中，只是不能任意高。

最后一个也是最愚蠢的进入太空的方式是由于 Dirtbag（土袋）。想法如下：Dirtbag 在死亡时会掉落一点地形，那么为什么不堆叠500个左右在同一个位置然后引爆它们？结果是一个非常细、非常高的尖刺。当人们想出如何将 Crabs（螃蟹）故障地弄到这个尖刺上时，这成了一个问题，由于其弹道轨迹，它们可以在荒谬的距离射击。这是一种为 Crabs 创建即时尖刺的巧妙创意方式，但尖刺可以如此之高，以至于地图的其余部分变成几个遥远的像素。我们通过限制 Dirtbag 创建地形的陡度来修补了这个问题。这样的尖刺现在被限制在800 elmo，这仍然显著，但不会将 Crabs 变成全地图炮兵。

这就是进入太空的方式，除了简单的漏洞，例如一个允许玩家储存冲量并一次性释放的漏洞。这也是关于已被修补掉的创造性战术的全部，因为总的来说我们尝试保留它们，牛顿大炮就是一个主要例子。太空是主要的例外，因为它与使 Zero-K 伟大的地形和互动相去甚远。地下单位也可以这样说，但由于那只是一种鸡的能力，我不指望很快会有关于移动挖掘的文章。

# 冷评 #24 - 平衡爆发伤害阈值
日期：2025-03-29

轻型单位在吸收大量爆发伤害方面表现出色。一次爆发——即单次高伤害射击——来自 Heavy Laser Turret（重型激光炮塔）可以瞬间蒸发一只 Flea（跳蚤），但至少它只杀死一只。Recluse（隐蛛）也会被一击毙命，而它的造价是 Flea 的十倍，因此 Flea 在吸收（或者说浪费）爆发伤害方面相对高效。Zero-K 中广泛使用高爆发和伤害浪费机制，部分是为了弥补我们取消伤害倍率政策留下的空白，但主要是为了满足对大型高冲击力武器的追求。

爆发伤害并非新鲜事物，因为装填时间是即时战略游戏的基础要素，大多数游戏至少都有一定程度的变化。Zero-K 的讨论中也用"alpha"作为爆发的同义词，快速搜索表明这可能是早期从 EVE Online 交叉传播而来的。在平衡层面，爆发伤害与持续伤害（DPS）之间存在权衡，因为爆发往往使单位更强大，但要注意其浪费伤害造成过度击杀的缺陷。爆发让单位能在战斗初期集中输出伤害，因为它们在战斗开始时状态最佳，这提高了它们的有效输出并限制了敌方的反击。从游戏体验的角度看，大型单位一击消灭小型单位强化了规模感，而爆发伤害的可能性为战斗增添了多样性和刺激感。

Zero-K 喜欢探索设计空间的极端，因此我们最终拥有了一些爆发力相当强的单位。Lance（长枪）每 23 秒开火一次，远超其前身在 Total Annihilation（横扫千军）中 6 秒的射速。如此长的装填时间放在其他游戏中完全可以是一个手动技能，而 Zero-K 玩家也经常将这些武器当作手动技能使用。方法是切换单位的停火状态，这样玩家就能在需要时直接控制，而其余时间无需与界面作斗争。游戏中存在各种装填时间的单位，甚至连装填时间低至 10 秒的单位也经常被设为停火状态以狙击指挥官。

如前所述，爆发伤害可以用来特化武器以对付重型单位。突击单位围攻 Heavy Laser Turret 是经典案例，但当重型坦克或火炮的炮弹造成远超必要的伤害时，这种情况同样适用。相反，镇压单位应具有较短的装填时间，以应对蜂拥而至的单位群。突击单位对突击单位是一个有趣的案例，因为一定程度的爆发是相当大的优势，但这种优势因对局组合不同而差异很大。高爆发的轻型突击单位——Duck（鸭子）和 Dagger（匕首）——特别难以平衡，至今尚未找到真正合适的位置。问题甚至不是由于纯粹的过度击杀，而是更微妙的阈值问题。

即使爆发伤害不能一击消灭敌人，它仍然很重要。以 Dagger 为例，这是一款轻型突击单位，每 2.83 秒造成 110 点伤害。轻型突击单位的血量在 200 到 500 之间，因此 Dagger 的精确伤害量与特定突击单位的血量相比可能相当重要。例如，Bandit（匪徒）有 340 血量，Glaive（长柄刀）有 230 血量，分别让 Dagger 需要四次和三次射击来击杀它们。如果出于某种原因需要专门削弱 Bandit 对抗 Dagger 的能力，我们可以将 Bandit 的血量降至 330。这几乎不会影响它的其他对局表现，但会有效降低它对抗 Dagger 时的"血量"达 25%。或者，我们可以将 Dagger 的伤害提高到 114。当前版本的 Dagger 伤害最低为 95，最高不超过 110，因为我们不想越过 Bandit 的爆发阈值。

有些武器如此强大、如此普遍，以至于它们扭曲了游戏其余部分的设计。最典型的例子是 Raven（渡鸦），一款携带 800 伤害炸弹的轰炸机。轰炸机每次出击只射击一次就返回基地装填，因此它们的爆发伤害至关重要。Raven 在十多年里一直是主力轰炸机，单位的存亡取决于它们与 Raven 的互动。最轻的镇压单位——Reaver（掠夺者）——拥有 820 血量，因为被单架 Raven 击杀会让它的工厂过于脆弱，无法承受突击单位的后续进攻。Mace（钉锤）的造价几乎是 Reaver 的两倍，却一直饱受只有 1400 血量的困扰，而 Light Laser Turret（轻型激光炮塔）拥有 785 血量以让 Raven 能将其清除。Recon Commander Chassis（侦察型指挥官底盘）有 3250 基础血量，因为指挥官的平衡是围绕轰炸成本来设计的。工厂拥有 4000 血量，因为假定附近的建造者可以在轰炸过程中修复它们，使其恰好能承受五架 Raven 的攻击。

其他爆发伤害来源也很重要，但影响力不如 Raven。Lance 用其重型激光造成 3000 伤害，进一步防止指挥官血量降至 3000 以下。同样，Phantom（幽灵），一款隐形狙击手，造成 1500 伤害。Phantom 展示了围绕阈值平衡如何变得有点荒谬：Felon（重罪犯）被加强到 1600 血量，专门为了不被 Phantom 一击毙命。Felon 是 Shieldbot（护盾机器人）的标志性单位，造价几乎与 Phantom 相当，而 Phantom 是 Cloakbot（隐形机器人）的标志性单位，因此一击丧失它实在太过惩罚性。将 Felon 的血量调校到 Phantom 的爆发阈值让我们做出的改动几乎不影响其他对局，因为对大多数其他单位来说多造成 100 点伤害并不太困难。所以爆发是精确平衡的好工具，但过度使用会让互动感觉像在玩游戏机制，不够"自然"。

想象一个拥有 1501 血量的 Felon。从机械层面看，这样的单位没问题，但每当它以一点血量幸存于 Phantom 的射击时，看起来就很怪异。禁止 Felon 拥有 1501 血量类似于《瞄准与射击》中的真实感原则，即一个"真正的"Phantom 肯定可以"超频"其步枪来多挤出 0.1% 的伤害。正是出于这个原因，我们坚持使用相对圆整的血量和伤害数值。一个 1600 血量的 Felon 看起来更自然，即使 1501 血量能实现相同的平衡目标。

伤害与血量之间的接近程度存在一个限制，这个限制似乎是 20。以 Reaver 的 820 血量对抗 Raven 为例。另外两个案例明显体现在精准轻型缠斗单位上：Moderator（裁判）和 Scalpel（手术刀）。这两个单位分别造成 460 和 660 的爆发伤害，且各自的血量都比爆发伤害高 20。我们很久以前就决定，让这类单位互相一击毙命太过波动性，而让单位对自己特别弱势会鼓励更有趣的军队组合。Lance 没有得到同样的待遇，因为视野不如射程远已经创造了足够的反制手段，而且火炮本应特别脆弱。

Revenant 对抗 Engineer Commander（工程师指挥官）是最近一个通过爆发阈值平衡的典型案例。Revenant 是一款耐久型武装直升机，发射一组中等伤害的导弹，之前有点弱势，所以我们给它增加了一枚导弹。这将其伤害提高到 1980（9 枚导弹，每枚 220 伤害），使它能两击击杀未升级的 3800 血量 Engineer。这在此后数月都未被注意，直到 Engineer 获得了 blueprint（蓝图）能力，增加了它在前线的使用率，这反过来提高了 rush 一架 Revenant 来狙击指挥官的可行性。随后的削弱包括将导弹伤害降至 210，总爆发为 1890，使 Revenant 距离两击击杀 Engineer 还差 20 点伤害。

爆发为血量和伤害创造了阈值效应，但它也为时间创造了阈值。这在防空和轰炸机之间体现得最为明显，因为轰炸机只会在射程内停留几秒钟。结果是防空武器的装填时间与飞机在射程内停留时间之间的敏感关系。一把武器在纸面上可能很强，但如果目标往往在它装填之前离开射程，实战中就会令人失望。爆发也有助于将防空特化为反飞机和反武装直升机。反飞机武器往往是高爆发导弹，用高 alpha 伤害在早期输出伤害，而反武装直升机武器则偏向高每秒伤害，因为武装直升机会停留输出伤害，并且很可能在爆发型防空武器发出第二或第三次射击前将其击杀。

通过爆发阈值进行平衡存在将游戏充满晦涩且不直观互动的风险，就像在其他游戏中过度依赖护甲类别一样。没有任何玩家能在不经过任何经验或研究的情况下知道 Revenant 只能三击击杀 Engineer Commander。重复这个过程——大约 25 个装填时间超过五秒的单位——就会产生潜在的记忆问题。其中许多单位造成不精确的范围伤害，这模糊了阈值，而另一些单位的爆发如此之大，以至于在大多数情况下它们的伤害可以视为无限。对于其余单位，界面中有一部分会显示所选单位的综合爆发伤害。这也帮助了那些碰巧记住了单个爆发伤害的玩家。

即使名义上具有精确爆发伤害的武器，在实践中也有点模糊。炮弹可能会未命中，burst laser（爆发激光）在构成一秒的 30 个游戏帧内持续造成伤害。这给 Flea 与 Heavy Laser Turret 之间的关系（以及 Lance 与 Grizzly 之间的关系）加了一个星号，因为几个紧密排列的单位可能都被同一激光击杀。这种反蜂群能力对 Quant's Rule（量子规则）来说并不理想，但物理化的死亡光束酷得多，而且让玩家能玩出诸如将两门轻型炮塔对齐以一击双杀之类的技巧。让激光造成单次伤害实例会更容易，也更"平衡"，但归根结底，体验感胜过平衡性。

# 冷评 #25 - 完全免费
日期：2025-04-19

Zero-K 比一般的免费游戏更加自由。即便是没有任何内容被锁在付费或肝度背后的事实都让人惊讶，但它的自由远不止于此。Zero-K 是自由软件，任何人都可以根据自己的需要修改，前提是也必须自由地分享修改。这可能更让人惊讶。制作一款功能完备的即时战略游戏肯定花了不少钱，那为什么它完全免费——谁付了钱？简而言之，没有人因为任何事获得报酬，因为 Zero-K 完全是由志愿者利用业余时间构建的。它之所以能运转，正是因为它免费。

首先，一些免责声明，因为上述说法在几个方面技术上不太准确。捐款涵盖了服务器运营费用，这些捐款以指挥官皮肤和论坛徽章作为回报。这意味着有东西可以买，尽管完全不像完整的装饰品商店那样规模。我还有一个每年几百美元的 Patreon，如果你要非常挑剔的话，战役中的连续任务在技术上算是一种锁定内容。

Zero-K 继承了底层引擎 Spring Engine（Spring 引擎）早期社区的许可证。Spring 由一个名为 Swedish Yankspankers 的 Total Annihilation 战队创建，最初设计用于运行 TA 内容。他们在 2005 年以 GNU 通用公共许可证（GPL）发布了这个引擎，这本质上允许任何人修改他们的代码，前提是必须在相同许可证下发布修改。我不知道他们为什么选择 GPL，也许他们更广泛地参与了自由软件运动，但这似乎是一个有意识的选择。

GPL 是一种限制性许可证，因为它限制了你收回其所提供自由的能力，但这对希望自己的作品保持自由的贡献者来说非常棒。因此 Spring 吸引了开发者并不断成长，它所运行的游戏也是如此。GPL 也适用于游戏本身包含的代码，这使良好运作的 mod 社区中常见的分享行为正式化了。这比任何团队单独所能做的更快地扩展了引擎的有效功能列表，例如 morph（变形）和 jumpjets（跳跃喷射器）是出现在一些游戏中的游戏端 gadget。项目甚至可以完全分叉以走向不同的方向，这就是 Complete Annihilation（完全歼灭），Zero-K 的前身，从 Balanced Annihilation（平衡歼灭）分叉的方式。引擎本身最近也被分叉了（并更名为 Recoil），证明了 GPL 如何让项目在维护者离开后依然存续。

听起来 Spring 似乎废除了所有所有权概念，但记住，游戏不仅仅是代码。该许可证将游戏视为具有独立许可证的内容包，并明确将美术排除在 GPL 之外。甚至有一款名为 P.U.R.E. 的商业发布游戏对其美术使用了专有许可证。这种排除的部分理由是美术是静态的且针对特定游戏的，而代码不是，因此共享代码的大多数理由在共享美术中并不存在。无论如何，这意味着 Zero-K 比引擎许可证要求的更加自由，所以我们需要多了解一些早期 Spring 的历史。

Spring 在 2000 年代后期处境尴尬。少量 Total Annihilation 的 mod 被移植到 Spring 上，仍在使用 1997 年的 TA 模型。这充其量在法律上是可疑的，而且看起来也不好，特别是对于那些投入工作制作原创美术的非 TA 游戏开发者来说。美术排除条款甚至被澄清过，以避免精美的新美术在未经许可的情况下与被盗的 TA 内容并列出现。因此，替换 TA 模型的想法在社区中一直存在，建模者不时发布模型包，所有这些都以自由软件的精神发布。Complete Annihilation 是第一个拥抱这个目标的游戏，我们甚至建立了一个模型仓库来跟踪重新建模的工作。

Zero-K 诞生于将 Complete Annihilation 中的旧 TA 内容清除出去的最后冲刺。在某个时刻我们意识到，大多数剩余的旧模型是 Arm 和 Core 之间本质上重复的单位或建筑，因此我们将精力集中在重新建模剩余的有趣单位上，放弃了其余的。这就是工厂作为阵营的推动力，尽管它也有坚实的设计原则作为支撑。Zero-K 出现后，一些重新建模的工作仍在继续，以及在旧单位被多次重新建模的情况下添加新单位。这就是为什么 Jack 和 Buoy 都基于旧的 Can。

Zero-K 继续保持免费，以尊重数十位在理解其会保持免费的前提下做出贡献的人。此外，一些模型和大多数音效使用了非商业许可证。缺点是志愿者通常缺少空闲时间，无聊的任务会被搁置，但优点是任何人都可以贡献。没有许可证需要处理，只需克隆仓库就可以开始开发。偶尔有玩家参与进来，积累专业知识，然后成为开发团队的一部分。我就是这样开始的，尽管不可否认现在难度更大了，因为低垂的果实更少了。幸运的是，Zero-K 的许可制度也让 mod 制作成为了一个很好的起点。

志愿者开发模式持续带来回报，过去几年中许多最酷的功能都是热心社区成员的一次性项目。ThornEel 的等离子炮拖尾只是最新的例子。这些改进随着时间不断累积，其优势在于至少解决了一位玩家想看到的问题，而不是由可能不玩游戏的开发者自上而下地制定。每个人都是志愿者意味着这些想法可能需要更长时间才能实现，但有些任务甚至受益于缓慢推进。例如，对地形改造和飞机的修改必须在玩家群体中酝酿多年，漫长的时间框架给了我们迭代和改进的机会。

大多数想法可以相对快速地着手实施，而且可以在实践中学习很多东西。我们尤其需要在我们的薄弱领域，如市场营销方面招募志愿者。如果你想看到某些东西被完善或添加到 Zero-K 中，请来与当前的志愿者们交流，并考虑贡献你的一份力量。

# 冷评 #26 - 可靠的 Raven
日期：2025-05-10

Raven 可以说是 Zero-K 中最具影响力的单个单位，也绝对是被重做次数最多的。作为 15 年的主力轰炸机，它的爆发伤害定义了决定空中脆弱性的血量阈值。我说"曾经"是因为轰炸机阵容此后得到了充实，加入了 Magpie（喜鹊）和 Odin（奥丁），Phoenix（凤凰）也从"猫头鹰化"中解脱出来。但这只是故事的结尾，在此之前经历了无数次调整和重设计。完整的故事是物理性与可靠性之间的冲突，涉及 Raven 尝试过的武器和能力几乎与其余所有轰炸机加起来一样多。

早期 Raven

原型 Raven 继承自 Total Annihilation（经由 BA），最初是通用的轻型轰炸机。它是"通用"的，意味着它是重型轰炸机的较弱、较便宜版本。这不够好，我们需要有各自角色的单位，而不是通用的单一量产，因此首要任务是让这两款轰炸机有所区别。我们还希望轻型/重型轰炸机的划分符合我们对科技等级的模糊处理方式，所以如果重型轰炸机在后期接管战场，那应该是因为情况变化了，而不是因为它单纯有更好的属性。

我们的解决方案是给重型轰炸机更多的炸弹，制造真正的地毯式轰炸机，而轻型轰炸机则精简为投下一颗精确的炸弹。Raven 由此诞生，某种意义上 Phoenix 也是，尽管 Phoenix 最终更接近旧的重型轰炸机。以下是角色分工背后的理论。

地毯式轰炸擅长对付密集的军队，而密集军队在游戏后期更常见。
精确轰炸擅长消灭关键单位，如炮塔、镇压单位和指挥官，而且单次击杀在游戏早期更有影响力。精确轰炸机最重要的任务是狙击指挥官，这是一种经典的战术——攻击者试图一击炸死敌方指挥官。这种狙击的可能性会扭曲游戏设计，指挥官的血量在很大程度上围绕着使他们难以被狙击来平衡。原始的轻型轰炸机已经是此类狙击的首选单位，因为它是性价比更高的伤害来源，这使它成为被重做为精确轰炸机的自然选择。

指挥官狙击者需要知道需要多少架轰炸机才能杀死一名指挥官。太少的话，指挥官存活并被修复，浪费了所有在尝试中损失的轰炸机。太多的话，尝试会被延迟，给防空更多时间来建立防线。像原始轻型轰炸机那样发射一片弹幕，会使结果有一定随机性。也许四架轰炸机只有 80% 的机会投出足够密集的弹幕，所以你需要带第五架来将击杀概率提高到 99%。这种程度的随机性感觉不好。

更准确地说，随机性可以分为两类：输出随机性和输入随机性。这个概念在其他地方已经被大量讨论，简而言之：

输入随机性是指某种随机因素设定一个情境，然后你做出应对决策。想想随机地图生成。
输出随机性是指你做出决策，然后一个随机过程决定结果。想想 X-Com、D&D 或 Starcraft 中的命中判定。普遍共识是玩家不喜欢输出随机性，因为它剥夺了控制权。我的观点是，输出随机性对于 Zero-K 这样的游戏是必要的，因为一个决策的结果为下一个决策设定了情境。话虽如此，单一动作的结果范围——波动幅度——不应太大。指挥官狙击是一个对游戏有重大影响的单一决策，因此最终结果不应取决于炸弹碰巧飞向哪里。所以，就有了精确轰炸机。

武器开发

精确 Raven 背后的理论是合理的，但我们忽略了一件事：指挥官会移动。指挥官不喜欢被狙击，所以当炸弹投向他们时会闪开。事实上，Raven 甚至无法击中直线行走的指挥官，因为默认的轰炸机武器类型不会提前预判目标。这是一个引擎限制，源于炸弹继承了轰炸机的速度。炸弹不是被射出的，而是被投下的。这将瞄准与运动绑在了一起，没有人有足够信心在如此复杂的情况下搞定目标预判。所以 Raven 需要一种新武器。

第一次重做源于 Complete Annihilation 是一群贡献者在做酷东西的集合，而非出自设计。某天有人醒来想："所有 Arm 轰炸机应该射激光"，于是所有 Arm 轰炸机都射激光了。Phoenix 变成了灯光秀，Likho 用粗厚的紫色光束绑架单位，Thunderbird 发射一片闪电而非类似 Imp 的集束炸弹。这些改动中只有一个经受住了时间考验。至于 Raven 对应的轰炸机，它向下发射短促的爆发，像一个笨拙地在空中飞行的 Grizzly。

激光突出而非解决了纯粹基于移动瞄准的问题。Laser Raven 对正下方的一切都有完美精度，但同样，指挥官会移动。即使是缓慢的指挥官也能横向移动，速度足以超过 Raven 转向和修正的能力。一个闪避中的指挥官的角速度会随着 Raven 接近而急剧增加，需要荒谬的转向速率，使 Raven 的行为更像花滑运动员而非飞机。我们太执着于让飞机表现得像真实飞机，所以又回到了绘图板。

排除了激光后，我们转向火箭。从技术上讲，火箭具有圆柱形射程，因此短程火箭可以向下方很远的目标射击。真实感原则对此有些说法，但大体上可行。遗留问题通过将射击点沿机翼底面移向目标所在的一侧来解决。结果是一个可以可靠射击并命中直线行走指挥官的 Raven，甚至是垂直行走的。然而，曲线路径则是另一回事。

目标预判通过预测单位速度方向上其将要到达的位置来工作。它无法考虑转向，因为单位可以在任何时候选择转向。结果就是 Raven 的"快速反应事件"——每当 Raven 向指挥官开火时，玩家就让指挥官跳个小舞，或只是绕圈走。我们将火箭炸弹做得足够快以使闪避不轻松，但我们受到限制，要避免 Raven 对快速单位过于强势。理想情况是 Raven 能可靠地击中慢速单位，而不会屠杀中型突击单位。这个难题的解决方案以一种新能力的形式出现：俯冲。

俯冲的十年

俯冲最初是一个与之无关的、为了解决与界面斗争的尝试。我注意到了三件事：

Raven 的伤害足以一击摧毁大型护盾生成器。
Raven 的飞行高度略高于护盾，护盾可以吸收多枚炸弹。
降落使轰炸机更贴近地面。理论上，你可以将 Raven 飞过一个护盾上方，命令它停止，然后在进入护盾后的一小段时间内开火。我不确定是否真有人这样做过，但这不是重点，重点是存在一个单位误用自身能力的明显案例。这本可以通过让 Raven 飞得稍低一些来解决（最终也确实如此），但我给了它俯冲到护盾下方的能力。后来证明 Saktoth（游戏设计师）本意就是让 Raven 对护盾效果不佳，但与界面的斗争是显而易见的，所以护盾俯冲被加入了。当然，技能使用不能取决于敌人的存在，所以我们还允许玩家命令 Raven 持续在较低高度飞行。

效果立竿见影，人们开始让 Raven 飞得更低以更多地命中指挥官，显著提高了可靠性。随后默认行为被更新为让 Raven 在攻击移动目标时自动俯冲，它们甚至根据目标的速度和高度调节飞行高度。这个机制有一个巧妙的副作用：使 Raven 在面对更快的单位时更容易被反击，而炸弹减速更加剧了这一方面。最终我们让 Raven 在俯冲时减速，使其特别容易受到快速突击单位的攻击。这就是 2011 年到 2022 年间超过十年的 Raven，但它并非没有问题的解决方案。

第一个问题是俯冲是一个复杂而强大的能力。Raven 具有选择自身高度的隐含能力——可以实时调整，权衡速度——这正是单位 AI 所使用的。这使其成为有效的防空单位，因为它可以飞到其他飞机通常固定的高度，在它们体内生成炸弹，造成巨大伤害。我们最终添加了任意的目标限制来防止这种情况对非武装直升机生效，但需要这样做本身就是一个危险信号。在复杂性方面，Raven 有许多变蠢的方式，比如过早俯冲得太低去追击特别快的突击单位，结果永远追不上。

第二个问题是视觉上的。Raven 俯冲下去近身搏斗快速单位看起来有点滑稽。尤其是考虑到它相对较宽的模型。低飞在不平坦的地形周围看起来也很怪，因为 Raven 必须非常陡峭地爬升以避免撞上去。所以，在俯冲理念十年之后，我们再次尝试了从巡航高度开火。

滑翔炸弹

最终重做产生了 Zero-K 中较为复杂的武器之一，尤其是考虑到玩家详细理解它的重要性。但每个细节都解决了一个特定的问题，所以我们开始吧。

现代 Raven 发射一枚制导导弹，但与其他导弹不同，这枚导弹被限制为大致只朝地面方向飞行。结果是一种制导"降落伞"炸弹，能机动击中其下方锥形范围内的任何东西。射击逻辑在判断命中概率时会考虑炸弹的速度和锥形的角度。建筑会被在最大射程处射击，而慢速单位可以在稍短的射程内被射击。存在一个最小射击距离，以避免 Laser Raven 的问题，而快速突击单位甚至可以跑赢炸弹。Raven 仍然会尝试攻击突击单位，因为不这样做可能看起来像一个 bug，而且它在活跃战斗中总是可以作为区域封锁工具使用。

我们终于有了一个可以可靠地射击并命中指挥官的 Raven，除了跳跃型 Recon（侦察）指挥官，但它为此能力付出了降低血量的代价。说到这个，jumpjets（跳跃喷射器）和 Raven 面临类似的设计问题，因为它们都可能产生高波动的结果。区别在于 jumpjets 一开始就可靠，我们必须抵制增加不稳定的冲动，而精确轰炸本质上就很困难，所以花了多年才找到既可靠又符合物理逻辑的方案。

Raven 的内容一篇帖子装不下，所以未来会有关于 Raven 的更多内容，比如为什么轰炸机需要装填，或者为什么地形改造需要包含一款可以水下和向下射击的轰炸机。

# 冷评 #27 - Widget 和 Gadget
日期：2025-06-01

Zero-K 运行在 Spring 引擎（现为 Recoil）上，该引擎用 C++ 编写。C++ 是一种速度优秀的编程语言，但对一般游戏开发者来说，一大块纯粹的 C++ 太令人望而却步，不敢深入编辑。尤其是引擎需要处理如此多的基础系统，如多人游戏、寻路和物理。幸运的是，Lua 在 2006 年末被添加到了引擎中，结果 Zero-K 本身几乎完全用 Lua 编写。

Lua 是一种流行的游戏脚本和 mod 编写语言。Spring 游戏可以包含 Lua 代码，然后引擎会在广泛的可用事件上触发这些代码。一旦触发，Lua 脚本可以执行诸如修改游戏世界或在屏幕上绘制东西等操作。例如，喷射火焰的 Pyro（纵火者）会触发 UnitDamaged 事件来告诉 Lua 将单位点燃。着火不是引擎概念，而是 Lua 通过在 DrawWorld 事件中绘制火焰并在 GameFrame 事件上施加伤害来实现的。

脚本编写花了大约一年多才添加到引擎中，这改变了游戏规则。在 Lua 之前，新机制只能来自引擎，因此只能以引擎开发的速度添加。这要慢得多，因为 C++ 开发者的人数较少，而且引擎代码由于在游戏间共享而被要求更高的标准。更糟糕的是，引擎更新定期发布，进一步延长了反馈和迭代周期。引擎这种有条不紊的发布过程对即时战略游戏基础的稳定性很好，但对于快速迭代新机制就不太好了。

在 Lua 之前也添加了一些实验性机制，尽管大多数后来被废弃了。引擎功能的问题在于它们难以修改或更新，所以大多数游戏都发展过了那些较为晦涩的功能。即使你能凑齐 C++ 的"法力值"，其他游戏可能想要旧版本的功能，导致配置选项激增，增加了修改该功能的难度。调整最简单的东西也可能迅速变成多个游戏之间的讨论，而且一切都有性能或维护上的权衡。

不要把上述内容理解为引擎毫无用处。它在无需脚本的情况下支持完整的即时战略游戏体验，并有健壮的系统供 Lua 交互，例如命令处理系统。一些更具实验性的功能甚至被大多数游戏使用，如 bubble shields（气泡护盾）和 area commands（区域命令）（尽管我听说 COB 脚本在 Total Annihilation 中技术上可以制作区域护盾）。只是碰巧有很多其他残余机制，如 aircraft flares（飞机热焰弹）、repulsion shields（排斥护盾）、seismic detection（地震探测）和一种早期形式的 line move（直线移动），在 Spring 游戏中很少被使用。

Spring 本质上以两种方式支持 Lua：widget 和 gadget，统称为 wupget。Widget 在每个玩家的计算机上独立运行，有自己的内部状态，其公平性在于它们只能看到和做到与玩家相同的事情。人们甚至可以安装完全不同的 widget 组来玩同一局游戏。Gadget 则完全不同，它们作为游戏本身的一部分运行，实现游戏机制。Zero-K 中大多数炫目的新东西都是由 gadget 驱动的，如 jumpjets、terraform（地形改造）、overdrive（超载）、cloakers（隐形装置）和 unit AI（单位人工智能），但游戏中许多"平凡"的部分也由后台的 gadget 管理。

2007 和 2008 年间 widget 开发经历了爆发期，创建了许多支撑大多数 Spring 游戏基本 UI 功能的 widget，包括 custom formations（自定义编队，即 line move）和 command insert（命令插入，即 Space + 点击）等 widget。这主要是由玩家自由分享代码和想法、快速实验和发现有效方案来完成的。任何人都可以在任何游戏中安装 widget 运行，这意味着你可以在每场比赛之间调整你的 widget。迭代速度和玩家的直接参与前所未有地高，因此 widget 迅速超越了引擎 UI，以至于引擎端的 UI 开发完全停止了。

Widget 对 Zero-K 影响巨大，因为 Complete Annihilation 与 widget 爆发期同时进行，所以我们亲眼目睹了 widget 如何颠覆现有游戏的设计。例如，Fencer（击剑手）可以在移动时射击，直到我编写了早期版本的 autoskirm（自动缠斗）widget。玩家在自己策划 UI，而且这些 UI 越来越强大，因此选择要么完全阻止，要么拥抱它。我们选择了后者，并努力打造一款能承受强大 UI 压力的游戏——最终目标是"与对手战斗，而非与界面战斗"。

Gadget 也产生了影响，但更多是在开发方面。Complete Annihilation 的开发相当开放；一旦你有了提交权限，你几乎可以实验和添加任何东西。于是人们就直接动手了。jumpjets、area cloakers（区域隐形装置）、terraform、overdrive、shield link（护盾链接）、construction priority（建造优先级）、laser bombers（激光轰炸机）、morph——这些都是由想添加它们的人添加的，其他开发者没有什么余地说不。我们吸引了那些对突破 TA 限制感兴趣的人，这得到了回报。

最初的创造力爆发持续了大约五年，之后我们开始整合。最值得注意的是，我们不再只是将游戏塞满各种酷炫的 widget，而是开始创建合理的默认设置，着眼于为新玩家提供功能性的 UI。Widget 从玩家自行携带的东西，变成了游戏提供的东西。Gadget 也得到了精炼，一些如 experience morph（经验变形）的被完全移除。但移除是最后手段，我们想保留创造力。UI 用一个名为 Chili 的定制 Lua UI 框架重写了，尽管一些最重要的 UI 工作不是通过 widget 实现的，而是通过创建 unit AI gadget。

Unit AI（单位 AI）往往始于 widget，因为它允许玩家在无需任何权限的情况下自行调整，不必担心稳定性或发布周期。好的 widget 就来源于这个系统，但有两个大问题。第一个是公平性，因为增强的单位 AI 带来优势。这可以通过将所有 widget 包含在游戏中来解决，但这依赖于作者愿意分享和打磨 widget。分享很容易，从来没有问题，但打磨很难。第二个问题也是公平性，但更深层次的延迟问题，这是 widget 根本无法解决的。

Widget 受网络延迟（即 ping）影响。这是因为玩家有延迟，而 widget 实际上是与玩家并行运行的。我们通常将延迟视为在线游戏不可避免的事实，几百毫秒在即时战略游戏中意义不大，玩家可以适应。然而，widget 是脆弱的代码，难以适应延迟的命令，导致它们崩溃或效果大幅波动。本质上，AI widget 倾向于放大延迟固有的不公平性。从技术上讲，widget 的处理是立即发生的（由于引擎的网络结构，见下文），但响应仍然受到到服务器往返 ping 的影响。

Widget 无法解决延迟，所以我们不得不将单位 AI 迁移到 gadget。Gadget 直接作用于游戏状态，无延迟地观察和响应。这就是为什么没有 autoskirm widget；它现在已经内置于游戏中。不幸的是 gadget 不易于配置，这就是为什么 Zero-K 现在有大量的单位状态开关，但这代价不大。Gadget 也必须小心不作弊，因为从技术上讲它们是全能全知的，但 Spring 提供了充足的工具来将 Lua 限制在玩家可见的范围内。

Gadget 不仅用于添加机制和打磨 widget；它们还用于打磨（替代）底层引擎。例如，基本的移动命令——右键点击世界时发生的事——是一个由 gadget 处理的自定义命令。引擎提供了基本的移动命令，但它会导致单位在路径点周围堵塞，这不符合 Zero-K 设定的控制标准。在 Zero-K 替换它之前，引擎实现甚至更糟，但引擎因此有了一些改进。还有许多其他例子，比如引擎未能在建造者之间均匀分配资源。理想情况下这些应该在引擎中完成，但在它们实现之前，Lua 可以填补空缺。

用 Lua 替代引擎并非没有挑战。C++ 的性能好得多，所以我们尽量利用引擎的强大功能。幸运的是，引擎的大部分都暴露给 Lua，例如，自定义移动命令使用 Spring.SetMoveGoal 而不是试图为上千个单位自行寻找路径。这是很好的分工，引擎处理资源密集型任务，让 Lua 来整合结果。

以上仅仅是 Lua 的皮毛。整个主菜单恰好也是用 Lua 编写的，使用与游戏内 UI 相同的框架，但这更多是 Lua "法力值"的问题而非技术必要。单位用 Lua 脚本进行动画，整个对手 AI 也用 Lua 编写。但也许 Lua 最大的持续影响要追溯到 2007 年的创造力爆发：Zero-K 本身有许多使用 Lua 深入引擎才能实现的 mod 和新功能。所以如果你想写一些 Lua 而不仅仅是读读相关资料，不妨试试 mod 制作。

# 冷评 #28 - 载具和机器人
日期：2025-06-22

经过一年半的发文，最接近工厂总览的是关于 jumpjets 的帖子。是时候改变这一点了，所以作为铺垫，我们来看看底盘类型（机器人或载具）如何影响工厂设计。工厂是 Zero-K 的阵营，虽然每个工厂大约有十个独特的单位，但这些单位所承担的角色有一定的共性。机器人和载具之间的划分可以追溯到 Total Annihilation 的 Kbot 和 Vehicle 科技路径，此后经过了扩展和精炼。

Zero-K 中的机器人（bot）是拥有两条或四条腿的机器人，其特点是能够攀爬较陡的山坡。与机器人相对，载具（vehicle）是依靠轮子或履带行进的东西，或者甚至是作为科幻悬浮形式漂浮的，只能攀爬平缓的坡。我们进一步区分机器人和载具的方法是赋予机器人出色的转向速率和加速度，同时让载具平均速度更快。这很合理；人很灵活，可以随时启动或停止，而汽车有明显的转弯半径和更高的极速。这种类比在比较最大陡度或坡度容差时可能不太适用，但它至少在爬楼梯的体验中有根据。还有一个额外的坡度容差类别——spider（蜘蛛型），可以攀爬包括陡峭悬崖在内的所有坡度。

按坡度容差划分机器人和载具是一种继承的选择，在脱离语境时显得很奇怪。大多数游戏对移动采用二元方法；陆地单位要么能走到某处，要么不能，而且这是有充分理由的。支持两种通行能力对地图设计和平衡施加了相当多的约束。我们无法让所有工厂在所有地图上都可行，仅仅是因为载具无法在足够丘陵的地图上使用。载具变得无用的节点比你想象的要早，因为能在地图上高效移动相当重要。因此为了平衡这一点，载具被平衡为比机器人稍强一些，通常通过前述的速度优势，但这最终让它们在开阔地图上获得了优势。

我们能期望的最好结果是让大多数工厂在大多数地图上都可行。这有利于全面型而非专精型的游戏风格，因为很难选择一个工厂然后每局都使用它，尽管有几个工厂这样做是相当合理的。我们的目标是在 1v1 匹配地图上支持多种工厂，团队游戏通常也有适合大多数工厂的前线。简化的坡度容差会让 Zero-K 更易于上手，但我怀疑我们会因此失去单位和地图的多样性。

坡度容差只是机器人和载具之间区别的开始。大多数差异可以归结为灵活与速度，但这很可能是对合理设计的直觉理解之上的事后推理。蜘蛛（Spider）在区分中是荣誉机器人，而且往往成为载具的极端对立面。

机器人工厂倾向于更多的专精和实用功能。看看它们的名字：Cloakbots（隐形机器人）、Shieldbots（护盾机器人）和 Jumpbots（跳跃机器人）都体现了它们标题中的机制，而 Spiders（蜘蛛机器人）和 Amphbots（两栖机器人）分别都能攀爬悬崖和在水下行走。虽然这些机制并非其工厂独有，但大多数例子都能在各自的工厂中找到，而每个工厂的其余单位都围绕其机制设计。例如，没有护盾的 Shieldbot 喜欢待在有护盾的同族下方，而不隐形的 Cloakbot 利用其建造者的区域隐形装置。

三个载具工厂——Rover（越野车）、Tank（坦克）和 Hovercraft（气垫船）——则不那么聚焦。Rover 本质上是快速轻型载具，而 Tank 是耐久的、稍慢的载具。Hovercraft 的特点是能在水面上漂浮，但这几乎不像 Amphbot 在波浪下躲避那么有影响力。此外，Amphbot 和 Hovercraft 也设计为在陆地地图上使用，在那里 Hovercraft 最终成为极端的载具。这主要意味着它是一个由玻璃大炮组成的工厂——除了 Halberd（戟），它恰恰相反。

大多数工厂包含大多数角色，但工厂的侧重点和构成受到其移动类型的影响。爬行炸弹是机器人工厂独有的，Spider 稍微打破了这一趋势，拥有一个理论上可重复使用的隐形刺客。机器人还占据了大多数常规缠斗单位，唯一的例外是 Hovercraft 工厂的 Scalpel。这两个角色受益于机器人的灵活性，因为缠斗单位需要在射程内进出自如，而炸弹需要能够修正航向以命中敌人。Rover 曾经有一个正式的缠斗单位，但在一个 AI widget 搞坏了 Fencer 之后失去了它，而当 Tank 被组建时，缠斗单位的空位被故意留白。

灵活的机器人移动也支持了机器人对特殊能力的侧重：想象一下试图操控隐形载具穿过敌方基地，或者维持一个护盾载具的重叠阵型。这些能力也鼓励玩家在大型开阔地图上使用机器人，因为载具往往局限于更直接、更粗暴的手段。载具军队总是能从机器人的支援中受益，因为有时玩家需要一些隐形侦察兵或一点重力操控。

载具工厂更多关注蛮力。它们的突击单位稍好一些，或者至少更通用，这主要归结为载具的速度优势和坦克的耐久性。但载具除了速度之外的最大优势是它们的火炮。三个载具工厂总共有五个火炮单位，其中四个的射程超过所有机器人火炮。在五个机器人工厂中，只有 Jumpbot 和 Cloakbot 拥有射程不错且伤害实在的火炮，而 Spider 和 Shieldbot 在平坦平原上无法在其射程外击杀一个 Stinger（毒刺炮塔）。机器人工厂经常必须利用地形，或者建造自己的墙或散兵坑，来安全地交战敌方防御工事。

将火炮集中在载具工厂中，鼓励了它们在原本对载具来说过于丘陵的地图上使用，而快速机器人突击的缺乏使防御在这些地图上感觉更加坚固。所有这些都赋予了地图自己的感觉，因为如果所有工厂都具有相同的能力，在丘陵地图上使用机器人会感觉和在平坦地图上使用载具没什么不同，也就没有混合移动类型的理由。事实上，拥有更多可用单位的优势往往超过次优的地图条件，所以人们在团队游戏中仍然会使用各种工厂。然而，这给围绕特殊移动类型建造的工厂带来了一些问题：Spider 和 Amphbot。

Spider 是唯一完全全地形的工厂，这使它有风险成为在足够丘陵的地图上唯一可行的工厂。这不会造就有趣的地图，最终成为地图设计的约束，所以我们限制 Spider 以推迟"足够丘陵"的阈值。这就是为什么 Spider 特别慢，且缺乏真正的突袭单位或火炮单位。结果，Spider 依赖于有合适的地形可以利用，这使它处于一个介于独立陆地工厂和支援工厂之间的奇怪位置。

Amphbot 受害于它是 Zero-K 中最新的添加。当其他机器人工厂是从现有单位组合而成再充实的，Amphbot 是从零开始创建以匹配已建立的模式。为什么 Limpet（藤壶）是如此奇怪的爬行炸弹？因为 Amphbot 需要一个炸弹，而所有常规创意都已被占用。工厂仍处于一定程度的变动中，甚至有风险使这篇帖子过时，因为尚不清楚 Bulkhead（舱壁）是否会保留足够的射程来将其归类为火炮。Amphbot 也稍微受到上述 Spider 问题的影响，但由于海洋设计的原因没那么严重。

最终，载具和机器人的划分既关乎工厂阵容中刻意留下的空缺，也关乎现有单位。或者如果你愿意的话，整个设计是 Quant's Rule 在游戏范围内的应用：优势得到加强，而空缺得以保持。这些自我施加的约束相当任意，但它们保持了工厂和地形类型的独特性。这确实比把每种武器和每种底盘组合在一起要好得多。

# 冷评 #29 - 飞机、弹药和防空
日期：2025-07-13

严肃的策略游戏倾向于将其范围分布于三个标准战场：空中、陆地和海洋。Zero-K 遵循这一模式，尽管它明显更游戏化和爱玩乐。这一点在对飞机和武装直升机的写实化应用上体现得最为清晰。现实世界对飞机行为有一定影响，并非常宽松地为它们的支援角色提供了参考，但这只是一个起点。我们的首要任务是让空中在整个游戏中都好玩且有用，因为太多时候它最终变得有些单一维度，或者仅仅是被降格为奇招。

首先，空中单位应该如何表现？我们受粗略直觉的引导，所以如果看起来差不多，那就是对的。真实的飞机存在，但往往飞得太高太快以至于看不见，所以飞行的平均体验可能来自鸟类和昆虫，加上一点直升机和无人机的影响。这些东西很快，所以 Zero-K 中的空中单位也很快。共同的经验还催生了两种类型的飞机运动方式，和地面一样，我们用此将飞机分为两类工厂：Gunship（武装直升机）和 Plane（固定翼飞机）。

武装直升机类似于直升机和无人机；它们飞来飞去，有些相当灵活，都可以悬停在原地。这是标准的即时战略游戏处理方式，有时被批评为不切实际或使空中和地面战斗以相同方式展开。话虽如此，地面战斗很扎实，而且许多游戏中的武装直升机型空中单位通过比地面单位更快——至少平均更快——来彰显自身。武装直升机还擅长从任何方向进攻，并在试图撤退时避开障碍物。

固定翼飞机像真实的飞机一样飞行，分为三种：轰炸机、战斗机和侦察机。它们只有在降落时才会停下，拥有机翼、副翼和方向舵。有一个底层飞行模型以及上述参数和更多参数，我们用这些来调校出飞机那种现实中难以操控的感觉。Complete Annihilation 调整了这些参数，使其飞机比前代更可控，但飞机对飞机的战斗仍然相当难操控，以至于我们避免期望玩家点击单个战斗机，因为它们实在太快了。

快速单位很麻烦，因为它们天然主导早期游戏。速度是一个惊人的属性，尤其是当有大量空旷空间可以移动时。飞机最甚，它们是目前最快的单位，所以大量空中设计都集中在如何让它们在早期异常弱势上。如果不将空中偏向早期游戏之外，我们就会在早期看到它们，然后就再也不见了。

飞机在早期被削弱的方式是要求轰炸机在每次出击后装填。在此之前，轰炸机可以绕着敌方基地不停飞行，只要没有防空（AA）来击落它们。这在现实中只会在早期发生，最可能是在一次突袭之后。装填还会消耗能量，使其在早期比后期更加昂贵。

武装直升机没有干净的早期解决方案，但它们不像固定翼飞机那么快，所以问题有所减轻。部分解决方案是它们糟糕的属性：一架武装直升机通常花费两倍的代价来获得两倍的速度，但表现与地面单位相同甚至更差。其中许多对突击单位也很糟糕，而且太贵不适合快速rush，这增加了它们早期的脆弱性。不过轻型武装直升机仍然处于尴尬的位置，而防空使情况更加尴尬。

防空是空中设计中必要但略微令人遗憾的一部分。飞行单位从物理模拟炮弹中受益匪浅，因为这使它们能避开大多数武器的大部分伤害。即使是即时精确命中的光束武器，要么造成持续低伤害，要么有持续爆发但只命中快速单位一小部分伤害。加上光束具有球形射程，所以不擅长向上射击。因此 Zero-K 拥有只能射击飞机的单位，称为 AA 或专用防空。这是单位可以向任意方向射击这一理念的最大例外，但这是我们为了释放空中设计空间而做出的权衡。

除了专用防空，Zero-K 还有 flex-AA（灵活防空）。这不是一个定义明确的单位集合，而是一种理念——许多单位对飞机来说还不错，特别是当飞机直接飞越头顶时。灵活防空对武装直升机尤其有效，因为它们比固定翼飞机更慢，甚至在某些情况下使专用防空变得多余。事实上，我们曾多次增强武装直升机对抗专用防空的能力。早在 2010 年，我们将飞机血量和防空伤害都降低了 25%，这等同于让飞机对灵活防空更弱，而武装直升机对专用防空更强。这在 2014 年以 1/11 的幅度重复，因为当时 Raven 有 1100 血量。不过这种方法有限度，因为在某个节点上最有效的灵活防空开始摧毁飞机。

专用防空存在一个潜在问题：如果你只面对空中单位，它可以被无限量产。我们通过宣布这个问题不存在来解决它。Plane 和 Gunship 被刻意设计为支援工厂，它们不应该能独自守住前线，而且应该在 1v1 中无法放置。当然它们技术上可以放置，因为基于玩家数量阻止放置会太随意，所以我们只能尽量劝阻。Gunship 在 1v1 中是特别的问题，因为逼出一些防空然后快速切换到地面会带来巨大优势。除了让它们的单位在早期很糟糕外，我们还不得不让它们的建造者太慢太贵以限制早期扩张。

空中工厂不像其他工厂。陆地工厂是某种程度上自给自足且灵活的单位集合，而空中工厂只构成反制三角的一边。空中克制地面军队，地面军队克制防空，防空反过来克制空中。所以如果敌人造大量防空，空中玩家可能需要切换到地面来消灭防空。这使空中成为升级游戏局势的力量，因为它增加了额外的互动层，类似于核弹与反核弹的关系。这使空中不那么像一个阵营，而更像一种技术选择，至少在 1v1 中是这样。

即使作为后来建造以升级游戏局势的第二工厂，Gunship 的可行性一直存在问题。当你意识到 Plane 提供最快的单位、最好的侦察兵，并且可以执行毁灭性的突袭轰炸时，这就很好理解了。Gunship 只比地面单位稍快，可以从任何地方进攻，属性很糟糕。侦察问题尤其催生了 Sparrow（麻雀）的添加：一款可以通过变形雷达塔生产的轻型侦察单位。这给了所有工厂侦察的能力，因此添加另一个陆地工厂也更加可行，这是观察跨工厂协同效应的好方法。

最终，空中并非 Zero-K 的主要领域，但它一直在努力突破并成为整个游戏的全部，而不仅仅是反制三角中一个充实的角落。这对仅仅大约二十个单位来说要求太多了，结果它们成为游戏中被调整和重做最多的单位，Raven 领跑其中。具体内容可能在未来的帖子中出现，因为那会使这一篇变成三倍长。

# Cold Take #30 - 基地建造模块

基地建造似乎是 RTS 类型中不可或缺的部分，而且实现方式多种多样。有些游戏将大部分建筑限制在主基地内，而另一些则鼓励玩家将建筑散布到整个地图上。许多游戏将基地作为具象化的科技树来解锁更强大的单位，或者至少将基地建造作为某种升级形式。但基地建造最有趣的部分——至少对我而言——是其具象化特性。建筑在地图上物理存在，这反过来改变了每场战斗的地图运行方式。

放置建筑本质上是一种地形改造形式。至少在最低限度上，建筑会阻挡单位移动。这使得穿越某个区域需要更长时间，并且可以聚拢、分散或保护单位以创造战术优势。许多游戏更进一步，防止单位从建筑之间挤过去，从而能够建造不可逾越的墙。堵墙在一些最受欢迎的竞技 RTS 中扮演核心角色，例如 Starcraft 2（星际争霸 2）和 Age of Empires 2（帝国时代 2），但在 They Are Billions（亿万僵尸）等单人游戏中也很突出。

堵墙与建筑网格之间有着有趣的关系。在千禧年之交，我们作为一个文明开发出了将建筑放置在任意位置而无需吸附网格的技术，这在当时可是了不起的成就。许多游戏转向了无网格模式，但建筑网格一直存在并延续至今。原因如下：小整数是你的好朋友。

粗糙的建筑网格对建筑放置施加了显著限制，而这种限制带来了清晰度。玩家可以快速学会区分几种不同的间距大小，而最小间距很容易与零间距区分开来。新侦察到的基地周围的路径可以一目了然；想象一下试图在一个无网格基地中导航，你无法确定某个间隙是单位宽度的 99% 还是 101%。粗糙网格也更容易将建筑最优地装入有限空间，这甚至还没有考虑某些游戏允许你自由旋转建筑的情况。

如果精确定位建筑很重要，那就重要到不能徒手操作。但游戏有很多理由选择无网格模式，而且许多缺点可以通过让单位从零间隙建筑之间挤过来来缓解。无网格放置让玩家能够榨取微小的效率，适合那些想要精细化布局的游戏，并且能带来更自由流畅的感觉。网格的外观也可能与更写实的设定或基调冲突，因为完美排列的建筑往往显得人工化。Planetary Annihilation（行星湮灭）因球形行星无法包裹方形网格而被迫采用无网格模式，这一点很值得注意。Populous: The Beginning（上帝也疯狂），另一款设定在行星上的游戏，则成功采用了网格，因为它的行星是环面的。

那么 Zero-K 在这一切中处于什么位置呢？你可能已经猜到了，我们喜欢堵墙的地形改造方面，所以我们使用网格来允许堵墙而无需与 UI 太多搏斗。更一般地说，Zero-K 的基地散布在整个地图上，因为资源也是分散的，而且包含大量类似墙壁的建筑和炮塔。炮塔可以被视为一种独特的地形改造形式，因为它仅通过施加危险来微弱地抑制移动，而且只针对敌人。

建造一个可防御的基地有点像一个异步拼图，防守方设定挑战，进攻方尝试解决它。过去和现在决策的相互作用是一种巧妙的方式来增加玩家互动，而无需疯狂的微操，尽管防守方通常也会调动机动部队。Zero-K 的物理弹道为这个拼图增添了另一层深度，因为友军火力并不友好。你自己的建筑会为炮塔和防守部队制造盲区，这往往有利于进攻方而非防守方。

让玩家参与地形改造的一个好方法是免费给予他们一些。想想 Age of Empires（帝国时代）的房屋或 Starcraft（星际争霸）的补给站。这些就是我所说的强制资产，因为玩家被迫建造它们，但它们也有次要用途。补给建筑是非常好的强制资产，Blizzard（暴雪）尤其用它们来容纳相当多有趣的能力。但在这里，在 Total Annihilation（横扫千军）子类型中，我们喜欢给玩家大量选择，所以 Zero-K 只有一个弱强制资产——Solar Collector（太阳能收集器）。太阳能足够坚固可以充当可行的墙壁，但由于存在其他更高效的能源发电机，它并非完全免费。尽管如此，效率差距实际上就是对堵墙的大折扣。

Solar Collector 的对应物是 Wind Generator（风力发电机）：游戏中最脆弱的建筑之一。它体现了将资产物理放置在地图上的脆弱性，而玩家总会拥有一些脆弱建筑。这种脆弱性是 RTS 的驱动力，因为如果每个人在任何地方都很强大，就没有理由优先攻击某个特定位置而非其他位置。脆弱建筑为对手充当基于位置的目标，这在某种意义上是对地形的改造。其他基于地图的类型，如 DotA 类和团队射击游戏，将精心设计的目标嵌入地图中，而这种情况在 RTS 中较少见，因为玩家基地非常擅长生成动态变化的目标。

将建筑放置视为地形改造让我们能够更宏观地看待玩家与地图的互动方式。每当玩家投入资源刻意以任何方式改变地图时，他们就在声明受影响区域的重要性。防御工事对军队在地图上的移动和互动方式有长期影响，并进一步改变其他区域的重要性。游戏的胜负取决于对什么重要的分歧，而通过建筑放置来表达这种分歧是战略核心中长期决策的重要组成部分。

Zero-K 拥有一种更炫酷的地形改造形式：我们称之为 terraform（地形改造）的移山填海机制系列。这篇文章的种子是关于地形改造的，但如果把所有功劳都归于它来营造在地图上扩张并按你的意志弯曲它的地形改造感，那就错了。声称 Zero-K（以及其他少数有地形改造的游戏）创造了这种感觉也是不对的，因为不真正搬运地面也能做到很多。地形改造很有趣，特别是与更传统的地图改造方式对比时，因此在这里涵盖基础知识为我们留下了大量空间来探讨使地形改造系统真正运作所涉及的挑战。敬请期待三周后的 Cold Take #31。

# Cold Take #31 - 地形改造

我是地形改造的忠实粉丝，即使在即时战略游戏中也是如此。在最佳状态下，地形改造是一种超强版本的地图修改，建成后不属于任何一方，允许双方适应新的地形。一面用来躲避炮击的墙在当前可能是好主意，但当敌人利用它的阴影让蜘蛛群翻越而过时，它就变成了累赘。地形改造所鼓励的创造性问题解决和战略规划非常适合 Zero-K，因此建造墙壁、壕沟和坡道已成为游戏的核心部分。但达到这一点花了十多年时间，因为地形改造充满了问题。

当我试图客观看待它时，RTS 中的地形改造往往很糟糕。90 年代有多款游戏尝试过它，但大多只是一种勉强可用的噱头。基地建造已经足以让玩家在地图上留下印记，而基地建造比完全自由的地形改造安全得多。首先，地形的形状必须真正重要，修改它才有意义。很少有游戏满足这个要求，而且理由充分，因为让地形太重要是更大的风险。太多的细节会让人应接不暇，而操控这些细节的能力加剧了这个问题。

自由操控地形是 UI 噩梦。在 2D 屏幕上绘制 3D 形状并不容易，而玩家往往毫无准备就直接上手。这一点，加上可用操控的巨量数量，很容易导致繁琐的 UI。地形改造固有的自由度也极有可能被以无聊的方式使用。一座带有城垛和护城河的堡垒可能看起来很酷，但在战场上散布细小的尖刺来彻底搞乱敌方寻路肯定更便宜。单人游戏和慢节奏游戏有更宽松的环境，因为玩家愿意学习更复杂的 UI，而且没有对手会抗议廉价伎俩。但对于多人 RTS 来说，地形改造很可能使用起来不好玩，应对起来更不好玩。

上述问题是可解决的，但我们是在多年迭代后才做到的。最初的地形改造用一个可建造的地形方块替代了 Dragon's Tooth（龙牙），这可以追溯到 2008 年——Complete Annihilation（全面横扫）的鼎盛时期。Dragon's Tooth 是一种墙壁部件，建造时变成耐压残骸来工作，新系统的工作方式相同，只是改用了地形修改。有两种方块类型，用于升高或降低地形，两者都是 2x2 的占地面积，这是游戏中最小的占地面积。建造墙壁需要拖动一排方块，建造高墙则需要玩家在已建造的方块上放置方块。事情从这里开始变得繁琐。

地形改造 gadget 可能是一时兴起加入的，这在当时是常态，尽管在这个案例中恰好是由首席开发者 Quantum（量子）添加的。地形改造方块是一个异常保守的功能，但他很快通过下一个添加弥补了这一点：可以想象到的最粗糙的平整地形改造。Quantum 制造了一种新型方块，仍然是 2x2 的占地面积，但建造时会平整其周围大约 8x8 的区域。新的高度基于方块中心，因此通常与深坑或高塔配合使用。这非常强大，个人来说也非常有趣，但也非常破坏平衡。允许可持续地形改造乐趣的现代限制尚未出现。

此时地形改造在某种意义上是免费的，因为它只消耗工兵时间。它还可以移动建筑，所以标准操作是建造类似 Singularity Reactor（奇点反应堆）的东西，用壕沟方块挖一个深洞，然后在底部进行平整地形改造，将建筑移到地下深处。这个过程甚至可以重复执行，让 Antinuke（反核）可以上下移动来应对威胁。但这并不像你想象的那么常见，因为破坏性的地形改造甚至不是 Complete Annihilation 最粗暴的部分。你可以在核弹上伪装，为什么还要地形改造你的核弹？

我是地形改造的坚定支持者，想要更多功能，所以我添加了一个创建小坡道的方块，就像平整方块一样。坡道足以激发第一个 Pyro（火焰兵）发射器，如果我没记错的话是在 Victoria Crater（维多利亚陨石坑）的北侧，但它太有限了，固定在 45 度且只能指向基本方向。所以到 2009 年底，我决定从头重写地形改造。目标是制作真正的多功能坡道，修复建造墙壁的繁琐性，并添加套索 UI 作为平整方块更合理的替代方案。重写的骨架一直沿用至今，不幸的是，大部分旧代码也是如此。系统得以延续得益于严格的分工。

地形改造系统的第一部分充满了命令模式和关于地形改造允许操作的规则。玩家可以使用坡道、平整、升高、平滑和恢复命令，这些命令在某些方面是可修改和受限的。系统第一部分的全部目的是指定通用的地形改造段，它们只是"将位置 (X,Z) 移动到高度 Y"的列表。系统的第二部分处理这些段的创建和进度，不需要知道每个段来自哪里。所以实际上，所有地形改造类型都以相同方式构建和定价。

新系统奠定了良好基础，但仍有许多问题需要解决。有些通过匹配 2x2 方块系统的分辨率和隐含限制来解决，这砍掉了一部分多余的自由度。剩余的问题以三种形式出现，尽管解决方案通常同时攻克这三个方面。问题是：

单位很蠢，尤其是在瞄准方面。
鼓励有趣用法而非无趣用法。
如果 UI 很糟糕，没人会费心用地形改造。

最早的调整之一是添加了 Shraka 金字塔，以一位玩家命名，因为在我将一台 Razorback（剃刀背，与 Dante 相当的 Reaver）放在 Desert Triad（沙漠三星）的 FFA 中一座非常高塔上后，他提出了这个建议。高地形改造的性价比太高了，我那极端的应用简直相当于上了太空。一个可能的修复是让地形改造越远离原始地图越贵，但这种参考旧地形的方式感觉太随意和不透明。相反，我们添加了一个要求：地形改造需要通过在自身周围建造一种裙边来支撑自己。额外的泥土消耗资源，自然使高地形改造更贵，也占用更多空间。有限的斜率对单位 AI 也很友好，它们不太适应被要求直上直下。另外，垂直悬崖存在严重的纹理拉伸问题，而且往往会让蜘蛛抓狂。

Shraka 金字塔破坏了精确成本估算的可能性，但这与它解决的所有问题相比微不足道。还有其他单位 AI 问题，有些现在可能依然存在，但它们通常通过瞄准位置调整来解决，例如让单位可以对被墙壁防御的敌人还击。解决这些问题还有一个巧妙的副作用：让地图更加健壮，因为如果单位能处理对手驱动的敌对地形改造，它们也应该能应对合理地图制作者可能抛出的任何东西。

增强有趣地形改造和削弱退化用法主要通过成本调整。第一次成本调整很容易：如果地形改造要易于使用且足够强大值得投入，那它应该消耗金属和能源来反映这一点，其余的设计要求这些成本比例相等。但之后很快就退化成了只有靠感觉才能理解的公式。如果地形改造成本直接与挖掘体积相关就好了，但这是最不可行的方法之一。

破坏地图本质上比开辟地图更容易，这对少于四条腿的东西来说是坏消息。载具受害最严重，所以地形改造成本公式对专门针对载具的地形改造施加了高额惩罚。我们希望看到战略性放置的墙壁和坡道来开辟地图，而不是廉价的不可通行坦克陷阱场。对载具的保护集中在一个特别有害的情况：齐膝高的阻挡载具的墙。这种墙周长很长但体积很小，所以地形改造因大周长而受惩罚。以前还有一个面积成本惩罚，但幸好因为冗余被删除了。

在幕后，地图地形由一个点网格组成，每个点间距 8 elmo（elmo 是距离基本单位）。最小的地形改造方块是 32x32 elmo（2x2 占地单位），对应 25 个独立点。这个 5x5 的点网格有四个角、十二个边缘点和九个内部点。所以如果我们想将这样一个方块升高 100 elmo，成本将是：

25 x 0.0064 x 100 = 16 金属，每个点的体积成本。
12 x 0.75 = 9 金属，十二个边缘点的成本。
4 x 1.05 = 4.2 金属，四个角点的成本。
基础成本 6 金属。
总计 35.2 金属，加上建造过程中产生的任何金字塔，按体积费率 0.0064 金属/elmo 支付。对于 100 elmo 高的方块，平地上的金字塔成本为 24.6 金属，但如果涉及悬崖，那就另当别论了。

地形改造成本计算有点复杂，但能完成任务。看看上面的图片，每面墙花费 250 金属，但最平坦的那面体积最少，而 250 金属与用炮塔覆盖相同距离的成本相当。边缘和角点的成本因子 0.75 和 1.05 也不是静态的，对于修改地形不到 10 elmo 的边缘点会降低。这增强了与现有地形齐平的地形改造，主要是平滑和恢复操作。

周长成本在低于 10 elmo 的变化时降低，因为这是阻挡载具所需的高度。载具阻挡阈值实际上已经变过几次了，对于一个能运行二十年前地图并仍然按制作者意图运行的游戏来说，这很令人惊讶。第一次阈值变化是修复寻路中一些边缘情况的结果。2009 年时指挥官能爬上齐胸高的墙，而载具能穿过现代机器人难以应对的悬崖。这只对突然的地形变化是问题；平滑坡道没问题。这由引擎开发者和 Lua 向导 jK 解决，他还顺便让单位可以走在地形改造墙的顶部。增加的一致性很好，但这意味着低至 6 elmo 的悬崖就能阻挡载具。

载具阻挡阈值被提高到 10，这是 Shraka 金字塔更大改动的一部分。原始金字塔不擅长让单位向坑内射击，特别是像 Scorcher（焦灼者）和 Dagger（匕首）这样的长平板载具。随着 UI 改进使掩埋 mex（金属提取器）变得流行，这让载具很难有效突袭，这就成了一个问题。解决方案是在金字塔最外缘添加一种凸缘或减缓斜坡。载具可以爬上这个斜坡窥视坑内并射击里面的任何东西，作为副作用，它们可以爬上更高的地形改造墙。壕沟型地形改造的内缘也添加了凸缘，为进攻方提供更好的射击角度。这不仅让射击坑中的 mex 变得更容易，还为防守方提供了更多空间来射击 Sky Jack（天降者）。

地形改造不可能在不改进 UI 的情况下得到改善。UI 必须足够好，功能才能好，而且地形改造只有在 UI 足够好值得投入时才被测试和滥用。到现在我们已经有了各种地形改造的热键和预设。Alt+鼠标滚轮可以将建筑放置在任意高度，甚至可以在当前不合适的地形上排队。高度选择 UI 有点繁琐，所以 Alt+V/B/N/M/H 预设创建各种方便高度的地形改造，前两个分别阻挡载具和机器人。Alt+J 挖洞，Alt+G 匹配你首次点击位置的高度。坡道 UI 的默认行为经历了几次重做，甚至有一个区域 mex 模式来掩埋或围住 mex。

持续的测试对地形改造的发展至关重要。玩家需要时间来适应 UI 改进并探索它们开辟的可能性，这反馈到机制中并推动进一步的 UI 改进。机制根据不有趣的发现而改变，而 UI 改变则是为了支持有趣的发现。我们也支持强大且不可避免的用法，即使它们不好玩，因为地形改造不应该涉及与 UI 搏斗。所需的多年迭代可能是 RTS 大多放弃地形改造的原因。一个只有几年大部分是封闭开发经验的设计师处于显著劣势。

设计工作只是一半的故事，因为地形改造没有引擎方面的大量工作根本不可能实现。地图变形本身就不容易，因为单位必须更新其寻路数据以导航任何新地形，所以路径更新速度快到足以考虑大规模地形改造，这令人印象深刻。另一个令人惊叹的项目是 kloot 的视野受限高度图，这意味着你必须实际侦察敌方基地才能看到他们对地图的修改。没有这个地形改造仍然可行，但当你把反核放在坑里或墙后面时全局暴露它们总是感觉不好。他还添加了修改地图纹理的能力，结合 jK 的斜率修复，使得根据谁能在上面行走来为地形改造赋予纹理成为可能。

到现在地形改造系统已经坚如磐石，我还没有看到任何游戏做得更好，至少在快节奏竞技公平性的约束下是这样。Populous: The Beginning 可能接近，但它是一个非常不同的游戏，尽管它确实启发了 Zero-K 的坡道。最后一次重大调整是在 2020 年，添加了金字塔凸缘和禁止在敌方单位附近地形改造。禁止敌方地形改造很有趣，因为它花了很长时间才变得必要。以前有一种反向打地鼠的游戏，Caretaker（看护者）或一群建造者试图困住进犯的坦克。这通过给地形改造高昂的基础成本部分解决了，但没有完全解决，所以我们硬着头皮添加了在敌人附近地形改造的有些随意的限制。

尽管我有种种疑虑，禁止在敌人附近地形改造效果很好。它最终强化了我最喜欢的地形改造部分：没有人拥有它。以前防守方有足够的时间在墙变成累赘时拆除它，但如果敌人已经在你中间，现在就不可能了。玩家必须在战斗前规划并承诺地形改造，这给了对手时间考虑如何利用它。Zero-K 玩家经常花费资源在他们根本不拥有、甚至可能在未来被用来对付他们的东西上，这简直令人惊叹。这类选择是对战略思维的真正考验，这就是为什么如此少的游戏尝试地形改造或任何形式的无人地图改造，尽管存在挑战，真是太遗憾了。

# Cold Take #32 - 眩晕与闪电

策略游戏向玩家呈现不断演变的问题集合以及用来解决问题的工具。RTS 中的问题往往归结为一个事实：敌人有单位，所以许多工具提供了伤害和摧毁单位的方法。但反复使用同一种工具会感觉重复，所以许多游戏还给玩家提供了在不杀死敌人的情况下削弱他们的方法。这些非致命工具被称为状态效果或减益效果，它们通过让玩家将非破坏性行动编织到战斗计划中来增加多样性。

有些游戏不用状态效果，但 Zero-K 的目标需要一个大工具箱，所以忽略状态效果不是选项。第一个状态效果——paralysis（瘫痪）或 stun（眩晕）——存在于我们更早的前身 Total Annihilation（横扫千军）中，尽管它几乎没有被利用。事实上 TA 有两个眩晕来源：Spider（蜘蛛）和一种全球范围的 EMP 导弹。Spider 可以通过持续射击来瘫痪单个目标，而 EMP 导弹瘫痪一个区域内的 Core（核心）单位。这种阵营特定性是因为 Core 是机器人阵营，而 Arm（臂膀）单位是载人的，这使他们对免疫 EMP。但 Core 用杀死 Arm 的 Neutron（中子）导弹进行了反击。

Modder（模组制作者）在 TA 上大展身手，添加了一些更多眩晕单位——但很随意。我怀疑瘫痪是一个小众机制，因为 TA 几乎没告诉玩家这个机制如何运作，它甚至没有指示一个单位是否被眩晕。Spring（Zero-K 背后的引擎）在这方面做得好多了，但到 Complete Annihilation（全面横扫）登场时也没有多太多。Arm 获得了一些 EMP 死亡爆炸和一些眩晕副武器，而 Core 获得了一个类似 Gnat（蚊子）的廉价眩晕无人机。EMP 和 Neutron 导弹的阵营特定效果也在某个时候被移除了。从这里我们扩展了武器库并修复了底层系统的问题。

Complete Annihilation 的早期目标之一是彻底改造武器视觉效果，其中一部分涉及视觉效果和底层机制之间的一致性。这意味着不同的武器应该看起来不同，相似的武器也应该避免通过随意改变视觉效果来暗示机制差异。这对外观朴素的瘫痪激光来说是坏消息，但新想法也突出了花哨的闪电武器的问题，因为它们在机制上不过是脉冲激光。当我们给所有闪电武器添加眩晕能力并将现有眩晕效果改为发射闪电时，这些问题互相解决了。

给 EMP 伤害一个独特的武器类型是个开始，但我们还必须围绕护甲类别的移除来平衡 EMP。一些单位对 EMP 伤害有抗性或免疫，包括指挥官、造成 EMP 伤害的单位和 Strider（步行机甲）。Strider 的 EMP 免疫可能是最过分的，因为瘫痪单个巨大单位是 EMP 的绝佳应用。玩家应该因为围绕他们的工具发挥创造力提出强大策略而获得奖励，而不是撞上随意的限制。大多数东西都可以围绕其平衡，所以移除"太强"的东西是最后手段，因为隐藏的限制惩罚不知情的玩家，并让他们不信任游戏给他们的工具。

闪电武器通过造成在目标上累积的 EMP 伤害来眩晕单位。这种替代伤害类型与生命值并列，显示在主生命值条下方。当单位承受的 EMP 伤害超过其当前生命值时，该单位变为眩晕状态。眩晕完全禁用单位，但效果是暂时的，因为 EMP 伤害会随时间衰减。衰减速率为每秒生命值/40，这就是为什么单位从 100% EMP 降到 0% 需要 40 秒。当前生命值的分数不太好用，所以是时候眩晕一些 Jack（杰克）了。

一个 Jack 有 6000 生命值，这意味着它每秒恢复 150 EMP 伤害。一台承受了 6600 EMP 伤害的 Jack 需要四秒衰减到 6000，所以 6600 EMP 伤害对应四秒眩晕。注意，眩晕一个单位所需的大部分 EMP 伤害是用于达到眩晕阈值的，只有少量需要将眩晕时间推得更高。如果 EMP 伤害可以无限上升，大多数单位最终会有非常长的眩晕时间，这就是为什么每种武器对其目标生命值之上能造成的 EMP 伤害有限制，称为眩晕计时器。Knight（骑士）的眩晕计时器为 1 秒，所以无法将满血的 Jack 推到超过 6150 EMP 伤害。

事实上，Knight 无法对满血的任何东西造成 EMP 伤害，只是因为它们同时造成常规和 EMP 伤害，而常规伤害先应用。旧的闪电武器在获得 EMP 时保留了一些常规伤害。差异通过颜色显示，蓝色闪电造成两种伤害类型，而黄色闪电是纯 EMP。Venom（毒液），TA 蜘蛛坦克的后代，最终获得了少量常规伤害，所以混合了一些蓝色闪电。伤害的应用顺序很重要，因为 EMP 和生命值之间存在交互。

Spring 内置的 EMP 系统有问题。默认眩晕阈值基于最大生命值，而非当前生命值。这工作正常，但让 EMP 和常规伤害完全独立浪费了系统的潜力。有一个甜蜜点，游戏中不同的系统以松散的交互网络存在，绑定不那么紧密使协同效应不会太明显，但又足以让人们找到创造性组合。所以在 2009 年，CA 开发者 SirMaverick 添加了一种使用当前生命值作为眩晕阈值的方法。

使用当前生命值作为眩晕阈值在考虑眩晕单位失去生命值时会发生什么之前运行良好。假设我们有一台满血的 Jack 承受了 6600 EMP 伤害，对应四秒眩晕。然后一些 Glaive（弯刀）在 Jack 恢复之前赶来杀掉它，但它们不用担心，因为伤害 Jack 会导致它眩晕更久。确实，如果 Jack 在四秒内降到 4500 生命值，它仍然有至少 6000 EMP 伤害，给它至少 13 秒的眩晕。这是一个问题：我们想要 EMP 和常规伤害之间有一些协同，而不是 EMP 在第一击后就变得多余。

不断升级的眩晕时间问题通过将 EMP 伤害缩放随生命值变化来解决。考虑一台满血的四秒眩晕 Jack。如果 Jack 突然降到 4500 生命值，它的 EMP 伤害会从 6600 重新缩放到 4950。在 4500 时眩晕以每秒 112.5 衰减，所以 Jack 保持四秒眩晕。这就是为什么混合常规和 EMP 伤害的武器先施加常规伤害：为了避免立即缩放它们的 EMP 伤害。结果是 EMP 和常规伤害之间一种很好的博弈互动。你是在战斗开始时使用眩晕来获得早期优势，还是冒险先等一些伤害再追求更长的眩晕？

技术上 Zero-K 使用最大生命值作为眩晕阈值并对 incoming EMP 伤害应用乘数，但结果在数学上等价，而移动阈值比动态乘数更容易理解。我是在编写了一个从头重新实现 EMP 伤害的 gadget 之后才意识到这一点的，但工作没有白费，因为该 gadget 后来被用来实现 disarm（缴械）。这是一种新的状态效果，工作方式类似 EMP，但允许单位移动和自毁。

Disarm 在 2013 年合并为单一阵营后添加，当时削弱了 Arm 对闪电的垄断。目的是创建一个比眩晕弱的版本，用于完整眩晕太强的情况。它最初被 Thunderbird（雷鸟）和 Racketeer（暴徒）使用，以防止它们被其他方式削弱到底。Disarm 允许移动是因为我们特别想削弱这两个单位对移动目标的效果，而移动的普遍性使 disarm 与 EMP 有意义地区分开来。Disarm 也允许单位自毁，这需要更复杂的解释。

考虑一个不允许自毁的 disarm 版本，想象一个处于效果下的爬行炸弹。要造成伤害，这个炸弹需要跑到敌人中间并被致命弹道击中，弹道由敌人或附近盟友提供。这很繁琐，但对另一方更糟。在其他条件相同的情况下，单位宁愿不爆炸，所以需要避免射击炸弹。然而，如果军队深处有更有价值的东西，那么应该射击炸弹来保护它。是否射击一个被缴械的炸弹可以是战略上重要的决定，但其实现涉及快速切换状态开关或点击小型快速单位。这将导致与 UI 的可怕战斗，所以我们通过让炸弹自爆来移除了这个选择。现在摧毁一颗炸弹只会剥夺它的选项，所以杀掉它几乎不会适得其反。这个额外的许可只需要用于被缴械的炸弹，因为被眩晕的炸弹更容易躲避。

Disarm 充实了我们的状态效果阵容，但有与 EMP 伤害冗余的风险。对于自我强化的机制，当对一种的投资与另一种的投资权衡时，这总是有风险。为了解决这个问题，我们通过让 disarm 伤害"浮在"EMP 伤害之上来使两种伤害类型交互。在这个系统中，当单位的 EMP 和 disarm 伤害之和超过其生命值时，该单位被缴械。所以一台承受 4000 EMP 和 2600 disarm 伤害的 Jack 将被缴械四秒。Disarm 也只在没有 EMP 时衰减。技术上这个系统等同于给所有 EMP 伤害来源添加 disarm，并让两种伤害类型在其他方面独立。

Zero-K 的 disarm 和 EMP 武器可以分为两类：单次能力和持续火力。前者包括爆炸炸弹、一次性导弹和闪电炸弹，持续时间从 16 到 45 秒不等。它们用于禁用反核和发起短暂的保护眩晕单位小游戏。我很喜欢眩晕昂贵东西会围绕临时目标创造迷你游戏，结果决定眩晕该单位花费的金属是否值得。一些一次性眩晕可以重复使用，但有很长的冷却时间，而且使用这种眩晕的单位常常冒着致命风险来施加它们。

持续火力眩晕和缴械由更普通的单位使用，通常也造成常规伤害。这些眩晕持续一到两秒，所以更多充当阻碍而非完全禁用。集中火力实现完全禁用通常是可能的，但代价是承受来自其他来源的更多伤害。持续火力武器的低眩晕时间和高装填时间意味着锁定目标取决于避免眩晕重叠。考虑三台 Knight，有 1 秒眩晕计时器和 2.2 秒装填时间。如果其中一台每 0.73 秒射击一次，目标将被永久眩晕。然而，如果它们同时射击，目标将在每个装填周期享受 1.2 秒的自由。后一种情况感觉像在与 UI 搏斗，因为 Knight 只需要稍微停顿一下就能错开射击。

浪费眩晕时间是一个长期的单位 AI 问题。过量预防对 Racketeer 做得很好，但我们刚刚在上周发布的 1.13.9.0 更新中开始尝试更通用的解决方案。当前方法不是添加更多单位 AI，而是通过减少重叠眩晕的成本来从系统中设计掉愚蠢。新机制叫 overstun（过度眩晕），允许单位造成比其眩晕计时器允许的稍多一点伤害，但前提是目标已经接近眩晕计时器。例如，Knight 有一秒眩晕和一秒过度眩晕，所以它可以给任何最多一秒眩晕持续时间的东西增加整整一秒眩晕。例如，Knight 射击一台承受 6100 EMP 伤害的 Jack，将能够把 Jack 推到 6250 EMP 伤害。这是对持续火力眩晕武器的增强，但应该让我们比围绕愚蠢平衡或试图用单位 AI 解决它走向更好的方向。

完全眩晕有通过剥夺玩家对单位的控制来使他们沮丧的风险。这通过使大多数眩晕短暂、精确或部分来缓解。Slow damage（减速伤害）在减轻我们状态效果的锁定负担方面发挥了作用，因为它永远无法完全禁用单位。事实上，有 12 种减速伤害来源，不包括指挥官，而 EMP 有 11 种，disarm 有 4 种。然而，对减速的探讨必须等待，因为这篇文章已经快装不下了。我勉强挤进了 disarm，因为它的工作方式与 EMP 相同。Slow damage 像 EMP 伤害，只是几乎在每方面都不同。

# Cold Take #33 - 给它减速

减速伤害是 EMP 和 disarm 更多产但不那么戏剧性的表亲。它比 EMP 弱，因为它永远不会完全禁用单位，但效果从第一次伤害开始就逐渐累积，使其在更多情况下有用。这种渐进而持久的效果让我们能够将减速分散到更多武器类型和角色中，直到它变成了一个梗。但 EMP 是从前 Complete Annihilation（全面横扫）模组继承的，而减速是从零创造的，回顾历史会有些惊讶它居然存在。

上次我们介绍了 EMP 的发展，以及它如何与闪电同义。但我没有提到在 Total Annihilation（横扫千军）中，眩晕和闪电都是 Arm（臂膀）专属的，这种侧重主要被 TA 模组制作者继承。Complete Annihilation 推动了阵营差异化，所以当我们合并闪电与眩晕时，少量的 Core（核心）眩晕被移到了 Arm。这让 Arm 垄断了禁用状态效果，结果证明有点太一边倒了。尽管 Core 有一些更多伤害选项，比如用来镜像 Arm 战术 EMP 导弹的战术核导弹，我们最终还是寻找方法打破垄断。

减速伤害是解决 Arm 对状态效果垄断的方案：一种专门为 Core 创建的新状态效果。我记得 MidKnight 添加它是为了替代将主要 Core 眩晕单位（Gnat 等价物）移到 Arm。Gnat 最终获得了闪电并被转移，但减速保留了下来。多年来众多 Core 单位获得了减速伤害，直到这个机制在两栖工厂（Amphbot factory）中找到了归宿。这个工厂在合并阵营之后添加，但在许多方面保持了旧 Core 原始力量和韧性的美学。

减速伤害的底层机制在过去 16 年中相当稳定。减速伤害在单位上累积，显示在其主生命值条下方的条上，与 EMP 相同。具有减速伤害的单位会立即被减缓，减缓量等于累积的减速伤害占当前生命值的比例。例如，一个有 1000 生命值和 150 减速伤害的单位被减缓 15%。减速伤害上限为当前生命值的 50%（少数例外），并以每秒 4% 的速率随时间衰减，尽管受到减速伤害后衰减会暂停半秒。

一些减速武器（Moderator（调解者）、Limpet（附着物）和 Disco Rave Party（迪斯科狂欢派对））可以将单位推到 58% 减速伤害，但仍然只减速 50%。减速以每秒 4% 衰减，所以这对应满减速状态下额外两秒。但这不是延长残留减速的唯一方式，因为常规伤害减少生命值，这会增加减速占当前生命值的比例。这种状态效果百分比与当前生命值之间的交互对 EMP 来说是巨大问题，但减速衰减太快以至于很少有影响。

减速推动并受益于越来越强大的 gadget 的发展。没有明确规定什么可以被减速，所以我们通过让更多东西可减速逐渐扩展了其范围。到现在几乎所有东西都被覆盖了，包括移动速度、装填时间和经济速率，主要例外是跳跃速度。这对 modding 来说很棒，并展示了引擎的力量，因为减速在一个更新频率低于每秒 30 次的引擎上会更难实现。高 tick 率让我们能够将时间视为连续的而不会出太多问题，即使操作只每 33 毫秒发生一次。

在减速快速射击武器时需要注意一些事项。例如，Light Laser Tower（轻型激光塔）看起来像在发射连续激光，但它实际上每 0.1 秒射击一个 0.1 秒的连射。这对应每 3 帧一发，当减速 50% 时增加到每 6 帧一发，正如预期。然而，当减速 45% 时，它每 4 帧射击一次，造成 40% 的伤害输出减少而非 45%。这对 Zero-K 使用减速的方式几乎不重要，但对一个例如实现全局射速升级并期望伤害输出绝对一致增加的 mod 来说会很重要。

减速伤害的首批接受者是 Outlaw（法外之徒）和 Moderator。Outlaw 曾是标准的近战单位，一手拿着 Bandit（土匪）爆破枪，另一手拿着 Ripper（撕裂者）枪（当时叫 Leveler（推土者）），这就是"Leveler 副武器"短语的由来。它获得了干扰脉冲，而其他 Core T1 机器人正在进行护盾改造，因为没有友军火力的范围伤害在护盾球中效果很好。干扰脉冲的独特之处在于它不会伤害友军单位，被构想为某种控制系统攻击。

Moderator 获得减速是因为旧版 Moderator 太普通了。这使它容易被单一生产，而且被四面八方的类似单位围堵。想象一个像 Rogue（流氓）一样的远程骚扰单位，带着类似 Sling（投石器）的等离子炮和 Recluse（隐士）的重量。射程在 RTS 中特别难平衡，尤其是对能在移动时射击的单位，所以在 Zero-K 中它带有严重缺点；Recluse 和 Sling 很脆弱，Bulkhead（舱壁）和 Fencer（击剑手）部署后射击，而 Rogue 有非常慢的弹道。Moderator 周围的设计空间太满或太麻烦，找不到一个健康的位置，所以我们给了它减速。

一些单位因与 Moderator 类似的原因获得了减速，以至于"给它减速"成了一个梗。这被用于那些表现平平或缺乏独特角色的单位。Harpy（鹰身女妖）骚扰机在更快的 Locust（蝗虫）和更坚韧的 Revenant（亡魂）之间处于尴尬位置，而 Dart（飞镖）在 Rover（漫游者）中缺乏明确目的。Cyclops（独眼巨人）曾经有一个火焰喷射器副武器，但这使它难以用蜂群对抗，所以被换成减速光束以使其比 Minotaur（牛头人）更有实用性。Seawolf（海狼）在海洋重做中获得了一枚减速鱼雷。

减速伤害也是全新单位的热门选择。两栖工厂在工厂取代阵营后创建，所以它可能包含任何东西，但我们决定将减速作为其主要机制之一。在某种程度上，Buoy（浮标）造成减速伤害是因为它的同级别普通骚扰单位已经没有空间了，而 Limpet 爆炸造成减速伤害是因为其他炸弹类型已被占用。但我们有更好的理由给工厂减速。计划的一部分是制作一个特别慢速的工厂，可以利用减速伤害将敌人降速到自己的水平。Limpet 大型干扰脉冲背后的理念是让它从海底到达水面。

Buoy 造成减速伤害的最强理由是减速对重型单位有效，而船往往是重型的。重型单位容易受到减速影响，因为它们往往存活更久，给了效果更多时间发挥作用。减速也使撤退修理更困难，这是使用重型单位的重要部分。Bolas（流星锤），干扰光束气垫船突袭者，被添加到 Hovercraft（气垫船）工厂也是出于类似原因，尽管无可否认它也造成减速是因为没有其他合适的突袭者这样做。

我们避免添加过多减速，但它在多年来不断累积，没有真正考虑它在游戏世界中是什么。是时间膨胀，还是像 Outlaw 暗示的某种病毒？单位是否被某种紫色粘液机械性地粘住了？减速伤害由各种类型的导弹、光束和弧形弹道造成，唯一的统一特征是紫色。与闪电和 EMP 的一致性相比，它有点混乱。这可能是个小问题，但系统以一致的方式呈现通常感觉更好。

在 12 种减速伤害来源中，只有一种——战术导弹——造成纯减速。这不是不想尝试，因为最初的减速 Moderator 是纯减速单位。然而，我们未能为它找到一个既不太小众也不太烦人的位置。这个问题的一部分是纯状态效果武器有堵塞屏幕和声音空间而不推进游戏的风险。Racketeer（暴徒），缴械炮兵，就遭受这个问题，但 disarm 比减速更大的实用性让它感觉强大而无需 spam 弹道。纯状态效果也给单位更多犯蠢的方式，因为传播效果变得更核心于该单位的实用性。

如今大多数减速伤害武器将减速作为附带效果，而非主要目的。Outlaw、Moderator、Buoy 和 Constable（治安官）没有减速会差得多，但相应的成本降低后可能仍会有用。尝试更多纯减速会很有趣，但目前，减速为重型单位提供了急需的反制手段并缓解了一些伤害。更专门的减速来源，比如大范围的，相当小众，但这可能比永远以半速玩游戏更好。

# Cold Take #34 - 建造海军

这是一篇由 Aquanim 撰写的客座文章，他领导了 2016 年的海洋重做。

我们自己星球超过三分之二的面积被海洋覆盖。因此，在行星景观上进行的战略游戏有理由考虑如何处理水体。最简单的方法是将水体视为陆地单位不可逾越的障碍，仅此而已。在单位可以与水互动的游戏中，这通常保留给特殊类别的舰船型单位，但除此之外"水"只是另一种地形类型。

Zero-K 的物理驱动机制要求我们更深入地思考单位在水上和水下应有的行为。这一直深入到基本问题，如"水在哪里"、"单位能去哪里"、"单位何时能看到彼此"以及"单位如何相互战斗"。我们可以快速解决第一个点：海平面被引擎定义为海拔零，所以不幸的是我们无法有华丽的瀑布或风景如画的山间湖泊。

在陆地上，所有单位（非飞行、跳跃或被投掷的）都站在地面上。地面可能是平坦的、陡峭的或介于两者之间的任何情况。在水中，另一方面，单位可能合理地处于四个位置：

- 悬浮在水面之上
- 在水面上
- 浮在水面之下
- 在海底

这已经相当复杂了，所以 Zero-K 立即通过不区分前两种状态来缩减这个列表。这一选择的主要后果是 Hovercraft（气垫船）对鱼雷和类似武器变得脆弱。气垫船、水面舰艇和浮出水面的两栖机器人行为类似于陆地单位——显著的区别是所有水体都是平坦的。这限制了单位互动和地图设计的潜在范围，并使直射武器如 Lance（长矛）占优，因为没有地形可以躲藏。

原则上，浮在水下的潜艇型单位可以在三维空间移动，上升到水面或下降到其深度极限。再说一次，这听起来相当复杂，我们不希望玩家与二维界面搏斗来明确命令三维移动。因此，我们假设潜艇在水面下固定深度移动，除非被迫在浅水中浮出水面。在撰写本文时，这是 Zero-K 中最罕见的移动模式，仅由 Seawolf（海狼）和 Scylla（塞壬）共享。

更广泛的单位（统称为两栖的）可以在海底行走：指挥官、所有两栖机器人、一些工具单位和几台步行机甲。根据单位大小和水深，两栖单位可能部分露出水面。事实上，一些两栖单位能够自行浮到水面。这使其水面武器可以使用，但阻止进一步的（自愿）水平移动，直到它们返回海底。我们容忍这种复杂性，因为大量纯水下作战单位会带来更大的设计问题，我们稍后会讨论。一般来说，这些单位只在攻击时才想要浮出水面，所以与界面的搏斗还不算太糟。

所有两栖单位都是机器人，不是载具或蜘蛛，所以它们能爬中等坡度但不能爬悬崖。管理单位被允许在水下游走的地方听起来相当烦人，所以我们通过避免载具移动类型和在没有水下悬崖的地图上游戏来简化情况。这仍然是我们三种水上移动类型中最麻烦的，因为水深会显著影响战斗——特别是，它决定了两栖单位需要多长时间浮出水面并开始还击。

真实感要求水下的单位比水面上的更难看见。然而，就像普通战争迷雾机制偏离"现实"视线以服务于游戏性一样，我们将水下观察简化为一个机制：声纳。敌方水下单位只有在友方单位声纳半径内才对我们可见。为了更简单，在现代 Zero-K 中，所有"海上单位"的声纳半径等于其视野半径，而大多数其他单位不是（值得注意的是 Owl（猫头鹰）侦察机有有限的声纳能力）。

水下单位仍然比水面单位更隐蔽，因为它们不在远程雷达上显示，也无法被金属提取器等非战斗建筑探测到。它们不如隐形单位隐藏得好，这似乎合理，因为隐形单位更少见并且通常在能量或静止方面付出代价。如前所述，水下单位无法隐形，因为解除隐形半径机制与可变水深配合不好。

除了与视野相关的游戏机制外，海洋地图还给玩家带来视觉清晰度问题。在海水色调、水面涟漪和反射、在多个不同深度运作的单位以及通常信息量较少的地形纹理之间，往往更难确切知道水下正在发生什么。这是 Zero-K 水下战斗武器类型较少且现有武器不强烈奖励快速反应和微操的部分原因。

水下单位可用和对抗它们的武器的第二个约束是游戏机制必须在相当宽范围的水深内保持合理。例如，想象一个水面单位用一枚非制导鱼雷，类似 Ronin（浪人）导弹的机制，射击一组在海底的 Duck（鸭子）。在浅水中，鱼雷路径是一条穿过鸭子的直线，很可能击中一个。在深水中，鱼雷只有很短的窗口处于适合击中鸭子的深度，然后就撞地了。这一切听起来相当混乱，所以我们给了所有水下可用武器跟踪或其他弹道属性来避免这种差异。

从平衡而非设计的角度来看，水下可用武器还有另一个约束：如果最远射程的传统水下瞄准武器本身附属于水下单位，那么这些单位很可能在海上对战中占据主导，因为如果它们能抵御蜂群，几乎没有其他东西能超越它们。通过扩展水面与水下对抗的游戏来解决这个问题，添加大量单位和能力，表面上是个有趣的想法，但有两个大缺点。首先，正如我们已经讨论过的，合理的水下可用武器的潜在多样性相当有限。其次，虽然水面单位可以以有趣的方式直接与陆地单位互动，反之亦然，但让陆地单位和水下单位良好互动似乎困难得多。

因此，在 Zero-K 中我们走向了将水下互动减少到最小可行集的方向——在步行机甲级别以下的单位中，只有轻型潜艇和少数两栖机器人能在水下战斗，而且它们的武器都是短程的。旧的 Serpent（巨蟒）远程狙击潜艇是过去十年中极少数被从 Zero-K 中彻底移除的单位之一（我们稍后会提到第二个），Scallop（扇贝）的骚扰范围鱼雷被缩减到近战范围（尽管这绝不是 Scallop 作为游戏平衡头疼问题的终结……）

由于很少有陆地资产能与水下单位互动，水下单位可以自由攻击陆地单位而不担心报复是相当不公平的。因此，唯一拥有设计用于水下对陆地战斗武器的单位是 Scylla 战术导弹潜艇，它为每枚导弹支付金属成本，因此并非"自由"交战。

我们以一场快速讨论来结束 Zero-K 海洋设计的其他有趣方面。

经济：Zero-K 的经济核心原则在海洋地图上保持不变，但有一些差异改变了典型的游戏模式（增加了一些令人期望的多样性）。可以说最重要的是单位能去哪里的间接后果。被摧毁的单位残骸沉入海底，在那里它们不太容易被践踏或被流弹削减。因此，金属回收往往是长期海战的重要因素。由于风力发电机在海平面表现不佳，它们被潮汐发电机取代，后者结合了风力发电机的效率和太阳能收集器的可靠性和稳健性。Singularity Reactor（奇点反应堆）无法在水下建造，但可以地形改造一个陆地平台来放置它们。这背后的动机与阻止建筑隐形的决策类似，并给 Fusion Reactor（聚变反应堆）一个差异点。

工厂平衡：在陆地工厂之间有一个（松散的）原则，即每个工厂在适合的地图上 1v1 中应该同样强大和有竞争力。我们在海战中不坚持同样的规则，原因有二。首先，做两次相同的事会很无聊。其次，由于无法在陆地上移动，Ship（舰船）单位比混合 Hover（悬浮）和 Amph（两栖）工厂的移动自由度更小，即使在只有小岛屿的地图上也是如此。因此，我们的（松散的）目标是 Ship 单位稍微过度调整，但 Hover、两栖和空军工厂仍然应该提供有用的工具。

陆上舰船：好吧，我撒谎了。舰船可以去陆地……某种程度上。Lobster（龙虾）、空运或地形改造可以将舰船放到陆地上，虽然舰船无法在没有帮助的情况下移动，但仍然可以战斗。Envoy（使节）和 Shogun（将军）炮舰最能利用这一点，因为它们的远射程缓解了移动问题，弹道武器受益于更高的海拔。

运输：与几乎所有其他有专门"舰船"类别的游戏不同，Zero-K 没有传统的跨水体陆地单位运输工具。Surfboard（冲浪板）曾经担任这个角色，但与 Djinn（精灵）传送器和 Charon（卡戎）空运相比没什么用处，最终与 Serpent 一起被送入舰船墓地。

酸性水域：一些地图用水替代为酸液、熔岩或虚空。在这些地图上 Zero-K 的表现更像我们开头提到的"水是不可逾越障碍"的游戏，尽管一些酸液弱到单位可以在其中存活一段时间。这些地图不受本文描述的大多数问题影响，但它们确实提供了一些新颖的战术机会……

# Cold Take #35 - 护盾作为球形外壳

护盾发生器是各种媒介中科幻作品的支柱，RTS 也不例外。游戏倾向于将护盾作为额外类型的血量，一种往往有某些优势或至少更容易补充的血量。它们也构成了很棒的视觉效果。所以奇怪的是 Total Annihilation（横扫千军）没有护盾，而且可以理解它衍生的几乎每个游戏都包含它们。甚至 TA 模组制作者也参与其中，尽管据报道护盾相当卡顿，因为它们复用了核弹拦截系统。Zero-K 在近战和防御远程炮击中都使用护盾。

非常老的玩家，或者对 BAR 有所了解的人，可能想知道为什么 Zero-K 的护盾是硬壳，而它们本可以把弹道到处弹飞。这对其他人来说是个奇怪的具体问题，但考虑到 Spring 底层引擎的历史就说得通了。默认护盾是排斥护盾，虽然它们不太可能回归，但它们的移除是一系列定义 Zero-K 设计的原则的绝佳测试台。

护盾支持于 2005 年由 SJ 添加到 Spring，当时引擎是实验性的游乐场，Lua 脚本尚未添加。为了不被 Complete Annihilation（全面横扫）未来的疯狂超越，这些护盾排斥弹道，Zero-K 的硬壳版本只是后来才加入的。排斥护盾被 TA 衍生模组采用，主要作为防御后期 Big Bertha（大贝莎）spam，20 年后这仍然是它们在 BAR 中的角色。但 Complete Annihilation，以及后来的 Zero-K，非常致力于将机制推向极致，而伤害缓解打开了太多可能性，不能让它在游戏角落里枯萎。此外，狭窄的后期关键角色已经被核弹和反核完美填补，所以我们着手将护盾集成到游戏的其余部分。我们成功了，但不得不放弃排斥。

排斥护盾很有趣，失去它们很遗憾，但最终它们是不可行的。首先，阻挡等离子炮但让其他一切通过感觉太随意了。考虑 Glaive（弯刀）和 Bandit（土匪）：两者都是轻型突袭者，但一个射击等离子，另一个有激光爆破枪。这导致等离子排斥理论上克制 Glaive 而非 Bandit，但这在之前的游戏中没问题，因为突袭者在护盾出现时已经过时了。我们可以围绕这种克制关系设计，C&C: Generals（命令与征服：将军）中的导弹防御激光就是一个例子，但这会施加很多设计限制。到此时我们已经有了足够多的限制和在这些限制下设计的多样化武器，所以添加一个硬克制会摧毁我们已经建立的大部分东西。结果是我们的新护盾必须与所有类型的武器交互。

引擎支持非等离子武器的排斥，但行为不一致。激光直接弹回，而火箭和导弹使用它们通常的转向半径被弹开。自导弹甚至在离开护盾后恢复制导，这看起来有点荒谬，而且可能使护盾变成累赘。但更普遍的问题是成本：排斥护盾随时间将弹道推走，只要弹道在护盾内就消耗护盾电荷。这很酷，因为它让不堪重负的护盾逐渐失效，但它严重干扰了任何平衡消耗率的尝试。弹道的速度相当重要，它击中护盾的角度也很重要，因为擦边撞击需要更少的力来偏转。特别快的弹道，如近战炮或战术导弹，可以在被显著偏转之前深入穿透护盾。激光的电荷消耗是合理的，因为它们是瞬时反射的，但攻击者仍然可以排列射击来同时从多个护盾吸取电荷。

也许这些问题是可以解决的，通过足够多的调整，我们可以找到平衡排斥所有类型弹道成本的方法。但排斥的致命缺陷是它给单位太多犯蠢的方式。看到弹道飞来飞去无疑很酷，但它们大多飞回发射它们的军队。一种解决方案可能是让单位稍微向上瞄准，这样弹道返回时越过头顶，但这有两个问题。首先，它不是一个万无一失的默认值，因为消耗护盾的最好方法通常是直接瞄准它，所以是否冒着友军火力的风险变成了额外的战略决策。其次，我们已经极力避免让单位向天空射击，允许这样做会打开一整罐新的麻烦。

最终，有趣的物理学不得不被搁置，护盾变成了阻挡敌方弹道的硬壳。弹道通过在空中引爆被阻挡，这样做消耗等于弹道伤害的护盾电荷。如果护盾在撞击时电荷不足，弹道就被放行。护盾随时间恢复电荷，并在附近护盾之间共享电荷以尝试均衡。这是一个相当简单的系统，但有足够的内容来调出令人满意的互动。

一个重要的技术细节是弹道在护盾上爆炸，这甚至可以伤害护盾后面的单位。必须这样是因为阻挡爆炸的范围伤害（AoE）在计算和直觉上都是噩梦。一个阻挡爆炸伤害的护盾也必须阻挡击中护盾外地面的射击伤害，否则单位会被激励向地面而非护盾射击，这会看起来很傻。但伤害不应该被免费阻挡，这意味着附近的爆炸应该消耗护盾电荷。多少电荷？嗯，爆炸伤害随距离减少，但护盾有可能堆叠在彼此后面，所以护盾必须能够互相保护。这因为单位可以跨越护盾边界而进一步复杂化，使被保护的对象不明确。所以与其阻挡范围伤害，我们不如接受 AoE 对护盾有效，特别是较小的个人护盾。

护盾机制的另一个值得注意的部分是拦截阈值是按弹道计算的。所以一个 599 电荷的护盾会放行 600 伤害的弹道，但会阻挡两发各 300 伤害射击的一半伤害。爆发伤害单位围绕这个事实平衡，一个极端例子是 3000 伤害的 Lance（长矛），它在一秒内以 150 的块造成伤害。Phantom（幻影），有单发 1500 伤害的子弹，通常因此对护盾更好。我们甚至利用爆发伤害来增强单位对护盾的效果，例如当 Firewalker（火焰行者）的双管射击被分成十发弹道，每发减少 AoE 但增加总直接伤害。爆发伤害和护盾的交互是涌现行为中较好的一种：玩家可以合理地凭直觉推理出来，而不需要那种深入了解排斥护盾如何最佳消耗所需的深入研究。

Zero-K 护盾最不寻常的特征叫做 shield link（护盾链接），它使一组护盾逐渐均衡其电荷。它很可能是为了解决浪费恢复的问题，因为护盾只有在未满电荷时才恢复。在与护盾战斗时最小化战斗中的恢复很重要，可以通过集中火力实现。然而，机动护盾发生器可以通过跳舞让每个护盾轮流在前面来抵消集中火力。这是通过实现最终结果来跳过单位 AI 军备竞赛的一个案例，所以高电荷护盾的电荷流向附近较低电荷的护盾，为恢复留出更多空间。

Shield link 结果成为对爆发伤害的增强，因为它导致成堆的护盾一起失去电荷，而不是单独失去。最好的证明是战术核弹对抗护盾的方式。战术核弹造成 3500 伤害，而大型护盾有 3600 电荷，所以没有 shield link，你需要为每面护盾花费一枚战术核弹来消耗一组护盾足够开始穿透。有了 shield link，一枚战术核弹可以禁用任何合理大小的护盾组。你只需要发射第一枚战术核弹，等几秒，然后发射后续弹。第一枚战术核弹命中后电荷涌入新耗尽的护盾，所以当第二波到来时没有护盾能阻止战术核弹。

护盾不太想被战术核弹利用，但正如我们在 25 篇文章前看到的，shield link 是可选的。Shield link 的缺点给护盾空间在数量少时强大，但在 spam 时不会变得那么强大。这是我们在 Zero-K 中追求的甜蜜点，因为它鼓励人们使用混合工具，对于可以通过阻止伤害来减缓游戏的机制来说尤其重要。目标是在许多战斗中看到一点护盾，而不是在少数战斗中看到很多。这一直追溯到将机动护盾和隐形装置作为所有工厂静态版本的变形可用的决定。

在技术方面，shield link 经历了两次迭代。它最初是一个确定性算法，每个护盾轮询其邻居并向平均值靠近，对每个邻居调整相同的量。这有性能问题和边缘情况，所以被随机采样方法取代。现在 shield link 的工作方式是每个护盾每秒 15 次选择一个随机邻居来交换电荷。这足够快，使随机变化趋于平均，而且比以前的方法简单得多。

今天的最后一个问题是关于状态效果。武器应该通常与护盾交互，从主题上说闪电忽略护盾或被无害消散会很奇怪。但让护盾像完全独立单位一样承受 EMP、disarm 或减速伤害会太复杂。所以为简单起见，状态效果对护盾电荷造成伤害，就像普通伤害一样，但有注意事项。状态效果的伤害数字比普通伤害高得多，因为状态效果伤害必须命中生命阈值才能生效。它们的原始伤害会完全摧毁护盾，所以状态效果只对护盾造成三分之一的伤害。这是 Zero-K 中伤害乘数的罕见实例之一，但它仍然足以使闪电和减速伤害成为消耗护盾最有效的方法之一。

护盾最终在多年来被削弱后处于良好位置，现在是一个强大但可选的工具。它们在跨地图炮战中的作用比以前小了，但这主要是因为地形改造提供了替代的防御方式和 Odin（奥丁）轰炸机压倒护盾的出现。护盾是稳定的，尽管在整篇文章中我试图但未能说服自己重新审视排斥护盾。也许这就是 modding 的用武之地，肯定有一个关于到处弹射弹道的全新游戏。最后说明一点，Funnelweb（漏斗蛛）护盾的行为与其他护盾不同，但细节必须等待另一篇文章。

这是 2025 年的最后一篇 Cold Take。我们将在十二月休息，预计一月回归。


# Cold Take #36 - 界定战略三角形
日期：2026-01-31

战略自由是一种脆弱的东西，然而制定自己计划的感觉是许多游戏试图灌输的。问题在于，随着玩家探索和完善策略，可行策略的空间可能会缩小到几种选择。这通常被视为竞技多人游戏的问题，但单人游戏也并非免疫。任何提供战略探索和精通机会的游戏，都存在暴露自己只有单一最佳策略的风险，而当挑战变得足够大时，玩家就会诉诸于此。这个问题永远不会被完全解决，只能加以管理，但管理本身仍有很多文章可做。

这篇文章讨论一些有助于解决上述问题的概念：战略 RPS（rock-paper-scissors，石头剪刀布）和战略三角形。

战略 RPS 是一种简单的策略模型，将决策简化为三种方式之一：进攻、防御和经济。这些方式形成了一种天然的克制关系，其中进攻克制经济，经济克制防御，防御克制进攻。许多玩家对战略 RPS 有直觉性的理解，因为这是一个常识——防御性打法会被投入经济建设并最终压倒防线的策略所击败。

战略 RPS 的力量在于其通用性，至少如果我们足够宽泛地定义它的话。这需要简要绕道讨论一下经济的概念。RTS（即时战略游戏）的标准经济涉及开采矿石、金属或矿物等资源，然后将其花费在单位和更多经济建设上。但战略 RPS 中的经济部分涵盖了更多内容。首先，将建造能力（Zero-K 中称为 buildpower）视为一种本地的、不可囤积的资源是很有用的。我们可以更进一步，因为任何事物的成本都可以转化为组装其前置条件所需的时间，这暗示着行军时间也可以被视为一种成本。许多游戏都以最一般的意义拥有资源，但策略游戏的独特之处在于能够做出大幅提高资源收入的选择。

我们将增加收入的决策称为"经济投资"，这个术语涵盖了所有以牺牲当前实力换取未来更大实力的方式。这包括建造金属提取器和发电厂，也包括研究和提升科技等级等行动。升级后的单位是更高效的单位，因此升级大致等同于产生只能用于单位生产的额外收入。局部行动也可以是经济投资，比如将建筑工移动到更好的位置。建筑工用它现在本可以进行的工作换取将来在更有利位置工作的能力。在远离工厂的地方"生产"一个建筑工需要更多时间。

值得仔细看看战略 RPS 三方面之间的关系。在这样做时，我想证明战略 RPS 是策略游戏的内在属性。一旦你有了经济投资，你就有了战略 RPS，而要淡化或消除它需要额外的工作。这一切都取决于经济投资，所以我们从那里开始。

进攻克制经济，因为经济投资使投资者容易受到攻击。RTS 需要进攻，因为否则就没有任何东西能阻止玩家将所有收入投入越来越多的经济建设。严格来说，一个完全没有进攻的游戏将缺乏胜利条件，因为即使是最注重经济的策略也需要将其压倒性的资源优势转化为杀死敌人的手段。这就是经济克制防御的方式：如果双方一段时间内互不干扰，经济更强的一方将压倒另一方。在战略 RPS 的经济互动关系确定之后，进攻和防御之间的关系只能朝一个方向发展。

"最好的防御就是进攻"，这句格言是这样说的，但在战略 RPS 中并非如此。进攻已经克制经济了，所以为了进攻不至于成为主导策略，它最好被防御所克制。RTS 的战略深度要求存在用资源进行防御性投资的方式，而且防御必须足够强大以抵御更大的进攻投入。请注意，这并不意味着所有游戏都需要炮塔。生产以防御为导向的单位算作防御，以更防御性的方式建造（以牺牲一些经济为代价）也算作防御。Starcraft（星际争霸）提供了一个坡道堵口的例子，因为当建筑建造得更远时，工人到达建筑需要更长的时间。

战略 RPS 是一个不错的模型，但它有点不切实际，因为玩家很少会全押进攻、防御或经济。以前我们会用这个术语作为相关概念的简写。为了我们的目的，我们最好把这个术语搁置在这里，开始讨论战略三角形。

战略三角形是一张策略空间地图，策略空间可以被想象为一团模糊的云，代表玩家在游戏中可以采取的每种方式。战略三角形通过评估每种方式的进攻性、防御性和经济性来添加结构。然后空间被拉伸和压扁成三角形，使得最极端的策略位于三角形的顶点，而其他一切存在于中间某处。为了简化，我们也丢弃了无意义的策略，比如在游戏开始时炸掉自己的指挥官。如果你想技术性地描述，可以将三角形视为策略空间投影到三个维度上的帕累托前沿。

换句话说，战略三角形是战略 RPS 的精细化版本。玩家可以混合进攻、防御和经济。战略 RPS 的关系仍然成立：我们预期一个主要注重经济的策略会输给一个更注重进攻的策略。

关于战略三角形首先要注意到的是它本质上是不平衡的。还记得"最好的防御就是进攻"吗？这在迂回的方式下确实是个好建议。最常见的 RTS 策略建议通常归结为：尽可能多地建设经济，同时只投入刚好足够的防御。进攻通常是让对手陷入困境并诱使他们建造超过需要的防御、减少他们在进攻和经济上投入的有效手段，因此有了这句格言。经济是获得长期优势的最佳方式，而经济克制防御，所以谁建造的防御最多谁就输了。RTS 中的这种偏向可能源于与其他策略游戏类型相比，对军队和快速战斗的强调。

战略三角形帮助我们创造战略自由感，因为作为策略空间地图，它的大小反映了玩家可用选项的范围。但它的主要目的是突出并避免战略 RPS。你可能已经从它略带贬义的名字中猜到了，当战略 RPS 变得太明显时，玩家往往会抱怨。一个令人满意的游戏涉及在战略三角形上争夺位置，通过侦察和直觉来试图对抗敌人的行动。

这里存在一个矛盾：战略自由需要大的策略空间，而对过多战略 RPS 的反感要求它保持较小。Zero-K 和许多其他游戏采用的解决方案相当简单：以一个小三角形开始战斗，随着比赛进行而增大它。这就是为什么游戏早期选项相对较少，且克制关系相对软。只有到后来才会出现硬克制手段，比如核弹和轰炸机。但我们如何调整策略空间的大小，或者换句话说，我们如何界定战略三角形呢？

界定战略三角形的技巧在于认识到大多数行动都实现了进攻、防御和经济增长的混合。例如，如果我们想防止玩家将超过 60% 的资源花在进攻上，我们只需要避免给他们任何"进攻性超过 60%"的选项。这些数字只是假设的，记住三角形是概念性的而非可计算的，所以是时候举些例子了。

以游戏中的第一个单位——指挥官为例。它的激光和高生命值为玩家提供了基线水平的防御，而它的收入和建造能力构成了经济。这将玩家置于三角形的防御-经济区域，而他们的决策能将他们推向其他方向的速度是有限的，这限制了玩家全力投入进攻的程度。我们曾经尝试过让玩家以普通建筑工开始，但缺乏指挥官作为"战略压舱石"导致了过多的战略 RPS。

这里是另一个例子：建筑工可以建造炮塔，这使得派遣建筑工出去建造金属提取器成为一种部分防御性的行动。建筑工甚至在足够靠近敌方基地时可能承担进攻角色。所以即使是最贪婪的经济开局也内置了防御。想象一下，如果 Zero-K 拥有专门的经济建筑工和防御建筑工会感觉多么摇摆不定。金属提取器的成本也随时间增加，限制了贪婪开局能多快地弥补其弱点。

大多数单位执行所有三种角色。一个清理潜在扩张区域中突袭者的暴兵单位正在执行经济任务。或者它可以防御现有的金属提取器，或者缓慢推进到敌方扩张点。对资源最通用的看法是将单位视为被部署在地图不同位置的资源，因为它们一次只能在一个地方。这种部署灵活性平滑了许多潜在的过度战略 RPS，因为这意味着决策不仅仅在单位建造时才做出。

我们界定战略三角形的尝试经常与其他设计原则发生冲突。原子化单位设计防止单个单位执行太多不同的角色，而 Quant's Rule（量化法则）总是将一切推向三角形的边缘，在那里战略 RPS 最强。我们曾考虑过给工厂内置炮塔来增强其固有防御，但最终觉得这样的东西会感觉太随意。类似地，我们避免添加武装金属提取器，因为我们想保留纯经济扩张的可行性——至少在建筑工离开之后——不带任何防御成分。

战略三角形对于思考游戏的早期阶段最有用。这没问题，因为每个游戏都有开头，而这部分游戏往往是最难做好的。然而，这确实限制了它在大型团队游戏中的适用性，因为这些游戏很快变得太复杂而无法分析。这些概念具有广泛的适用性，例如，一个占领地盘并投资经济的团队会击败一个没有这样做的团队，但这些很少是个别玩家的决策。幸运的是，这些游戏很少有战略 RPS 的问题，所以战略三角形可以专注于做它最擅长的事。

# Cold Take #37 - 侦察难题
日期：2026-02-21

大多数策略游戏都有侦察敌人的工具，通常需要消耗资源。这被称为侦察，它是一个棘手的设计课题，因为其平衡性取决于信息的有用程度。如果你已经知道敌人的计划，或者知道他们的计划也不会改变你的应对方式，那么侦察就是浪费资源。在另一个极端，拥有大策略空间的游戏依赖强力的侦察来遏制战略 RPS。更复杂的是，侦察的力量随着游戏进程而变化，因为策略空间在扩大。

侦察始终在与本可用于建设经济或部署更大军队的资源竞争。战斗早期的小差异可能像滚雪球一样变成后期的巨大优势，这使得侦察在游戏早期难以推销。在 Zero-K 中，这反映在缺乏专门的轻型侦察单位。早期侦察由突袭者完成，它们既能造成伤害也能侦察敌人。即使是名义上的"轻型侦察兵"也是武装的，能够骚扰没有防御的建筑工和金属提取器。

要在早期游戏中看到纯粹的专门侦察兵，需要一定程度的重新设计。这样的侦察兵要么非常便宜，要么非常强大以抵消其缺点。这种力量可以是相对的，即突袭者可以变得几乎失明，但失明单位的问题在于经常意外撞入敌人手中感觉很糟糕。最终我们避免将专门侦察兵强行加入早期游戏，因为管理起来太复杂了。记住，单兵种刷屏比混合单位容易使用得多，而早期已经有很多事情要做，不需要额外的复杂性。我们是在加入 Sparrow（麻雀）之后才相对近期发现这一点的。

Sparrow 是一种轻型侦察机，所有工厂都可以通过改造雷达塔获得。现在 Sparrow 在开局时有点贵，但最初它便宜得多，只需 120 金属。事实证明这太强大了，因为能看到一切使得惩罚偏离中庸开局的策略变得太容易。总的来说，当侦察消除了原本合理的战略选择时，它就太强大了。大多数在战略三角形边缘试探的策略都得益于敌人不能立即开始反制这一事实。

除了太强大之外，早期的 Sparrow 也太耗费注意力了。侦察兵需要关注以保持存活，因为它们想要靠近敌人，而使用它们揭示的信息则需要更多注意力。被侦察的建筑会留下一个持久的"幽灵"这一事实在一定程度上减轻了即时负担，但单位没有这样的功能。早期的 Sparrow 尤其成问题，因为它可以绕着炮塔飞行几分钟。这在游戏后期就不是大问题了，因为一旦专门的防空单位出来，侦察任务往往是快速且单向的。还有其他类型的侦察兵，比如隐形单位，它们可以潜伏，但管理它们需要的注意力比一架脆弱的飞机少得多。

不同类型的侦察兵擅长不同类型的侦察，玩家根据他们想发现什么来选择。一种侦察是"弱点侦察"，即寻找敌人防御中的漏洞。突袭者是特别好的弱点侦察兵，因为它们装备精良，能够跟进它们发现的弱点。另一种侦察是"威胁侦察"，玩家试图揭露并预先反制敌人的计划。事实上，Zero-K 中最便宜的专门侦察兵就是威胁侦察兵——雷达塔。把一个建筑称为侦察兵可能有点奇怪，但我会把 Starcraft 的 Comsat Station（通讯卫星站）也称为侦察兵。

雷达塔提供威胁侦察的意义在于它给你额外几秒时间来应对即将到来的攻击。这额外的时间让玩家可以重新部署力量来击退敌人，这使得炮塔可以比原本需要的弱一些。如果雷达不存在，玩家可能还需要在领地边缘部署几个突袭者来探测威胁，这和 Sparrow 一样会太耗费注意力。玩家目前在雷达范围之外使用突袭者进行威胁侦察，但他们在地图的重要区域会谨慎使用。适度使用是好的，但如果将此作为唯一的威胁探测方式，则会涉及太多微操。

威胁侦察的性质在游戏过程中会发生变化，主要区别在于威胁被揭露的时间早晚。所有计划最终都会被揭露（比如当核弹发射时），但实施时间较长的计划可以比其他计划更早被发现。对长期计划的威胁侦察是围绕战略三角形博弈的重要组成部分，因为不了解对手的动向就很难反制他们。

长期威胁侦察在中后期变得突出。到这时，战线已经接触，大多数短期威胁已被雷达充分覆盖。在这种情况下，揭露的威胁类型取决于你在侦察上的投入以及敌人试图阻止你的程度。浅层侦察可能发现中期威胁，比如 Cerberus（刻耳柏洛斯）或 Missile Silo（导弹发射井），而深层侦察可以发现长期威胁，如核弹、Striders（步行机甲）和超级武器。长期弱点侦察也是可能的，最常见的是检查反核覆盖。

Zero-K 的后期游戏侦察设计为几乎不可能被完全反制，但它也不便宜到可以随意进行。如果侦察容易被阻挡，人们就会这样做，但这会造成盲目战略 RPS 的局面，策略空间可能不得不收缩。最常见的深层侦察形式是将几架 Swift（雨燕，轻型战斗机）加速到敌方基地上空。加速使 Swift 在几秒钟内移动得更快，虽然它最初不是作为侦察工具设计的，但它以这种形式保留了下来，并为全面侦察设定了合理的成本上限。

发现了一个侦察成本下限，那就是飞行跳蚤侦察——使用廉价的 Newton 发射器将 Flea（跳蚤）发射到太空中。着陆不是问题，因为当时的视线是圆柱形的，所以跳蚤只需在地图上空飞过就能揭露一大片敌方基地。这种战术比其他选择便宜得多，所以我们通过降低飞行中地面单位的视野范围来削弱它。通过飞行跳蚤进行侦察仍然是可能的，但它们需要低空高速飞行才能看到东西，这需要更昂贵的发射器，并给敌人击落它们的机会。

跳蚤发射器有一个优点：敌人可以看到所有飞过他们基地的跳蚤。能够看到自己被侦察到通常是好事，因为它促进了更多的玩家互动和心理博弈。例如，我见过有人开始核弹rush，被侦察到，然后取消rush，指望敌人花费资源反制一个不存在的核弹。不过侦察兵的可被发现性有例外，主要是隐形侦察。这里的权衡是隐形侦察兵更慢、更难使用，而且经常被发现，这会提醒敌人他们被监视了。

侦察兵暴露自己在试图确定敌人选择了哪个工厂时会造成一种奇怪的情况。你可能发现了工厂或来自工厂的单位，但你派出的侦察兵肯定会暴露你自己的工厂。有几种方法可以绕过这一点，比如使用 Sparrow，但在早期不实用。在 1v1 中，被侦察是在不暴露自己工厂的情况下了解敌人工厂的唯一可靠方式。不过侦察和突袭在其他方面足够重要，这不成为问题，而且这种互动可能并不罕见。大多数多人游戏在大厅中显示阵营选择，但有些有隐藏的随机阵营选项，只有在侦察时才揭露。所以 Zero-K 中的情况更多是工厂选择（实际上就是你的阵营）默认被隐藏的结果。

尽管有一些持续存在的怪异之处，我对 Zero-K 中侦察的现状相当满意。关于扩张和突袭的信息在早期很有价值，随着游戏进行有许多重要的侦察目标可以发现。玩家有许多侦察方式，包括各种飞机、隐形单位，以及一些足够坚固能够突入敌方基地的单位。甚至 Sparrow 也找到了作为雷达干扰器和反隐形侦察器的位置，超越了它作为便捷深层侦察兵的用途。侦察感觉有回报且很少不可能完成，但足够昂贵以成为一个有意的决策点。侦察的决策是最重要的部分，因为如果它是例行公事且无需决策的，就可以被自动化，但有效地移除如此丰富的玩家互动来源将是一件憾事。

# Cold Take #38 - 避免状态切换
日期：2026-03-15

玩家在 RTS 中做的几乎所有事情都是通过发出直接命令或切换状态来完成的。直接命令给单位一个临时目标，包括经典的"移动到这个位置"或"射击这个敌人直到它死亡"。状态切换对单位行为做出更持久的更改，至少直到状态被切换回去。例子包括自动施法切换、可切换技能，以及 Age of Empires（帝国时代）系列中的姿态和阵型按钮。切换在谨慎使用时可以很好，但 Zero-K 做了相反的事，到了我们可能比其他任何 RTS 都多的地步。好处是我们学到了很多关于如何以及为何避免状态切换的知识。

状态切换的诱惑在于它们似乎体现了"对抗你的对手，而非对抗界面"的理念。玩家应该能够用最少的命令表达他们的意愿；在 Warcraft III（魔兽争霸III）中，你想让牧师用多余的法力进行治疗，然后能够告诉它自动治疗就用一键实现了这个愿望。在 Zero-K 中，玩家通常希望单位与敌人保持最大距离，这用状态切换比不断发出冗余移动命令容易得多。我们也可以从单位能力和行为之间的划分来思考状态切换。例如，射击感觉像是一种行为，所以拥有一个停止开火状态切换是有道理的。

状态切换大致分为两种类型：技能切换和行为切换。这是一个有用的区别，尽管从根本上说，单位能做的一切都是能力，它能选择做的每种不同方式都是行为。在实践中，控制特定能力或一般行为之间通常有明显的区别，尽管有些切换介于两者之间。

技能切换存在于许多游戏中，包括隐形、挖掘和部署。准确地说，我们可以将技能切换定义为使用某种能力所必需的状态切换。你不触碰攻城模式切换就无法让坦克进入攻城模式，但你可以在 Warcraft III 中不切换自动施法就施法，所以自动施法是行为切换。关于 Warcraft 和 Starcraft，它们的保持位置命令根本不是切换，因为每次想让单位静止不动时都必须重新发出。

Total Annihilation（横扫千军）是最早将保持位置等行为切换置于核心位置的游戏之一。所有能移动的单位都有一个包含三种状态的移动切换：保持位置、机动和漫游，后两种实际上分别是"稍微移动一下"和"到处移动"。所有有武器的单位也有一个包含三种状态的射击状态：停止开火、还击和自由开火，还击使单位在受到伤害后开火还击。Total Annihilation 还有技能切换，比如隐形和开关太阳能收集器的能力。

状态切换的问题在于它们很麻烦，而且在大多数情况下是控制单位最不令人满足的方式。发出命令——比如说布置侧翼或对一群敌人释放技能——比摆弄藏在界面角落里的按钮有趣得多。许多状态切换有微妙的效果，这使得玩家很难看到他们的行动是否产生了任何影响。查看单位状态本身就是一个问题，因为没有好办法完全显示更抽象的状态。Zero-K 无法显示混合选择中状态切换的所有不同方式，虽然有一个选项可以在按住 Shift 时在头顶显示状态，但非常混乱，不是我们想要启用的东西。

缺乏视觉反馈可以通过在游戏世界中反映单位状态来解决，而将自己限制在少数高影响力技能切换中的游戏往往做得很好。想想透明的隐形效果或单位部署动画。然而，这种方案不能扩展，因为它在单位整个身体被变形时最有效，所以多个状态最终会争夺空间。它也不能减轻切换状态的麻烦，或确保你的单位处于正确状态所需的心智负担。拥有高影响力技能切换的单位需要更多玩家注意力，这与界面战斗的配方相吻合。

加剧这些问题的是状态切换的表达能力不足。移动命令包含很多关于你希望单位去的确切位置的细微差别，而线性移动包含更多，而状态切换只能处于少数几种状态之一。状态切换是向单位传达你意图的极其低效的方式。诚然，在一支军队中设置状态有指数级多的方式，但切换它们所有会花费本可用于发出许多命令的时间。我们希望尽可能有效地将玩家意图转化为单位行动，为此，状态切换不是正确的方式。

Zero-K 使用相对较少的技能切换，并且正在恢复行为切换成瘾的状态。我们对技能切换的使用受到状态切换应尽可能可选的理念限制。这似乎排除了许多能力，比如部署开火，但严格来说它只排除了需要手动切换的版本。一个可切换的能力可以在切换永远不是坏主意的前提下拥有自动行为。例如，我们的可部署单位通过调整收起时间来避免状态切换，使得部署几乎永远不会是错的，所以它们在空闲时可以自动部署。这限制了我们的设计空间，但移除一个切换是值得的。

我们有一些真正的技能切换，但其中许多被设计为很少需要切换。隐形是一个例子，因为在早期游戏之后，关闭它很少是个好主意。隐形建筑工甚至有免费隐形以避免在开局时进行微操优化。在必要切换的情况下，我们试图将它们限制在昂贵的单位上。这些包括引力枪推/拉切换、特别重型火炮的高/低弹道射击，以及重型炮塔的固守技能。最常见的重要技能切换可能是两栖单位的浮沉状态，它类似于部署，但过渡时间有点太长，而且使单位容易受到水面火力的攻击。

有时可以通过搭便车来避免新的技能切换。例如，少数轻型单位和炮塔也可以固守以减少受到的伤害，但它们做得足够快以至于行为是自动的。大多数时候这让它们不需要触碰切换就能使用该技能，而在单位应该保持装甲的情况下，可以将它们设置为停止开火。浮沉切换的工作方式类似，因为默认情况下单位只在向敌人射击时浮出水面。在更极端的情况下，我们重新设计单位以移除状态切换，或者从一开始就避免添加可切换的能力。Raven（渡鸦）俯冲是前者的例子，而护盾链接是后者的例子。

重要的是状态切换只能由玩家切换。这听起来很显而易见，但默认引擎行为并不遵循这一点。隐形单位过去在能量耗尽时会自动切换到"解隐"状态，但如果能量变得可用就恢复到"隐形"状态。我猜意图是让切换状态告诉玩家技能的状态，但想想当你想在能量不足时解隐一些单位时会发生什么。看起来被命令解隐的单位实际上处于一种秘密的"有足够能量时隐形"状态，而这种状态无法被检测到，更不用说切换了。试图关闭一群波动的隐形单位是一场噩梦，所以面向玩家的状态隐形和开关状态现在由 Lua 处理，并始终反映玩家告诉单位做的事。

我们设法避免了太多技能切换，但在行为切换上有点过度了。在 Complete Annihilation（完整歼灭）早期，有许多开发者添加各种行为，所以作为一种安全措施，每种行为都有自己的切换。许多这些行为本应是严格改进，但我们足够谦虚，不期望每个人都想要新行为。结果是一些切换被设计为永远不会被触碰，因为新行为应该更好，更灵活，能涵盖玩家可能合理想让单位做的一切。

行为切换问题在 2020 年得到了处理。更冷门的切换默认隐藏，界面改为一次最多显示十二个，这个限制很少被达到。到这时，很少有玩家能说清引擎单位 AI 在哪里结束而 Lua 在哪里接管。战术 AI gadget 扩展了许多攻击移动和空闲行为，包括让远程单位保持在最大射程。它的切换可以被隐藏，因为它会被移动命令覆盖，并部分被保持位置覆盖。过杀预防也有一个隐藏切换，但会被强制攻击命令覆盖。在清理过程中，我们也隐藏了移动和射击状态中大多冗余的第三个选项，因为两态切换比三态切换容易使用得多。

一个切换的可接受性取决于它预期的使用方式。建造优先级是我们期望玩家最常触碰的切换之一，但它没问题，因为它很少在战斗激烈时被切换，而且它实现了一个简单的决策，只有少数选项：高、低或中优先级。建造是一个长期行动，手动控制其流程会涉及很多冗余命令。对于更快节奏的情况，比如战斗，切换被设计为被直接命令覆盖，让玩家发出命令而不是摆弄切换。

所有这些状态切换泛滥的亮点是无与伦比的单位行为定制。早期我们添加了菜单的一部分，让玩家为每种单位类型设置自己的默认状态。一些玩家，包括我自己，喜欢将一切设置为保持位置，而其他人设置预设建造优先级以确保金属提取器尽快建造完成。那些没人想在战斗激烈时调整的多余切换在这里找到了归宿。玩家可以按单位类型禁用单位 AI 的各个部分，适用于未来的所有战斗。从 widget（小部件）混乱开始——每个都有自己的切换——最终变成了一套庞大的选项来按你的喜好调整单位行为。

# Cold Take #39 - 特殊技能
日期：2026-04-04

一场冲突正在这个系列文章中酝酿。一半的 Cold Take——那些关于通用设计原则的——通常暗示 Zero-K 回避特殊技能，而另一半则通过讨论具体的特殊技能打破了这种印象。在这篇文章中，我们将试图通过探索特殊技能如何运作以及什么使一个技能适合 Zero-K 来解决这种表面上的分歧。

反对特殊技能的设计原则阵列确实令人生畏。首先，需要狂热或精确点击的技能被"玩家不应该与界面作战"的理念所排斥。许多其他潜在技能因为太数字化、使单位太笨拙，或施加任意的目标限制而被排除。原子化单位设计推动我们拥有少量技能，无论是特殊的还是普通的。但尽管有这些限制，Zero-K 比其他继承自 Total Annihilation 的游戏拥有更多的特殊技能。

一些设计原则鼓励特殊技能。Quant's Rule 反映了单位独特有趣的愿望，而特殊技能无疑是实现这一目标的一种方式。更重要的是，特殊技能很有趣，不仅仅是使用起来有趣。Zero-K 通过其开源放任自利的开发风格稳步积累特殊技能，许多贡献者致力于他们觉得最有趣的东西。这个过程早于 Complete Annihilation，因为早期的引擎开发者添加了泡泡护盾和使地形改造成为可能的可变形地形等功能。

随着 Complete Annihilation 和 Lua 脚本的出现，特殊技能开发的步伐加快了。我们的许多设计原则也是在这个时期制定的，但它们的目的更多是引导这股创造力的涌流，而不是试图阻止潮水。这个时期给了我们区域隐形、变形、跳跃引擎、地形改造和大量手动发射的武器类能力。新技能的摆弄有多令人满足，怎么强调都不为过，我甚至写过一些我的设计者一面知道不太可能进入游戏的技能。

事实上，那些似乎限制特殊技能设计的设计原则实际上让它得以繁荣。我们可以在指导方针内自由实验，而不太担心结果是否与游戏其余部分根本不相容。我们的实验仍然可能失败，但不会那么灾难性；游戏不会崩溃。没有任何指导，我们可能太害怕或太无组织而不敢尝试任何新东西。

所以 Zero-K 允许甚至鼓励特殊技能，只要它们符合设计。在实践中，这意味着技能必须通过两个测试：它们需要在游戏世界中有意义，而且不应该太难使用。一个技能是否有意义是两个测试中较容易判断的，因为我们有相当清晰的关于护甲类型等的规则，而《瞄准与射击》中的"理论理想技能"测试可以用于武器以外的东西。一个技能有多难使用——即它有多麻烦——可能更难判断，因为它涉及与该技能为游戏增加多少价值之间的权衡。关于麻烦程度的两个指导原则是玩家不应该与界面作战，以及技能应该避免使单位隐含地变笨拙。很多麻烦程度归结于使用它需要多少注意力。

最不麻烦的技能类型是被动技能。减速和缴械等伤害类型可以被视为被动技能，护盾等不可控能力也可以。这些技能几乎没有麻烦或变笨的风险，然而风险并非零。例如，Racketeer（破坏者）需要相当多的 AI 工作才能使其聪明到足以错开它的追踪缴械导弹以最大化持续时间。技术上 Racketeer 的问题归结于射击是一种主动技能，但所有单位都使用它们的主动技能来控制它们被动技能的表现方式，所以接受这个技术细节作为定义将使被动技能的概念变得无用。

仅次于被动技能的是被设计为很少需要关闭的切换技能。在实践中这很像被动技能，但由于 UI/游戏世界的分离（如下图所示），在理论上相当不同。单位可以自由选择何时以及如何使用主动技能，这为单位未能正确执行打开了可能性，无论是在空闲时还是响应玩家命令时。自动而非被动技能的优势在于它可以在边缘情况下被禁用，而且能够禁用能力通常在世界逻辑中更有意义。例如，太阳能收集器有一个状态切换来保持它们在炮火下展开，因为关闭感觉像是一个选择而不是被迫的。

第三类技能——在被动技能和自动主动技能之后——是手动技能，需要玩家直接干预才能使用。这类技能造成的麻烦最多，手动技能的麻烦程度主要取决于两件事：控制技能使用的命令类型，以及技能使用的频率。

大致有三种命令类型，按它们通常需要的目标类型分类：无目标、地面目标或单位目标。最简单的命令类型是无目标的。例子包括 Swift 加速、Krow 集束炸弹和爬行炸弹上的自毁按钮。只需点击按钮（或按热键）技能就会发动。这并不意味着知道何时使用技能很容易，只是发出命令不是困难的部分。Warlords Battlecry（领主之战）中的所有特殊技能都是无目标的，这可能影响了 Swift 加速。

状态切换可以被视为无目标命令的一个子类别，但跟踪状态的心智负担使它们使用起来有点困难。大多数切换技能，比如隐形和浮沉，可以保持在默认状态，但有几个需要切换才能使用其全部机制。引力枪上的推/拉切换就是一个好例子，重型炮塔的手动固守技能也是如此。

地面目标命令是次容易的命令类型，原因是地面大而静态，而单位相对小且经常移动。地面目标命令可以进一步细分为线性移动和区域命令，但拖动一条线或一个圆圈的额外复杂性与增加的表达能力之间存在权衡。Zero-K 中每个有目标的手动技能都可以向地面射击，大多数都是以此用途设计的。

单位目标命令是最难使用的类型。建筑比单位更容易点击，盟友比敌人更容易点击，但它们都比地面更难点击，所以我们的手动技能很少是打算直接瞄准单位射击的。只有少数最昂贵的单位（Reef（礁石）、Scorpion（蝎子）、Detriment（弊害）和一些高级指挥官武器）拥有这种命令。结果是玩家很少需要点击敌方单位，然而，"需要"这个词在这里承担了很多。

严格来说，玩家不必点击敌方单位就能让自己的单位使用技能，然而，这把目标选择权留给了单位 AI。在实践中，玩家经常通过右键点击来发出强制攻击命令，告诉单位挑出某个特定的敌方单位。目标优先级甚至会持续到被另一个攻击类命令覆盖，这很巧妙。但强制攻击还有另一个用途，它与停止开火配合，规避了前面提到的手动技能设计指导原则。这是因为在 Zero-K 中，如果一个单位只有一个武器，那么无论它多"特殊"，它都（几乎）总是被视为普通武器。

Zero-K 中相当多的普通武器如果出现在其他游戏中会被实现为特殊技能。一些火炮单位的装填时间在十几秒，Lance（长枪）甚至需要 23 秒装填。Firewalker（烈焰行者）发射在地面持续燃烧的火弹，而 Placeholder（占位者）发射微型黑洞将单位困在原地。玩家经常将这些单位设置为停止开火并手动使用它们，就像特殊技能一样。甚至我们的核弹和导弹发射井也使用普通武器，尽管这些建筑默认设置为停止开火。唯一没有武器但拥有有目标特殊技能的单位是 Lobster（龙虾），因为它在过去响应强制攻击时造成了太多事故。

使用停止开火将普通武器变成特殊技能可能看起来有点取巧，但这样做有很多好处。首先，通过右键点击使用技能比选择技能然后左键点击的两步过程更流畅。停止开火还允许玩家选择性地使用复杂性，而过杀预防和反诱饵等单位 AI 切换让玩家配置单位在自主决定使用这些技能时有多积极。通常的模式是玩家在早期微操少数单位，然后在后期将它们设置为自由开火。

回到特殊技能，那些经常与停止开火一起使用的武器工作方式与自动主动技能非常相似。它们预期在单位 AI 手中也能正常工作，这使得它们比手动技能少了很多麻烦。这些武器实际上会是糟糕的手动技能，因为它们经常想要尽可能频繁地射击，这是手动技能需要太多玩家注意力的最明显指标之一。

对于 Zero-K 来说，一个好的手动技能是玩家经常有但不使用的。考虑一下替代方案：如果一个技能需要 30 秒装填，而你的目标是尽可能多地使用它，那么你现在有一个需要你每 30 秒关注一次的杂务。在是否使用它上没有决策，即使在如何使用上可能有一些决策，它只是需要做的事情。这就是为什么没有地面冲刺技能：冲刺太容易被spam来更快穿越地图。Swift 只是因为它不需要加速就能非常快地穿越地图才逃脱了这一点。

这引出了一个手动技能的简单规则：它们必须有较长的装填时间，或者至少有某种机会成本，以防止它们变成杂务。跳跃引擎通过在平坦地图上几乎不比步行快来避免这个问题，因此获得的距离很少值得被抓住没有跳跃的风险。特别昂贵的单位可以打破这个规则，比如 Paladin（圣骑士）可以spam的EMP导弹，但像这样的规则可以被预期成为关注焦点的大型昂贵单位打破。

Zero-K 的设计原则显著限制了特殊技能的设计空间，但我们仍然在其中设法获得了乐趣。被动和自动技能相对安全，原子化单位设计是主要障碍，而太多手动技能有变成杂务的风险。我们目前的手动技能少到几乎可以将它们全部放在同一个热键上。添加更多的最佳方式是添加更多单位，甚至工厂，给玩家不同的技能来管理，而不是一次管理更多。

# Cold Take #40 - 撤退射程加成
日期：2026-04-25

Cold Take 通常讨论 Zero-K 设计中使用的概念，但这一篇是关于玩家发明的一个术语，用来描述游戏中的一个问题。这个问题涉及"撤退射程加成"，但这个术语最初令人困惑，因为它描述的既是涌现的又是普遍的。没人能告诉我其他游戏如何处理它，而且在不破坏 Zero-K 一些基本规则的情况下似乎很难移除它。

撤退射程加成是单位从战斗中撤退时获得的明显射程优势。考虑两个相同的单位——追击者和逃跑者——锁定在一场追击者追击逃跑者而逃跑者试图撤退的战斗中。撤退射程加成的观察是，在某个距离上，逃跑者能够开火而追击者不能。换句话说，看起来好像逃跑者仅仅通过撤退就获得了对追击者的射程加成。从表面看，这像是一个奇怪的任意加成，正是 Zero-K 绝对不想要的那种东西。

撤退射程加成的悖论在于它不是被添加的；它只是常见游戏物理的必然结果。如果一个游戏物理模拟其弹道，那么它可能就有某种形式的撤退射程加成。许多——可能所有——第一人称射击游戏都有这个，甚至可能在那个领域有个专门术语。Supreme Commander（最高指挥官）和 Planetary Annihilation（行星毁灭）之类的游戏也应该有，但到目前为止我还没有找到关于那些游戏中撤退射程的任何讨论。当我问人们这个问题时，即使在解释了该机制的普遍性之后，我也经常遇到困惑。

撤退射程加成是由单位提前量射击引起的。换句话说，要击中移动目标，你需要射击它将要去的位置，而不是它当前的位置。特别是，为了击中其射程边缘的逃跑目标，单位必须射击超出其射程的点。这样做会违反无虚空光束规则（No Void Ray rule），即武器的能力独立于其目标，因此单位不能射击这样的目标。在追击者和逃跑者的案例中，两个单位都刚好在彼此射程内，但追击者向逃跑者移动而逃跑者在远离，所以只有逃跑者能开火。这种效果被单位预测敌人何时进入射程并朝其射程边缘射击的预期所放大。

撤退射程加成根本上是由缺乏伽利略相对性引起的。伽利略相对性只真正出现在实际运动中，负责助跑投球等加速机制。模拟游戏倾向于避免它，这样做是正确的。人类在做人类事情时相当擅长考虑相对性，但在假装发射火箭发射器时就不擅长了。想象一下一边移动一边射击，结果因为自己的速度而射偏。我绝对会玩一个有伽利略相对性的 Zero-K mod，就为了体验那种荒谬，但我预计它会让单位相当笨拙，因为它们很多会在开火时突然向目标猛冲以获得额外射程。

到目前为止一切顺利。撤退射程加成被内置在游戏中，移除它比保留它更没意义。当我们考虑战术含义时，问题就出现了。追击者-逃跑者的战斗可以处于三种状态之一，取决于参与者之间的距离：

A) 两个单位都无法射击对方。
B) 只有逃跑者能射击。
C) 两个单位都能射击。

逃跑者非常想通过将战斗保持在 B 状态的最佳点来免费造成伤害。追击者绝对不想免费受到伤害，要么想杀死逃跑者，要么想把它赶走。双方都可以通过减速来控制战斗——对追击者来说，这扩大了从 A 到 B 到 C 的差距，直到逃跑者完全逃脱；对逃跑者来说，减速缩小了差距。这种对局面的共同控制创造了一个紧张的机动和假动作反馈循环，逃跑者有可能冒险尝试以零成本杀死追击者。这导致了激烈的点击大战，特别是在低延迟的硬核玩家手中。

撤退射程加成是作为关于早期游戏波动性的一系列更广泛反馈的一部分被提出的。简而言之，一些 1v1 玩家觉得太多东西取决于最初的几次突袭者交战，太少的游戏能走出这个阶段。这绝不是普遍的看法，但足够普遍值得一些实验，因为我们最终可能达到大多数人更喜欢的地方。追踪对某个问题特别敏感的玩家的反馈可能很有用，因为这可以揭示通常改善游戏但在大多数人抱怨阈值以下的东西。反馈和测试最终促成了 Superfluid Update（超流体更新）。

Superfluid Update 移除了突袭者的撤退射程加成。这些单位射程低、弹道快，导致最佳点很小，但它大到足以主导开局。一些玩家反对移除撤退射程加成，因为他们喜欢战术选择和它产生大幅摆动的潜力。其他人只是喜欢能够疯狂微操一个突袭者来杀死远超其成本的敌人，但这被减轻了处于劣势玩家的挫败感所抵消。

移除撤退射程加成的决定性因素是与界面的战斗——单位因为被诱入最佳点而死亡。另一个选择是改进我们的单位 AI 来考虑最佳点，但我觉得这样的 AI 会太不可靠且侵入性太强。这些战斗不仅仅是关于微操；还有决策需要做出。例如，不同速度的单位之间的战斗可能有一个试图冲过最佳点，代价是承受一些伤害，是否这样做取决于你的策略和整个游戏状态。玩家需要能够实施这些选择，但剩下的选项是状态切换和特殊命令，两者都比让玩家依赖简单移动命令差得多。本质上，移除撤退射程加成是调整机制以保留最流畅和最有趣界面部分的力量。控制感觉好比任何特定交互都重要得多。

在技术层面，我们通过强制单位不进行提前量射击，然后修改物理使它们的弹道自行提前来移除撤退射程加成。每次 Bandit（强盗）射击时，它的弹道会被轻微重新定向以指向目标的预测位置。这使得突袭者可以稍微超出射程射击，打破了无虚空光束规则，但它们的高弹道速度使效果几乎不可察觉。除非你暂停游戏、放大、截图然后画箭头。如果你无法忽视游戏中那个小偏差，我很抱歉，但这就是拉开帷幕时我们承担的风险。

至于 Superfluid Update 本身，其余部分涉及让突袭者和建筑工更难被杀死，以及通常让单位感觉更灵敏。这些变化随时间被回调到旧新状态的平均值，但这是有意为之。我倾向于过度调整改造以探测设计空间的合理区域的外缘，从而绘制更好的地图来导航到最终状态。但突袭者的撤退射程加成没有回归，关于突袭者博弈变得琐碎或无意义的预测没有实现。突袭者战斗仍然涉及大量的位置调整，玩家试图让所有突袭者射击一个目标，同时试图将敌方突袭者挡在彼此身后。不过，最好能听到六年前在那里的一些玩家的反馈，看看这些变化如何经受了时间的考验。

# Cold Take #41 - 死球与光环
日期：2026-05-17

"死球"（deathball）这个术语在玩家抱怨用单一军队在地图上横冲直撞太强大时就会出现。这种抱怨混淆了死球是使用起来不好玩，还是对抗起来不好玩，但记住这两种可能性是值得的。大多数 RTS 设计都在一定程度上处理了死球问题，但在减轻其问题或直接抑制的程度上各有不同。

死球是 RTS 的自然状态，就像没有平衡的战略三角形时，玩家最终会进行全力rush或贪婪经济扩张一样。另一种思考方式是，如果你随机组合一个通用 RTS，它将是一个死球游戏。事实上，死球可能只包含一种单位类型。这是兰彻斯特平方定律（Lanchester's Square Law）的结果，该定律指出一支十单位的军队比五单位军队强大不止两倍。没有任何理由放弃这种力量，玩家就会建造死球。

有许多方法可以避免或减少死球。在战略层面，可以通过在地图上分散多个目标来抑制它们。这些可以是 Zero-K 中的金属点、Company of Heroes（英雄连）中的控制点，甚至是 Warcraft III 中的野怪营地。这些目标为分散军队同时控制多个目标的玩家提供更大的回报。这种方法往往在游戏早期最有效，因为地图是空的，维持控制通常比夺取控制容易。地图中央的一个死球可以通过威胁反制任何侧翼尝试来控制大片领土。

死球也可以在战术层面通过几种方式被抑制，尽管它们都归结为从兰彻斯特平方定律中移除"平方"。近战战斗遵循兰彻斯特线性定律，其中军队力量与军队规模成正比，只要较大军队的额外单位找不到空间参战。这种效果使得拥有大单位（相对于其射程）的游戏介于平方和线性之间。Zero-K 单位如果需要的话可以紧密挤在一起，而且很少是近战的，但友军阻挡我们模拟弹道的方式有类似的效果。限制死球或至少提供反制手段的另一种方式是使用范围伤害（AoE），它使你的伤害输出同时随敌人军队规模和你自己军队规模缩放。

范围伤害可能是 RTS 中最流行的反死球措施，但它的力量往往取决于与界面的作战，所以在 Zero-K 中较弱。我们有大量的范围武器，它能遏制大量小型单位，但中型单位通常能够分散开并仍然形成有效军队。Zero-K 应该涉及侧翼和机动，而不仅仅是将军队撞在一起，我们需要支持大型团队游戏中的持久多线战斗。这需要相对较弱的死球，所以我们最终得到一个反对缩放增益光环（scaling buff aura）的宽松规则。

光环（aura）是在单位周围区域施加恒定效果的技能。缩放增益光环是以某种方式增益友军单位，而且附近友军越多效果越强的光环。例如，有人建议 Funnelweb（漏斗网蛛）可以被动治疗附近的盟友，这将是一个缩放光环，因为总治疗速率会随单位数量增加。任何类型的增益都有光环，大多数在 Warcraft III 中。其他游戏中的许多光环对 Zero-K 来说也太数字化了，但治疗或加速等效果本身是可以的。减速伤害是可能的。

Zero-K 有一些表现得像光环但不缩放的机制。隐形场属于这一类，因为隐形使用的特殊性。回忆一下有两种隐形：战略隐形和战术隐形。战略隐形不随军队规模缩放，因为发现一个可疑单位就几乎能揭露整个隐形军队。大型隐形军队比小型的更可怕，但军队不会因为规模而获得额外的隐形力量，实际上更大的军队更容易被发现。战术隐形类似：虽然它使敌人更难选择目标，但只要有目标可以射击，军队整体就会受到伤害。

泡泡护盾同样不缩放，因为无论覆盖多少单位，它们都阻挡相同数量的伤害。这里有一个例外，即阻挡范围伤害的护盾可以防止该伤害被许多单位承受。这种保护随单位数量缩放，这就是为什么护盾特别擅长对抗高范围伤害的轰炸机和火炮。护盾还间接增益了 Felons（罪犯）的伤害输出，这就是为什么护盾球（shieldball）作为最死球的策略之一而存活下来。

Zero-K 中最像光环的机制——在给区域内友军提供缩放增益方面——很可能是 Lobster（龙虾）的投掷。这是一个主动技能，但投掷的威胁使附近单位安全得多。Lobster 随单位数量缩放，因为范围内的每个单位都感受到投掷的好处，这就是为什么 Lobster 经常被纳入护盾球和其他紧密集群的军队中。所以事实证明，即使有反光环规则，类似光环的效果还是设法溜了进来。至少 Lobster 的范围较小，而且作为主动技能，有冷却时间。

如果你眯起眼看，许多东西都像光环。防空单位有驱散飞机的光环，暴兵单位对突袭者有类似效果，而容易被飞机或突袭者攻击的单位喜欢这些光环在周围。避免光环的目的不是阻止好的军队组合，而是通过避免简单的乘法加成让我们丰富的微妙交互闪耀。设计策略游戏通常涉及剔除最简单和最有效的策略以促进战略多样性。

# Cold Take #42 - 可回收残骸
日期：2026-06-07

基本回收机制如下：单位死亡时留下残骸，这些建筑工可以收割以获取额外金属。这听起来是一个足够简单的机制，但我想得越多，就越觉得没有它 Zero-K 就无法运作。回收是大多数受 Total Annihilation 启发的游戏的共同特征，尽管其重要性取决于它有多赚钱。一如既往，我只对其中一款游戏的设计了如指掌，所以虽然以下想法可能广泛适用，但我专注于回收在 Zero-K 中如何以及为何以这种方式运作。

回收的主要作用是在地图上动态创造利益区域。一些单位在冲突中死亡，留下几百金属在地面上，这使得守住该区域稍微更有吸引力。这导致更多战斗并留下更多残骸，甚至可以滚雪球成价值数千金属的激励以继续在该区域战斗。最终，一方获得足够控制以耗尽整个区域时，反馈循环被打破。这种动态的节奏很好：赌注不断上升而结果不确定，然后在一方获胜后迅速消散，释放紧张感。

一个所有人都投入军队的单一残骸区域过一段时间会变得有点无聊。幸运的是，地图上通常有多个分散的区域。回收本质上生成了多个变化的类似抢旗式目标，但以自然的方式针对地图上最有争议的部分。玩家转移注意力以确保残骸，这鼓励使用能够进行此类战略移动的机动军队。在某个区域投入更多单位的一方有更大机会获胜，但也使自己暴露在其他地方的攻击之下，这可以触发一个新的残骸区域供争夺。有时一方同时在所有战线获胜，但当这种情况发生时，回收让他们能快速结束游戏，这比在游戏结束后很久还继续进行更可取。

可争夺的残骸区域只有在玩家有理由在地图上活动时才会形成。金属点普遍满足这个角色，但许多地图也通过以可回收的地图特征开始来启动这个过程。经典的是岩石和树木，但有时地图会包含旧机器或魔法紫色水晶。树木通常只提供能量，这可能导致建筑工在早期比太阳能收集器更高效，而岩石包含金属。少量回收可以做很多事情，因为早期每一点都很重要。

除了改变区域的价值，残骸还改变了地形本身。大多数残骸阻挡移动和弹道，以至于开阔平原可以变成通道和掩体的迷宫。大型单位可以碾压较小的残骸，代价是减少可回收的金属量，而且并非所有残骸都阻挡。气垫船和船只残骸不阻挡移动，因为海底残骸过去会阻挡水面移动。一些轻型单位也留下不阻挡移动的残骸，大概是因为残骸太小了，不能从上面走过去会看起来很傻。

很容易将回收视为一种翻盘机制，在某些情况下它确实是，但更有用的是将其视为一种增加方差的机制。它当然可以提供在基地被突袭后重建所需的金属。然而，如果敌人知道你会占优势，他们一开始就不会浪费单位去冲你的基地。所以回收减轻了有效攻击的打击并遏制了边缘攻击，但通过增强防御可以达到类似效果。很难说这是一个特别强的翻盘机制，至少理论上是这样，因为玩家应该避免进行只是送金属的突袭。实践中可能会犯错，但优势方没有义务比劣势方更频繁地犯错。

当我们考虑有合理机会被任何一方获取的残骸时，事情变得更有趣了。一次攻击的长期成功往往取决于谁获取了产生的残骸，这通过扩大潜在结果的范围引入了波动性。所以即使一个玩家以积极结果的预期发动攻击，防御者经常有很好的机会集结并获取回收，或者至少获取相当一部分。这是回收提供的更强形式的翻盘机制，因为不确定性有利于劣势方。它必须如此，因为如果游戏只是可预测地推进，我们预期劣势方会输。基地突袭的这种方差比地图上的战斗少，因为防御者注定会获取残骸。

回收的重要性取决于残骸中能收集多少金属。太多金属会导致最早的回收滚雪球成胜利，太少则使系统变得无用。在 Zero-K 中，单位死亡时留下价值 40% 的残骸，除非它们死于大量爆发伤害，这种情况下留下 20% 的价值。这个价值在 Complete Annihilation 早期就被选定了，所以也许精确值并不重要，尽管我怀疑通过调整游戏其余部分的设计围绕一个简单的整数可以提供一些回旋余地。

回收的机制也很重要，尽管如果我们最初没有一个相当糟糕的实现，我可能不会这么想。引擎提供的默认系统让回收者在其目标残骸上填充一个进度条，填满时产出一块金属。这是一个丑陋的系统，原因有很多，比如一次性付款需要额外存储来处理，而被金属块击中使你的经济不必要地难以管理。当你考虑到多个玩家试图回收同一个大型残骸时引起的争吵和卡顿时，情况变得更糟，因为只有贡献最后一点进度的建筑工会获得奖励。

Zero-K 中的回收通过建筑工逐渐从残骸中吸取金属来工作。所有建筑工都可以回收，它们都以等于其建造能力的速率进行，因为任何其他选择都是任意的。这种拒绝应用任意乘数的做法使建筑工在回收方面相当糟糕，考虑到用 5 点建造能力回收数千金属需要多长时间。结果是回收可以在战斗中进行，但以任何显著速度进行需要大量投入，这增加了你输掉战斗的风险。一种流行的技术是埋藏建造塔，或者将它们放在墙后面，这作为一种鼓励玩家通过地形改造在地图上留下印记的方式效果很好。

回收还有另一套回收存活单位的机制，在这种情况下引擎提供的默认更糟糕。旧的存活回收系统本质上给每个建筑工装备了一把反向修理武器，其中"杀死"单位会导致一次性付款。一次性付款就够糟糕了，但更糟糕的是它可以瞄准敌人单位。单位不会自动回收敌人，所以这种能力被隐藏在与界面的战斗后面。我们可以自动化攻击性回收，但决定给每个建筑工装备武器对突袭者来说是太大的削弱，所以让（完全建造的）敌人不可回收。

旧的存活回收系统还有另一个问题：它可以用来避免爆炸。被回收杀死的单位跳过死亡爆炸，这意味着一个 Singularity Reactor（奇点反应堆）可以安置在你的基地中，只要有足够的建筑工在它受到攻击时安全拆除它。这是一个巧妙的技巧，但如果自动化就会消除爆炸的意义，所以 CA 开发者 det 添加了反向建造存活单位回收系统。这种方法将单位转回纳米框架，然后可以反向建造以获得 80% 的金属返还。最后一步是让纳米框架在完成度超过 80% 时爆炸，用一个 gadget 很容易做到。

我不应该过多抱怨引擎提供的默认值，因为它们大多继承自 Total Annihilation，而且引擎很容易扩展。一些改进甚至早于 Complete Annihilation，比如区域命令。没有区域命令，残骸很难回收，因为选项要么是发出巡逻命令，要么是逐个点击残骸。但巡逻有风险，建筑工可能决定修理而不是回收，或者游走到炮塔射程内。区域回收让你对建筑工的前进距离有更多控制，甚至有键盘修饰符让它们忽略树木和其他无金属的可回收物。

精确控制你的回收建筑工至关重要，因为玩家被鼓励将建筑工放在前线，在战斗边缘回收。这很好，因为它给敌人提供了可以狙击的机会目标，但如果建筑工不在你打算的位置，这样失去它们会很令人沮丧。建筑工还有其他前线任务，包括修理、建造，有些甚至可以从残骸中复活单位。游戏的很大一部分是判断建造多少建筑工以及在哪里部署它们，但这将留待另一篇文章。

# Cold Take #43 - 自定义层级
日期：2026-06-28

Zero-K 非常可定制，这很好但有使游戏令人生畏的风险。自定义的问题在于它要求玩家优化自己的体验，而不一定给他们做出明智选择所需的背景知识。一般来说，如果玩家足够理解一个选项以便将自己归入该选项所暗示的游戏的各种版本中，这个选项就是有用的。好的例子是色盲模式和语言设置，但 Zero-K 的选项远不止这些。另一个问题是即使它们都很好，选项太多也会压倒玩家。Zero-K 是在开发过程中设计的，UI 从各种实验的混合中凝聚出来，所以使我们的选项合理化一直是一个挑战。

大多数人坚持使用默认设置。在那些触碰设置的人中，许多会停留在浅水区——语言选择器和音量控制。但一些玩家想要深入挖掘，而另一些玩家想要保持他们十多年前开始玩时的旧默认值。为了平衡这些愿望，我们考虑了不同自定义层级的玩家，并在它们之间设置了门槛以防止新玩家被压倒。我们的目标是让游戏在所有层级都能良好运作，甚至允许光谱一端的玩家与另一端的玩家一起玩。

在第 0 层，玩家几乎不识字，很少触碰键盘。他们的大部分输入要么是上下文相关的右键命令，要么是通过建造菜单选择的建筑。小按钮对我们来说等于不可见，包括状态切换和 UI 左上角的一切。Zero-K 不是最简单的游戏，所以它并不完全设计为一直工作到第 0 层，但我们可能达到大约第 0.1 层，涉及偶尔阅读工具提示和学习一两个状态切换。

在第 1 层，玩家环顾 UI 并阅读工具提示。这解锁了撤退区域和反诱饵切换等功能，以及关于地形改造等功能的信息。人们有时问为什么像反诱饵切换这样的功能大多默认禁用，答案是它只在第 1 层才可见。每个人都从第 0 层开始，可能一直停留在第 1 层以下，所以游戏需要在没有状态切换的情况下也有意义。对于不知道反诱饵切换的玩家来说，单位未能向敌人射击看起来像是一个令人沮丧的 bug，而不是激励他们仔细查看游戏中所有默认设置的东西。第 1 层还包括色盲模式、语言设置和基本音量控制，因为它们直接在主 UI 上。

在第 2 层，玩家打开了游戏内菜单。菜单默认启用简单设置（Simple Settings），移除了绝大多数选项。在这个层级，玩家只看到最重要的选项，以新玩家认为要找的东西来衡量，比如锁定光标到窗口或选择小地图在 UI 的左侧还是右侧。第 2 层还包括完整的热键和默认单位行为菜单，所以如果我们将这些层级应用于其他游戏，很难找到允许超出第 2 层自定义的游戏。

在第 3 层，玩家取消勾选简单设置。这是第一个赋予玩家能力使其 UI 客观上变得糟糕的层级。这是通过启用调整模式（Tweak Mode）——通过菜单中的热键或按钮——然后移动和拉伸他们的 UI 来实现的。还有许多世界内 UI 的替代样式，比如单位选择，来自许多 widget 被编写和包含的时期。许多大多多余的花哨功能，比如损耗追踪器和风速显示，也可以在这里启用。

在第 4 层，玩家已经完全"作废了保修"，意味着他们现在可以彻底破坏东西，以至于在寻求帮助时被告知重置设置。这个层级通过打开 widget 列表或在菜单中切换显示高级设置来达到。大多数高级设置是无聊的技术细节或调试工具，但它也揭示了打开 widget 列表的热键。这个列表允许玩家开关 widget，其中一些对 UI 功能至关重要。还有一些 widget 存在于第 4 层，因为它们不够完善或解释太差以至于无法存在于更低层级。有重大 bug 的小众 widget 可能最终被降级到第 4 层，直到有人修复它们。

在第 5 层，玩家找到并安装了自己的 widget。没有第三方 widget 的中央仓库，所以很难知道这个层级有什么。许多 widget 在被包含到游戏中并转移到更低层级之前从第 5 层开始。widget 经常由于不为人知或缺乏足够的兴趣来完善以供包含而留在这个层级。目标通常是将 widget 降到第 3 层或更低，因为 widget 列表有点冷清。

在第 6 层，玩家正在创建自己的 widget。我们鼓励有特别新颖或有用的东西的人与社区分享，将它放在第 5 层供其他人使用，但我们从未需要强制执行这一点。即使在 widget 被分享时，使用自定义 widget 也与大多只是点击默认 UI 的第 0 层玩家相去甚远，所以我们的挑战是让游戏在所有这些层级都能运作。

最初有引擎行为，提供有限的自定义，其中大部分涉及热键或 UI 颜色文件。然后添加了 lua，引发了 widget 的爆炸。许多玩家会浏览 Spring 论坛并安装自己的 widget，将自己定位在第 5 层。一些人通过快速迭代彼此的 widget 获得了开源精神。Complete Annihilation 很快包含了最好的 widget，将它们塞进第 4 层的 widget 列表。第 2 层和第 3 层不存在，因为 CarRepairer 还没有创建我们花哨的游戏内菜单。结果是 UI 的混乱片段和 widget 菜单的广泛使用。

为了清理 UI，Licho 宣布核心开发者必须使用默认设置，迫使我们以更低层级运作。任何想偏离默认的开发者都必须实现一个易于访问的选项，即玩家可以在不作废保修的情况下做的事情。就我个人而言，我想将所有单位设置为保持位置，并有一个 widget 来实现这一点，但需要将它推入更低层级导致了完全通用的单位默认状态编辑器。游戏内菜单被创建，选项多年来被过滤为简单和高级设置，最后的推动涉及隐藏大多数命令和状态切换。

游戏本身之外的主菜单中的设置没有在自定义层级中占有一席之地，因为主菜单相对较新。我会将它们放在第 1 层或第 2 层，因为它们很容易被在主 UI 上点击的人找到。这些设置的存在造成了一些混淆，但能够在游戏内即时编辑热键和设置并获得即时反馈有很多好处。主菜单主要包含大厅和图形设置，在游戏中更改不切实际。主要缺点是游戏内图形菜单看起来有点空。

自定义层级也是思考功能目标受众的一种方式。一般来说，我们的目标是将尽可能多的 UI 能力推到尽可能低的层级，这使得所有层级的玩家能够在同一场战斗中共存。编写自己 widget 的能力绝对会打破大多数游戏，并迫使任何中等认真的玩家进入第 5 层或第 6 层。我们之所以能够逃脱这一点，是因为致力于让基础 UI 尽可能强大，这意味着要么整合足够强大的 UI，要么编写我们自己更用户友好的版本，要么修改游戏使 widget 变得不必要。有少数 widget 被直接阻止而非整合或变得无关紧要的情况，但这种情况值得单独写文章讨论。
